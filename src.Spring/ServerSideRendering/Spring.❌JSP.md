과거에는 JSP가 대세였지만, 최근 **Spring Boot 환경에서는 JSP 사용을 권장하지 않고** 타임리프(Thymeleaf) 같은 템플릿 엔진을 쓰는 추세예요. 그 이유는 기술적인 한계와 아키텍처의 변화 때문입니다.

주요 이유를 핵심만 짚어드릴게요.

---

### 1. Spring Boot의 지향점과 충돌 (내장 서버 문제)
Spring Boot의 가장 큰 장점은 내장 서버(Tomcat)를 사용하여 `.jar` 파일 하나로 어디서든 실행하는 것입니다.
* **JSP의 한계:** JSP는 기본적으로 **WAR(Web Application Archive)** 패키징을 강제합니다. JAR로 패키징하면 JSP 렌더링이 정상적으로 작동하지 않는 제약이 많습니다.
* **의존성:** JSP를 쓰려면 실행 시점에 전체 서블릿 컨테이너가 필요하지만, 요즘 트렌드인 '가벼운 실행 환경'과는 거리가 좀 있습니다.

### 2. 성능과 렌더링 방식 (비효율성)
JSP는 동작 방식이 다소 복잡합니다.
1.  사용자가 요청함.
2.  JSP 파일을 **Java 서블릿(.java) 파일로 변환**.
3.  그 자바 파일을 다시 **컴파일(.class)**.
4.  메모리에 적재 후 실행.

이 과정이 첫 실행 시 발생하여 느릴 뿐만 아니라, 뷰(View)가 수정될 때마다 이 복잡한 과정을 반복해야 합니다. 반면 타임리프 같은 최신 엔진은 훨씬 가볍게 동작합니다.

### 3. "관심사의 분리"가 어려움 (스파게티 코드)
JSP의 가장 큰 고질병은 HTML 안에 자바 코드(`<% ... %>`)를 직접 넣을 수 있다는 점입니다.
* 코드가 복잡해지면 HTML인지 자바인지 구분하기 힘든 **스파게티 코드**가 되기 쉽습니다.
* 이는 유지보수를 지옥으로 만드는 주범이죠. 반면 최신 엔진들은 HTML의 형태를 최대한 보존하면서 데이터만 바인딩하도록 설계되어 있습니다.



---

### 4. 에러 디버깅의 어려움
JSP는 런타임(실행 중)에 브라우저에서 에러가 터지는 경우가 많습니다. 자바 코드가 컴파일된 후에야 에러를 잡을 수 있기 때문에, 개발 단계에서 오타나 문법 오류를 미리 발견하기가 상대적으로 어렵습니다.

---

### 💡 그럼 뭘 써야 하나요?

사용자님이 지금 구축하신 프로젝트 구조(`controller`, `dto`, `service`)에는 아래 방식들이 훨씬 잘 어울립니다.

1.  **Thymeleaf (강력 추천):** Spring이 공식적으로 밀어주는 엔진입니다. HTML 형식을 깨지 않아서 디자이너와 협업하기 좋고 서버 없이도 화면 확인이 가능합니다.
2.  **Mustache:** 아주 단순하고 가벼운 엔진을 원할 때 씁니다.
3.  **Rest API + 프론트엔드 (React/Vue):** 요즘 가장 많이 쓰는 방식입니다. Spring은 JSON 데이터만 주고받고, 화면은 리액트나 뷰가 담당하는 거죠.

**결론적으로,** JSP는 이제 구시대의 유물이 되어가고 있어요. 팀 프로젝트를 현대적인 방식으로 진행하고 싶으시다면 **Thymeleaf**를 공부해서 적용해 보시는 걸 강력히 추천드립니다! 지금 패키지 구조에 `Thymeleaf` 설정만 한 줄 추가하면 바로 사용 가능해요.
