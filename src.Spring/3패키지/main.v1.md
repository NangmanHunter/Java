

### 2. 스프링 부트의 스캔(Scan) 범위 설정
이게 기술적으로 가장 중요한 이유입니다. 스프링 부트 애플리케이션의 시작점인 `@SpringBootApplication` 어노테이션이 붙은 클래스를 떠올려 보세요.

* **컴포넌트 스캔:** 스프링은 이 메인 클래스가 위치한 패키지를 **루트(Root)**로 인식합니다.
* **하위 패키지 탐색:** 메인 클래스가 있는 패키지와 **그 하위 패키지들**만 뒤져서 `@Service`, `@RestController` 같은 부품들을 찾아내 스프링 컨테이너에 등록합니다.
* **만약 패키지 구조가 없다면?** 메인 클래스가 패키지 없이 최상단에 있으면 프로젝트 전체를 다 뒤져야 하므로 성능이 떨어지고, 원치 않는 설정까지 불러오는 대참사가 날 수 있습니다.

### 3. 유지보수를 위한 "계층화" (Layered Architecture)
단순히 깊이만 깊은 게 아니라, 그 아래에 보통 `controller`, `service`, `repository` 같은 패키지를 또 나누죠? 이건 각 코드의 **역할 분담** 때문입니다.

| 패키지명 | 역할 |
| :--- | :--- |
| **Controller** | 사용자의 요청을 받고 응답을 돌려주는 "문지기" |
| **Service** | 핵심 비즈니스 로직(계산, 규칙)을 처리하는 "뇌" |
| **Repository** | DB에 데이터를 저장하고 가져오는 "창고지기" |
