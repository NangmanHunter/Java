##
- Spring Data JPA
- Querydsl
  - 조회 조건이 동적이고 복잡해지면 Querydsl을 도입한다.
  - 이때까지도 JPA의 영속성 컨텍스트를 사용함
- 통계성 쿼리나 Native SQL
  - Querydsl로도 해결이 안 되는 통계성 쿼리나 Native SQL이 필요하면 그때 **JdbcTemplate(JDBC)**이나 MyBatis를 해당 기능에만 부분적으로 적용한다.


##
JPA에서 JDBC(또는 SQL 중심의 도구)로 내려가야 하는 시점은 단순히 '조회'냐 아니냐를 넘어, **"JPA가 생성하는 SQL이 감당이 안 되거나, 객체 지향적인 접근이 오히려 손해일 때"**입니다.

실무에서 JDBC로 내려가는 구체적인 기준 3가지를 정리해 드릴게요.

---

### 1. "성능"이 서비스의 생사줄을 쥐고 있을 때
JPA는 기본적으로 **'객체 그래프 탐색'**을 합니다. 즉, 연관된 데이터를 가져올 때 객체로 변환하는 오버헤드가 발생하죠.
* **JPA 유지:** 한 번에 가져오는 데이터가 수십~수백 건 내외이고, 단순한 Join일 때.
* **JDBC 강등:** 한 번의 쿼리로 **수만 건 이상의 로우(Row)**를 조회해야 하거나, 1ms 단위의 응답 속도가 중요한 **초고속 트랜잭션** 환경일 때. 
    * *예: 대량의 실시간 로그 데이터 분석, 전사적 자원 관리(ERP)의 복잡한 결산 조회.*

### 2. "복잡한 SQL"이 도저히 JPA(JPQL)로 표현이 안 될 때
JPA의 JPQL은 표준 SQL의 기능을 대부분 지원하지만, 특정 DB에 특화된 기능(Dialect)은 약합니다.
* **JPA 유지:** 조건문(Where), 정렬(Order by), 일반적인 Join.
* **JDBC 강등:** * **Window Function:** `RANK()`, `LEAD()`, `LAG()` 등 순위나 앞뒤 행 비교가 필요할 때.
    * **Pivot/Unpivot:** 행과 열을 뒤바꿔야 하는 통계 보고서.
    * **Native Hint:** DB 성능 최적화를 위해 특정 인덱스를 강제로 타게 하는 SQL 힌트가 필수적일 때.

### 3. "대량의 쓰기 작업 (Bulk Insert/Update)"이 필요할 때
이게 의외로 가장 큰 이유 중 하나입니다.
* **JPA 유지:** 사용자가 게시글 하나 쓰고, 댓글 하나 다는 일반적인 웹 서비스 로직.
* **JDBC 강등:** 수만 건의 데이터를 엑셀로 업로드하여 한 번에 DB에 밀어 넣어야 할 때.
    * JPA는 `save()`를 호출할 때마다 영속성 컨텍스트가 관리하고 하나씩 쿼리를 날리는 속성이 있어 대량 작업에 매우 취약합니다. 이때 JDBC의 `addBatch()`를 쓰면 **수십 배 이상 빠릅니다.**

---

### 💡 실무자들의 판단 기준 (Thumb Rule)
보통은 다음과 같은 우선순위로 결정합니다.

1.  **Spring Data JPA**로 먼저 구현한다.
2.  조회 조건이 동적이고 복잡해지면 **Querydsl**을 도입한다. (이때까지도 JPA의 영속성 컨텍스트를 사용함)
3.  Querydsl로도 해결이 안 되는 **통계성 쿼리**나 **Native SQL**이 필요하면 그때 **JdbcTemplate(JDBC)**이나 **MyBatis**를 해당 기능에만 부분적으로 적용한다.

### 한 줄 요약
> **"단순한 데이터 관리(CRUD)는 JPA로 하되, DB의 성능을 극한으로 뽑아 써야 하는 '헤비한 쿼리'나 '대량 작업'이 나오면 주저 없이 JDBC로 내려간다"**고 생각하시면 됩니다.

현재 만들고 계신 통계 계산기 프로젝트의 경우, 복잡한 통계 수식을 DB SQL만으로 한 번에 처리해야 하는 상황(예: 수백만 건의 데이터 분산/표준편차 계산 등)이 온다면 그때가 바로 JDBC를 고려해볼 타이밍입니다. 가벼운 데이터라면 JPA만으로 충분하고요!
