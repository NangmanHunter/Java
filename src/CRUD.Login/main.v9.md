좋은 질문입니다! "접근이 안 된다"라고 생각하신 지점이 아주 날카롭네요. 아마 보안이나 서버 측 관점에서 생각하셔서 그렇게 느끼셨을 수 있습니다.

하지만 브라우저는 생각보다 많은 데이터를 **사용자의 컴퓨터(클라이언트)**에 저장할 수 있는 공간을 제공합니다. JWT가 저장되는 대표적인 두 곳인 **로컬 스토리지**와 **쿠키**가 실제로 어떻게 작동하는지 쉽게 풀어서 설명해 드릴게요.

---

## 1. 로컬 스토리지 (LocalStorage) - "내 브라우저 전용 창고"

로컬 스토리지는 브라우저가 각 사이트(도메인)별로 제공하는 일종의 **작은 DB**입니다.

* **어디에 있나?** 사용자 컴퓨터의 브라우저 폴더 내부에 파일 형태로 저장됩니다.
* **어떻게 저장하나?** 자바스크립트 코드 한 줄이면 됩니다.
    ```javascript
    // 서버에서 받은 JWT를 'my_token'이라는 이름으로 저장
    localStorage.setItem('my_token', 'eyJhbGciOiJIUzI1NiIsInR5cCI6...');
    ```
* **어떻게 다시 꺼내나?** 역시 코드로 꺼냅니다.
    ```javascript
    const token = localStorage.getItem('my_token');
    ```
* **특징:** 브라우저를 껐다 켜도 데이터가 사라지지 않고 그대로 남아있습니다.

## 2. 쿠키 (Cookie) - "서버와 브라우저 사이의 포스트잇"

쿠키는 서버가 "이거 나중에 다시 올 때 보여줘"라고 브라우저에게 건네주는 메모지입니다.

* **어떻게 저장하나?** 서버가 응답을 줄 때 헤더에 `Set-Cookie`를 실어서 보냅니다.
    ```http
    HTTP/1.1 200 OK
    Set-Cookie: Authorization=eyJhbGci...; Path=/; HttpOnly
    ```
* **특징:** 쿠키로 설정되면, 브라우저는 해당 서버에 요청을 보낼 때마다 **우리가 코드를 짜지 않아도 자동으로** 이 쿠키를 헤더에 붙여서 보냅니다.

---

## 🔒 "접근이 안 되지 않나?"에 대한 답변

질문하신 의도가 **보안** 때문이라면, 두 가지 측면에서 답이 될 것 같습니다.

### 1. 물리적 접근
사용자가 자기 컴퓨터의 브라우저 설정(개발자 도구 - Application 탭)에 들어가면 로컬 스토리지와 쿠키에 저장된 내용을 **직접 눈으로 보고 수정할 수 있습니다.** 즉, 클라이언트 입장에서는 완전히 열려 있는 공간입니다.



### 2. 프로그램적 접근 (보안 이슈)
* **로컬 스토리지:** 자바스크립트로 접근이 가능합니다. 그래서 내 사이트에 나쁜 스크립트(XSS)가 심어지면 해커가 `localStorage.getItem()`으로 내 토큰을 훔쳐갈 수 있습니다.
* **쿠키:** 서버가 `HttpOnly`라는 옵션을 걸어서 주면, 자바스크립트가 절대로 이 쿠키를 읽을 수 없습니다. 오직 브라우저와 서버 사이의 통신에만 쓰입니다. 이게 보안상 더 안전하다고 평가받는 이유입니다.

개발자가 코드를 짤 때:
1.  로그인 성공 후 서버가 JWT 문자열을 줌.
2.  프론트엔드 코드(JS)가 그걸 받아서 **`localStorage.setItem()`**으로 브라우저 창고에 넣음. (또는 서버가 **쿠키**로 꽂아줌)
3.  그다음부터는 창고에서 꺼내서 서버에 보여주며 "나 아까 그 사람이야!"라고 인증함.

"접근이 안 된다"기보다는, **"해당 도메인이 아니면(예: 구글이 네이버의 로컬 스토리지에) 접근할 수 없다"**는 제약이 브라우저에 걸려 있는 것이라고 이해하시면 정확합니다!
