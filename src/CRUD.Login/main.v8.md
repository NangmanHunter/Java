정확한 핵심을 짚으셨네요! **JWT를 클라이언트에 전달하고 저장하는 방식**은 크게 두 가지가 있는데, 그중 하나가 말씀하신 **쿠키(Cookie)**를 활용하는 방법입니다.

어떻게 '전달'되고 어디에 '저장'되는지, 그 메커니즘을 쉽게 설명해 드릴게요.

---

## 1. 서버가 JWT를 주는 방식 (전달)

서버는 로그인에 성공한 사용자에게 JWT라는 긴 문자열을 보냅니다. 이때 두 가지 통로를 주로 씁니다.

* **방법 A: HTTP 응답 본문(Body)에 담아주기**
    * JSON 형태로 `{ "accessToken": "eyJhbG..." }` 이렇게 보내줍니다.
    * 이 경우, 클라이언트(프론트엔드)가 이 값을 받아서 수동으로 저장해야 합니다.
* **방법 B: 쿠키(Set-Cookie)에 담아주기**
    * 서버가 응답 헤더에 `Set-Cookie: token=eyJhbG...`를 포함합니다.
    * 이게 바로 말씀하신 **"쿠키마냥 붙이는 방식"**입니다. 브라우저가 알아서 저장해주죠.

---

## 2. 클라이언트가 JWT를 보관하는 장소 (저장)

받은 토큰을 어디에 두느냐에 따라 보안 수준이 달라집니다.

| 저장소 | 특징 | 보안 이슈 |
| :--- | :--- | :--- |
| **Local Storage** | 구현이 매우 쉽고 편함 | **XSS 공격**에 취약 (자바스크립트로 탈취 가능) |
| **Cookie** | 브라우저가 자동으로 서버에 보내줌 | **CSRF 공격**에 취약 (대신 `httpOnly` 옵션으로 XSS 방어 가능) |

---

## 3. 요청할 때 "어떻게 다시 보내나?" (사용)

서버는 "이 요청이 로그인한 사람의 것인가?"를 확인해야 합니다. 이때 클라이언트는 토큰을 증거로 제출합니다.

### 케이스 1: 로컬 스토리지에 저장했을 때 (표준적인 방식)
매 요청마다 HTTP 헤더의 **`Authorization`**이라는 항목에 토큰을 담아서 보냅니다.
* **형태:** `Authorization: Bearer <JWT문자열>`
* 이 방식은 서버가 여러 대이거나 도메인이 달라도 사용하기 좋아 요즘 아주 많이 씁니다.

### 케이스 2: 쿠키에 저장했을 때 (전통적인 방식)
브라우저가 알아서 해당 도메인의 서버로 요청을 보낼 때마다 쿠키를 헤더에 붙여줍니다.
* **형태:** `Cookie: token=<JWT문자열>`
* 개발자가 일일이 코드를 짤 필요가 없어서 편합니다.

---

## 🛠️ 요즘은 어떻게 하나요? (추천 조합)

요즘은 보안을 위해 **'하이브리드'** 방식을 많이 씁니다.

1.  **Access Token (수명이 짧음):** 응답 Body로 받아서 메모리(변수)나 로컬 스토리지에 두고 헤더(`Authorization`)에 담아 보냅니다.
2.  **Refresh Token (수명이 김):** 이건 탈취되면 위험하므로, **`httpOnly` 옵션이 걸린 쿠키**에 담아서 보관합니다.



### 요약하자면:
"쿠키마냥 붙이는 것"도 맞고, "헤더에 직접 담아 보내는 것"도 맞습니다. 다만 **세션 방식**은 서버가 그 값을 기억하고 있어야 하지만, **JWT**는 서버가 기억할 필요 없이 **전달받은 토큰의 서명이 위조되지 않았는지만 검증**한다는 점이 결정적인 차이입니다.

수학 공식 증명할 때 '좌변과 우변이 같다'는 걸 확인하듯, 서버는 토큰 안에 적힌 값과 암호화 서명을 맞춰보고 "음, 맞네!" 하고 통과시켜 주는 거죠.
