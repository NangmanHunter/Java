두 개념이 처음 접하면 굉장히 헷갈리는데, 핵심만 짚어드리면 **JWT는 '데이터 형식(티켓)'**이고, **OAuth 2.0은 '데이터를 주고받는 절차(시스템)'**입니다. 

비유를 들자면 **OAuth는 '여권을 발급받고 심사하는 공항 시스템'** 전체를 말하고, **JWT는 그 과정에서 사용하는 '여권'이나 '비행기 티켓'** 그 자체라고 보시면 됩니다.

---

## 1. JWT (JSON Web Token)
JWT는 정보를 안전하게 전송하기 위한 **'토큰 규격'**입니다.

* **정체:** 점(`.`)으로 구분된 세 파트(`Header.Payload.Signature`)로 구성된 문자열입니다.
* **특징:** 토큰 안에 사용자 정보(이름, 권한 등)가 직접 들어있어서 서버가 DB를 안 뒤져봐도 "아, 얘 누구구나"라고 바로 알 수 있습니다 (**Stateless**).
* **용도:** 주로 서비스 내부에서 "로그인한 사용자임을 인증"하는 수단으로 씁니다.

## 2. OAuth 2.0 (Open Authorization)
OAuth는 제3의 서비스(구글, 카카오 등)로부터 권한을 위임받는 **'프레임워크(프로토콜)'**입니다.

* **정체:** "사용자가 구글 버튼을 누르면 -> 구글로 이동하고 -> 동의하면 -> 코드를 주고 -> 그 코드로 토큰을 바꾼다"는 식의 **약속된 흐름**입니다.
* **특징:** 우리 서비스가 사용자의 비밀번호를 몰라도 안전하게 로그인 처리를 할 수 있게 해줍니다.
* **용도:** 소셜 로그인, 외부 API 활용(내 앱에서 유저의 구글 캘린더에 일정 추가 등)에 씁니다.

---

## 3. JWT vs OAuth 2.0 핵심 비교



| 구분 | JWT | OAuth 2.0 |
| :--- | :--- | :--- |
| **본질** | **토큰의 형태 (Artifact)** | **인증의 흐름 (Protocol)** |
| **목적** | 정보를 안전하게 전달 및 인증 | 권한 부여 및 데이터 접근 대행 |
| **상태 저장** | 서버에 저장 안 함 (Stateless) | 흐름 관리를 위해 상태가 필요할 수 있음 |
| **비유** | 위조 방지 처리가 된 **티켓** | 티켓을 발급받기 위한 **발급 절차** |

---

## 4. 둘은 같이 쓰이는 경우가 많습니다

실제 서비스에서는 이 둘을 분리해서 생각하지 않고 **함께 사용**합니다. 보통 이런 흐름입니다.

1.  **OAuth 2.0 절차 시작:** 유저가 '구글 로그인'을 합니다.
2.  **인증 성공:** 구글이 우리 서버에 사용자가 누구인지 알려줍니다.
3.  **JWT 발행:** 우리 서버는 "오케이, 확인됐어" 하고, 우리 서비스 전용 **JWT**를 만들어서 유저에게 줍니다.
4.  **JWT 사용:** 유저는 이제 구글을 거치지 않고, 우리가 준 **JWT**만 들고 우리 서버의 CRUD(글쓰기, 읽기 등)를 이용합니다.

### 요약하자면
* **JWT**는 "이 종이(토큰) 안에 정보가 다 들어있고 조작이 불가능해!"라는 **기술적 도구**이고,
* **OAuth 2.0**은 "남의 집(구글) 계정으로 우리 집(내 앱)에 들어오게 하자!"라는 **비즈니스 로직**입니다.

수학이나 물리 공부하실 때 공식(JWT)을 배우고 그걸 활용한 증명 과정(OAuth)을 배우는 것과 비슷하다고 생각하시면 이해가 빠르실 거예요. 이 흐름 중에서 특별히 더 궁금한 부분이 있으신가요?
