로그인 인증 방식은 서비스의 규모와 목적에 따라 선택지가 갈리곤 합니다. 기존에 사용하신 **세션(Session)** 방식은 서버에서 상태를 관리하기 때문에 보안 면에서 유리하지만, 서버가 늘어나는 확장성 측면에서는 조금 까다로운 면이 있죠.

요즘 가장 많이 쓰이는 방식과 앞으로의 발전 방향을 정리해 드릴게요.

---

## 1. 토큰 기반 인증 (JWT: JSON Web Token)
현재 웹/앱 서비스에서 가장 대중적으로 사용하는 방식입니다.

* **원리:** 사용자가 로그인하면 서버는 사용자의 정보를 담은 암호화된 토큰(JWT)을 발행합니다. 클라이언트는 이 토큰을 저장해두었다가 API 요청 시 헤더에 담아 보냅니다.
* **장점:** 서버가 사용자 상태를 저장하지 않는 **Stateless** 구조입니다. 서버가 여러 대(Scale-out)여도 세션을 동기화할 필요가 없어 확장에 매우 유리합니다.
* **단점:** 토큰이 탈취되면 유효기간이 만료될 때까지 강제로 로그아웃시키기 어렵습니다. 이를 보완하기 위해 짧은 유효기간의 `Access Token`과 긴 유효기간의 `Refresh Token`을 함께 사용합니다.

## 2. OAuth 2.0 & OIDC (소셜 로그인)
직접 아이디/비밀번호를 관리하는 대신 카카오, 구글, 네이버 같은 외부 플랫폼의 인증을 빌려오는 방식입니다.

* **특징:** 보안 전문가들이 만든 인증 시스템을 그대로 사용하므로 보안 사고 리스크가 줄어듭니다.
* **발전 방향:** 최근에는 단순 인증을 넘어, 사용자의 권한을 위임받아 특정 리소스(이메일, 프로필 등)에 접근하는 표준으로 자리 잡았습니다.

---

## 3. 인증 기술의 발전 영역

최근에는 '비밀번호 없는 세상'과 '강력한 보안'이라는 두 마리 토끼를 잡기 위해 다음과 같은 기술들이 주목받고 있습니다.

### 🔐 Passkeys (WebAuthn)
* **설명:** 비밀번호 대신 지문, 페이스 ID, 혹은 기기 PIN 번호를 사용하는 방식입니다. 
* **이점:** 피싱 공격이 원천적으로 불가능하며, 사용자가 복잡한 비밀번호를 외울 필요가 없습니다. 애플, 구글, 마이크로소프트가 주도하고 있어 표준으로 빠르게 확산 중입니다.

### 🛡️ Zero Trust Architecture (제로 트러스트)
* **설명:** "아무도 믿지 마라"는 원칙입니다. 한 번 로그인했다고 끝나는 것이 아니라, IP가 바뀌거나 평소와 다른 패턴이 감지되면 지속적으로 인증을 요구하거나 권한을 세분화합니다.

### ⛓️ DID (Decentralized Identity)
* **설명:** 중앙 서버가 개인정보를 가지는 것이 아니라, 블록체인 등을 이용해 사용자가 자신의 신원 정보를 직접 소유하고 필요할 때만 증명하는 탈취 불가능한 신원 인증 방식입니다.

## 요약: 무엇을 선택할까?

| 방식 | 추천 상황 |
| :--- | :--- |
| **세션(Session)** | 소규모 서비스, 보안이 최우선이고 서버 확장이 크지 않을 때 |
| **JWT(Token)** | 모바일 앱과 연동하거나, 서버 확장이 빈번한 현대적인 웹 서비스 |
| **OAuth** | 사용자 유입을 쉽게 만들고 개인정보 관리 부담을 줄이고 싶을 때 |
| **Passkey** | 최신 보안 트렌드를 반영하고 사용자 경험을 극대화하고 싶을 때 |
