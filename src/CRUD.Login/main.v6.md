이게 아주 좋은 질문입니다! 비밀번호를 안 담는다고 하면 "그럼 회원 관리는 어떻게 하지?"라는 의문이 생기는 게 당연하거든요.

OAuth 2.0을 써도 우리 서비스의 **DB에는 여전히 사용자 정보가 저장되어야 합니다.** 다만 저장하는 **내용**이 세션 방식과는 완전히 달라집니다.

---

## 🗄️ DB에 담기는 핵심 정보

보통 `User` 테이블에 다음과 같은 컬럼들을 구성합니다.

| 컬럼명 (예시) | 설명 | 비유 |
| :--- | :--- | :--- |
| **Provider** | 어디를 통해 로그인했는가? (google, kakao, github 등) | 출입증 발급처 |
| **Provider ID** | 해당 서비스에서 부여한 **고유 식별 번호** (문자나 숫자) | 회원 일련번호 |
| **Email** | 사용자의 이메일 (연락처 및 중복 가입 방지용) | 사용자 이름표 |
| **Nickname** | 서비스에서 보여줄 이름 | 별명 |
| **Role** | 서비스 내 권한 (USER, ADMIN 등) | 계급 |
| **Password** | **(중요) 보통 비워두거나 무작위 값을 넣음** | **필요 없음** |

## 🛠️ 실제 동작 시나리오

개발자가 구현할 때의 흐름은 이렇습니다.

1.  **사용자 로그인:** 사용자가 '구글 로그인'을 성공합니다.
2.  **정보 전달:** 구글이 우리 서버로 `Provider ID: 123456`, `Email: test@gmail.com` 같은 정보를 보내줍니다.
3.  **DB 조회 및 저장:**
    * **처음 온 사람:** 우리 DB에 `123456`이라는 ID가 없네? -> 새로 **Insert** (회원가입 완료)
    * **기존 사용자:** 아, `123456`이라는 ID가 있구나! -> 해당 유저 정보를 가져와서 **로그인 처리** (Update/Select)

## 💡 왜 이렇게 할까요?

### 1. "비밀번호"가 없다
가장 큰 차이점입니다. 우리 DB에는 비밀번호가 아예 없거나, 의미 없는 난수만 들어있습니다. 해커가 우리 DB를 털어도 사용자의 구글 비밀번호는 절대 알 수 없고, 우리 서비스에 다시 로그인할 수도 없습니다. (구글을 직접 거쳐야 하니까요.)

### 2. "연결 고리"만 저장한다
우리는 사용자의 실제 신분증(비밀번호)을 가지는 게 아니라, 구글이 "이 사람 맞아요"라고 보증해 준 **일련번호(`Provider ID`)**만 기록해두는 겁니다. 나중에 그 사람이 다시 오면 그 번호가 맞는지만 확인하는 거죠.

## 🚀 발전된 형태: 서비스 전용 JWT 발행

실무에서는 여기서 한 단계 더 나아갑니다.

1.  구글로부터 정보를 받아 DB에 저장/확인한다.
2.  확인이 끝나면, 우리 서비스 전용의 **JWT(자체 토큰)**를 발행해서 사용자에게 준다.
3.  그다음부터 사용자는 구글을 거칠 필요 없이, 우리가 준 **JWT**만 들고 우리 서버의 CRUD 기능을 이용한다.

이렇게 하면 **"외부 인증(OAuth)"**과 **"우리 서비스만의 인증(JWT)"**이 깔끔하게 결합됩니다.

지금 만드시는 CRUD 프로젝트의 DB 구조를 설계 중이시라면, `password` 컬럼 대신 `provider`와 `providerId`라는 컬럼을 추가해 보시는 건 어떨까요? 수학이나 물리 문제 풀듯이 구조를 딱딱 맞춰가는 재미가 있을 거예요!
