언어 명세(Specification)의 관점에서 깊이 있게 궁금해하셨군요! 자바 언어의 규칙을 정의한 공식 문서인 **Java Language Specification (JLS)**에서는 말씀하신 그 "덩어리"들을 아주 명확한 용어로 정의하고 있습니다.

질문하신 클래스, 인터페이스, 레코드를 한데 묶어 부르는 가장 공식적인 명칭은 **참조 타입(Reference Types)**이며, 그 구조를 정의하는 행위는 **타입 선언(Type Declarations)**입니다.

---

## 1. JLS에서의 공식 분류 (Java SE 17+ 기준)

자바 명세서 제8장과 제9장을 보면 우리가 소스코드에 쓰는 그 "키워드 자리"를 다음과 같이 분류합니다.

### **Kind of Type Declarations (타입 선언의 종류)**

1.  **Class Declarations (클래스 선언):** `class` 키워드를 사용.
    * **Normal Class:** 일반적인 클래스.
    * **Enum Class:** `enum` 키워드를 사용하는 열거형.
    * **Record Class:** `record` 키워드를 사용하는 레코드. (중요: 명세상 레코드도 클래스의 한 종류입니다!)
2.  **Interface Declarations (인터페이스 선언):** `interface` 키워드를 사용.
    * **Normal Interface:** 일반적인 인터페이스.
    * **Annotation Interface:** `@interface`를 사용하는 어노테이션.

## 2. 왜 `int`와는 차원이 다른 '덩어리'인가?

말씀하신 대로 `int`, `boolean` 같은 **기본 타입(Primitive Type)**은 자바 가상 머신(JVM)이 이미 크기와 동작을 다 알고 있는 '빌트인' 자료형입니다.

반면, **참조 타입 선언**은 명세서에서 **"새로운 참조 타입을 도입하는 행위"**라고 정의합니다. 즉, 단순한 자료형의 사용을 넘어 **새로운 세계(타입 시스템)를 창조하는 설계도**를 그리는 단계인 것이죠.

| 구분 | 기본 타입 (Primitive) | 참조 타입 선언 (Reference Declaration) |
| :--- | :--- | :--- |
| **명세상의 위치** | 이미 정의됨 (Pre-defined) | **사용자가 새로 정의함 (User-defined)** |
| **존재 방식** | 메모리에 값 직접 기록 | **객체로 생성되어 메모리(Heap)에 거주** |
| **추상화** | 불가능 | **상속, 구현, 캡슐화 가능** |

---

## 3. "그 부분, 그 자리"를 부르는 전문 용어

개발자들끼리 대화하거나 더 깊은 문서를 보실 때 다음 용어들을 사용하시면 딱 들어맞을 거예요.

* **Top-level Type Declarations:** 파일의 가장 바깥쪽(패키지 선언 아래)에 위치하는 클래스나 인터페이스 선언부.
* **Reference Type Definitions:** 참조 타입의 구조를 정의하는 본체.
* **Reference Types:** (가장 흔히 쓰는 표현) `int` 계열과 대비되는, 객체 지향의 핵심 단위들.

### **요약하자면**
수학에서 이미 약속된 기호($+$, $-$, $\int$)가 **기본 타입**이라면, 새로운 정리(Theorem)를 증명하거나 새로운 집합(Set)을 정의하는 문장 자체가 **클래스/레코드/인터페이스 선언**입니다. 

그 자리는 단순히 변수 타입을 정하는 곳이 아니라, **"이 프로그램에서 다룰 데이터와 논리의 '규격'을 선포하는 자리"**라고 보시면 정확합니다. 이제 그 덩어리감이 좀 체감이 되시나요?
