'참조(Reference)'라는 이름이 붙은 이유는 변수가 데이터 **그 자체를 갖고 있는 게 아니라, 데이터가 있는 '메모리 주소'를 가리키고(Reference) 있기 때문**입니다.

자바의 데이터 타입은 크게 두 동네로 나뉩니다.

---

## 1. 기본 타입 vs 참조 타입

### 기본 타입 (Primitive Type)
`int`, `double`, `boolean` 같은 녀석들입니다. 얘들은 아주 작고 가벼워서 변수라는 상자 안에 **실제 값**이 직접 들어갑니다.
* 예: `int age = 20;` → `age`라는 상자를 열면 숫자 `20`이 바로 들어있음.

### 참조 타입 (Reference Type)
`class`, `interface`, `record`, `array` 등이 여기 해당합니다. 얘들은 덩치가 커질 수 있어서 변수 상자 안에 직접 넣기가 힘듭니다. 그래서 데이터는 메모리의 별도 공간(**Heap**)에 저장해두고, 변수 상자에는 그곳으로 찾아갈 수 있는 **약도(주소값)**만 넣어둡니다.
* 예: `User user = new User();` → `user` 상자를 열면 '메모리 100번지로 가세요'라는 **주소**가 들어있음.



---

## 2. 왜 굳이 '참조'를 할까? (비유)

우리가 친구에게 책을 선물한다고 가정해 볼게요.

* **기본 타입:** 책이 아주 작아서 그냥 손바닥에 올려서 직접 건네주는 것과 같습니다.
* **참조 타입:** 책이 너무 크고 무겁거나 도서관에 보관되어 있어서, 친구에게 **"그 책은 OO도서관 3층 5번 선반에 있어"**라고 **위치(참조)**만 적힌 메모지를 주는 것과 같습니다.

여기서 메모지를 받은 친구는 그 정보를 **참조**해서 실제 책을 찾아가게 됩니다.

---

## 3. 참조 타입이라서 생기는 특징

이 '참조'라는 개념 때문에 자바 개발을 할 때 아주 중요한 현상들이 발생합니다.

1.  **공유가 가능함:** 여러 변수가 똑같은 주소값을 가질 수 있습니다. A가 가진 메모지(주소)를 B에게 복사해주면, 둘 다 똑같은 위치의 데이터를 보게 됩니다. 한 명이라도 데이터를 수정하면 둘 다 바뀐 데이터를 보게 되죠.
2.  **`null`이 존재함:** "메모지에 아무 주소도 안 적혀 있음" 상태가 가능합니다. 이게 바로 그 유명한 `NullPointerException`의 원인입니다. 가리키는 곳이 없는데 찾아가려고 하니까 에러가 나는 거죠.
3.  **비교의 기준:** `==`으로 비교하면 '주소'가 같은지를 보고, `.equals()`로 비교하면 '내용물'이 같은지를 봅니다.

결국 **클래스, 인터페이스, 레코드** 등은 모두 덩치가 큰 객체를 다루기 위한 설계도들이고, 이것들을 변수에 담을 때는 항상 '주소'를 통해 다루기 때문에 묶어서 **참조 타입**이라고 부르는 것입니다.

레코드를 쓰실 때도 내부적으로는 객체이기 때문에, 변수에는 주소값이 담긴다는 이 원리는 똑같이 적용됩니다! 이해가 좀 되시나요?
