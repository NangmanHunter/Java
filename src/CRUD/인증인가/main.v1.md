CRUD(Create, Read, Update, Delete)를 구현할 때 가장 먼저 마주치는 벽이 바로 **인증(Authentication)**과 **인가(Authorization)**입니다. 두 개념은 비슷해 보이지만 역할이 완전히 다릅니다.

쉽게 비유하자면, **인증은 '출입증 확인'**이고, **인가는 '출입 가능한 구역 확인'**입니다.

---

## 1. 인증 (Authentication): "당신은 누구십니까?"
인증은 사용자가 **자기 자신임을 증명**하는 단계입니다.

* **동작:** 아이디/비밀번호 입력, 생체 인식, SNS 로그인 등.
* **CRUD와의 관계:** 모든 CRUD 작업을 하기 전, "로그인한 사용자인가?"를 확인하는 첫 번째 관문입니다.
* **결과:** 인증에 성공하면 시스템은 사용자에게 **토큰(JWT)**이나 **세션 ID**를 부여하여 "이 사람은 우리 회원이다"라는 증표를 줍니다.



## 2. 인가 (Authorization): "당신은 무엇을 할 수 있습니까?"
인가는 인증된 사용자가 **특정 리소스에 접근하거나 동작을 수행할 권한**이 있는지 확인하는 단계입니다.

* **동작:** 사용자의 역할(Role) 확인 (예: 일반 유저, 관리자).
* **CRUD와의 관계:** 여기서 권한에 따른 차이가 극명하게 갈립니다.
    * **Read:** "이 게시글은 전체 공개인가? 아니면 작성자만 볼 수 있는 비밀글인가?"
    * **Update/Delete:** "로그인한 사람(인증)은 맞는데, **이 글의 작성자**가 맞는가?" 혹은 "**관리자**인가?"
* **결과:** 권한이 없다면 `403 Forbidden` 에러를 반환하여 접근을 차단합니다.

---

## 3. CRUD 단계별 적용 예시
게시판 서비스를 예로 들어보겠습니다.

| 작업 (Action) | 인증 (AuthN) 필요 여부 | 인가 (AuthZ) 체크 내용 |
| :--- | :---: | :--- |
| **C** (게시글 작성) | **YES** | "로그인한 유저라면 누구나 작성 가능한가?" |
| **R** (게시글 읽기) | 선택적 | "비공개 글인가? (작성자/관리자만 허용)" |
| **U** (게시글 수정) | **YES** | "**본인이 쓴 글**인가? 혹은 수정 권한이 있는 관리자인가?" |
| **D** (게시글 삭제) | **YES** | "**본인**인가? 혹은 삭제 권한이 있는 관리자인가?" |

---

## 4. 실무에서의 흐름 (Spring Security 등)

1.  **Request:** 사용자가 "내 글 수정(Update)" 요청을 보냄.
2.  **인증 체크:** "토큰(JWT)이 유효한가?" → **실패 시 `401 Unauthorized`**
3.  **인가 체크:** "이 토큰의 주인공이 이 글의 주인(`owner_id`)과 일치하는가?" → **실패 시 `403 Forbidden`**
4.  **Business Logic:** DB에 접근하여 데이터를 수정함.

> **요약하자면:**
> * **인증**은 로그인 여부를 따지는 것 (**401 에러**와 관련)
> * **인가**는 "내 글인가? 관리자인가?" 처럼 권한을 따지는 것 (**403 에러**와 관련)

현재 개발 중인 서비스에서 Spring Security 같은 프레임워크를 도입하실 계획인가요, 아니면 직접 인터셉터 등을 활용해 구현하실 생각인가요?
