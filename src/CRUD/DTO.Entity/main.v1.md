스프링 부트 아키텍처에서 계층 간 데이터 전달 방식은 정답이 하나로 정해져 있지는 않지만, 유지보수와 안정성을 위해 업계에서 통용되는 **권장 패턴**이 있습니다. 질문하신 것처럼 엔티티를 어디까지 노출하느냐가 핵심입니다.

일반적인 흐름과 각 구간별 데이터 전송 객체는 다음과 같습니다.

---

### 1. Controller ↔ Service
이 구간에서는 보통 **DTO**를 사용합니다.

* **이유:** 서비스 계층은 비즈니스 로직을 담당합니다. 만약 엔티티를 그대로 컨트롤러에 넘기면, API 응답 스펙이 DB 구조(Entity)에 종속되어 버립니다. 예를 들어, DB 컬럼명이 바뀌면 API를 사용하는 프론트엔드 코드까지 다 깨지게 되죠.
* **방식:** * **요청(In):** 컨트롤러가 받은 `RequestDTO`를 서비스에 전달합니다.
    * **응답(Out):** 서비스가 엔티티를 비즈니스 로직에 맞게 가공한 후 `ResponseDTO`로 변환하여 컨트롤러에 돌려줍니다.

### 2. Service ↔ Repository
이 구간은 기본적으로 **Entity**를 사용합니다.

* **이유:** 레포지토리는 DB와의 대화를 담당하며, Spring Data JPA의 핵심은 엔티티를 통해 영속성 컨텍스트(Persistence Context)를 관리하는 것입니다. 따라서 데이터를 저장하거나 조회할 때는 엔티티가 가장 자연스러운 단위입니다.
* **예외 사항:** * 단순 통계 데이터나 여러 테이블을 Join해서 가져와야 할 때는 엔티티 대신 **별도의 인터페이스나 DTO(Projections)**를 통해 필요한 필드만 조회하기도 합니다.

---

### 계층별 데이터 흐름 요약



| 구간 | 전달 객체 | 설명 |
| :--- | :--- | :--- |
| **External ↔ Controller** | **Request/Response DTO** | 보안과 API 스펙 유지를 위해 엔티티 노출을 철저히 차단합니다. |
| **Controller ↔ Service** | **DTO** (또는 Command 객체) | 서비스는 외부 환경(웹)을 몰라도 되도록 설계하며, 가공된 데이터를 주고받습니다. |
| **Service ↔ Repository** | **Entity** | JPA의 기능을 온전히 활용하기 위해 엔티티를 주고받는 것이 기본입니다. |

---

### 💡 왜 엔티티를 서비스 밖으로 안 내보내려 할까?

엔티티에는 `@Id`, `@ManyToOne` 같은 JPA 설정뿐만 아니라 민감한 정보(비밀번호, 주민번호 등)가 포함될 수 있습니다. 

1.  **순환 참조 방지:** 엔티티 간에 양방향 연관관계가 있으면 JSON으로 변환할 때 무한 루프에 빠질 위험이 있습니다.
2.  **영속성 컨텍스트 관리:** 서비스 계층 밖(컨트롤러나 뷰)에서 엔티티를 수정했을 때, 의도치 않게 DB에 반영되는 사고를 막기 위함입니다.
3.  **유연성:** 서비스 내부 로직이 복잡해져서 여러 개의 엔티티를 조합해 하나의 데이터를 만들어야 할 때, DTO가 그 완충 역할을 해줍니다.

**결론적으로,** `Controller ↔ Service`는 **DTO**로 소통하여 계층 간 결합도를 낮추고, `Service ↔ Repository`는 **Entity**를 사용하여 JPA의 이점을 챙기는 방식이 가장 일반적입니다.
