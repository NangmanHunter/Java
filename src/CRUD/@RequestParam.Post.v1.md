##
맞습니다! 보안에 대해 아주 중요한 포인트를 짚어주셨어요. **"쿼리 스트링(URL 뒤에 붙는 정보)은 위험하다"**라는 말은 절반은 맞고 절반은 틀린데, 상황에 따라 다릅니다.

혼란을 싹 정리해 드릴게요.

---

### 1. GET 방식의 쿼리 스트링 (진짜 위험함)
우리가 흔히 보는 `http://.../login?id=test&pw=1234` 방식입니다.
* **왜 위험한가?**: URL 자체가 브라우저 방문 기록(History), 서버 로그, 캐시 등에 고스란히 남습니다. 옆 사람이 내 화면만 봐도 비밀번호가 노출되죠.
* **용도**: 검색어 입력(`?search=spring`)처럼 노출되어도 상관없는 데이터를 보낼 때만 씁니다.

### 2. POST 방식의 `@RequestParam` (의외로 안전함)
사용자님이 방금 테스트하신 방식입니다. **"POST인데 `@RequestParam`을 쓸 수 있다고?"** 하실 수 있는데, 네! 가능합니다.

* **동작 원리**: POST 메서드에서 `@RequestParam`을 쓰면, 데이터가 URL 뒤에 붙는 게 아니라 HTTP **Body(몸체)** 안에 `x-www-form-urlencoded`라는 형식으로 숨겨져서 전달됩니다.
* **보안성**: URL에 비밀번호가 찍히지 않으므로 GET보다는 훨씬 안전합니다. (HTTPS를 쓰면 Body 데이터도 암호화됩니다.)
* **한계**: 하지만 현대적인 API 설계(REST API)에서는 **JSON** 형태(`@RequestBody`)를 훨씬 더 선호합니다.



---

### 3. GET vs POST: 결정적인 차이점

| 구분 | GET | POST |
| :--- | :--- | :--- |
| **데이터 위치** | **URL 뒤** (쿼리 스트링) | **Body(몸체)** 안 |
| **데이터 용량** | 매우 작음 (URL 길이 제한) | 사실상 무제한 |
| **캐싱/기록** | 브라우저에 남음 (위험) | 남지 않음 (안전) |
| **주 목적** | 데이터 **조회** (Read) | 데이터 **생성/변경** (Create/Update) |

---

### 4. 실무에서는 어떻게 할까요? (Member 담당자로서의 선택)

사실 로그인 기능을 만들 때 가장 권장되는 방식은 유저님이 처음에 시도하셨던 **JSON Body** 방식입니다.

```java
// 가장 추천하는 방식 (보안 + 깔끔한 구조)
@PostMapping("/login")
public String login(@RequestBody LoginDto loginDto) { ... }
```

**왜 JSON 형식을 더 많이 쓸까요?**
1.  **확장성**: 나중에 로그인 시 '접속 기기 정보', '위치 정보' 등 보낼 데이터가 많아져도 구조화하기 쉽습니다.
2.  **표준**: 요즘 웹(React, Vue)이나 앱(Android, iOS) 통신은 거의 100% JSON을 씁니다.

---

### 💡 결론
* **POST + `@RequestParam`**: URL에 노출되지는 않지만, HTML 폼 전송 방식에 가깝습니다. (틀린 건 아닙니다!)
* **POST + `@RequestBody` (JSON)**: 요즘 가장 많이 쓰는 **정석**입니다.

지금 연습하시는 단계에서는 **두 방식의 차이를 이해하는 것**만으로도 엄청난 공부가 된 거예요. 혹시 아까 로그인 성공하셨나요? 성공하셨다면 이제 이 로그인 정보를 어떻게 유지할지(세션이나 토큰)로 넘어갈 차례입니다!
