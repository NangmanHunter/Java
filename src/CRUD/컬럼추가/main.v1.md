

##
- DB 반영: 앞서 말씀드린 것처럼, 이미 서비스 중인 DB라면 ALTER TABLE board ADD COLUMN created_at datetime; 등의 SQL을 통해 컬럼을 먼저 생성해 주어야 코드와 싱크가 맞습니다.


##
Spring에서 이미 서비스 중인 프로젝트의 Entity에 필드를 추가하는 상황은 실무에서도 매우 빈번합니다. 단순히 코드만 수정하면 되는 것이 아니라, **코드와 데이터베이스(DB) 간의 싱크**를 맞추는 것이 핵심입니다.

상황에 따라 권장되는 작업 방식은 다음과 같습니다.

---

### 1. 개발 환경 (dDL-auto 사용 시)
Spring Boot 설정 파일(`application.yml`)에 `hibernate.ddl-auto: update` 옵션이 켜져 있다면, Entity 클래스에 변수를 추가하고 서버를 재시작하는 것만으로 충분합니다. 하이버네이트가 자동으로 DB 테이블에 컬럼을 생성해 줍니다.

* **장점:** 매우 빠르고 간편함.
* **단점:** 운영 환경에서는 데이터 유실 위험이 있어 절대 사용하지 않음.

### 2. 운영 환경 (직접 서비스 중일 때)
서비스 중인 DB는 데이터가 담겨 있기 때문에 훨씬 신중해야 합니다. 코드 수정만으로는 DB 구조가 바뀌지 않으므로 아래 절차를 따르는 것이 정석입니다.

#### ① DB 스키마 변경 (SQL 실행)
먼저 DB에 접속하여 직접 컬럼을 추가하는 쿼리를 실행해야 합니다.
```sql
ALTER TABLE 테이블명 ADD COLUMN 새컬럼명 데이터타입;
```

#### ② Entity 클래스 수정
DB 구조를 변경했다면, 이제 자바 코드에서 해당 필드를 추가합니다.
```java
@Entity
public class Board {
    // ... 기존 필드
    
    private String newField; // 새로 추가된 필드
}
```

#### ③ DB 마이그레이션 도구 활용 (권장)
직접 쿼리를 날리는 것은 실수할 위험이 큽니다. 그래서 실무에서는 **Flyway**나 **Liquibase** 같은 마이그레이션 도구를 사용합니다.
* `V1__init.sql`, `V2__add_column.sql` 처럼 버전별로 SQL 파일을 관리합니다.
* 애플리케이션이 실행될 때 안 읽은 SQL 파일이 있다면 자동으로 DB에 반영해 줍니다.

---

### 💡 주의해야 할 점

* **기본값(Default) 설정:** 새로 추가된 컬럼에는 기존 데이터들이 `NULL`로 들어가게 됩니다. 만약 이 필드가 필수(`NOT NULL`)라면, 기본값을 어떻게 채울지(예: 0 또는 "Empty")를 쿼리 단계에서 결정해야 합니다.
* **성능 이슈:** 데이터가 수백만 건 이상 들어있는 테이블에 컬럼을 추가할 때는 DB가 잠시 멈추는(Lock) 현상이 발생할 수 있습니다. 사용자가 적은 새벽 시간에 작업하거나, 무중단 배포 전략을 고민해야 합니다.
* **DTO 수정:** Entity만 고친다고 끝이 아닙니다. 화면에 데이터를 뿌려주거나 API로 응답할 때 사용하는 **DTO**와 **화면(Front-end)** 코드에서도 해당 필드를 사용할 수 있도록 업데이트해야 합니다.

결론적으로, **"코드를 수정하되, 반드시 그에 맞는 DB 컬럼 추가 작업(SQL)이 병행되어야 한다"**고 이해하시면 됩니다! 현재 소규모 토이 프로젝트라면 `ddl-auto: update`로 편하게 진행하시고, 나중에 규모가 커지면 **Flyway** 도입을 고려해 보세요.
