## ❌Setter
- DTO.❌Setter



##
결론부터 말씀드리면, **DTO(Data Transfer Object)에서 Setter를 쓰지 않는 것이 현대 Java/Kotlin 개발의 권장 트렌드**입니다. 

단순히 "안 된다"는 금기라기보다는, **데이터의 일관성(Consistency)과 예측 가능성**을 높이기 위한 선택인데요. 왜 Setter를 지양하는지, 대신 어떤 방식을 사용하는지 깔끔하게 정리해 드릴게요.

---

## 1. Setter를 지양하는 이유

### ❌ 객체의 일관성을 해칩니다
Setter가 열려 있으면 객체가 생성된 이후에도 언제 어디서든 값을 바꿀 수 있습니다. 프로그램 규모가 커지면 "도대체 어디서 이 값이 바뀌었지?"를 추적하기 매우 힘들어집니다.

### ❌ 의도가 불분명합니다
`user.setName("John")`과 `user.updateProfile("John")`은 천지 차이입니다. Setter는 단순히 필드 값을 넣는 기능일 뿐, 그 행위가 어떤 비즈니스적 의미를 갖는지 설명하지 못합니다.

### ❌ 불변성(Immutability)의 장점을 포기하게 됩니다
객체가 한 번 생성되면 상태가 변하지 않는다고 보장되면, 멀티스레드 환경에서 안전하고 디버깅이 훨씬 쉬워집니다.

---

## 2. Setter 대신 사용하는 방법

그렇다면 Setter 없이 어떻게 데이터를 전달하고 다룰까요? 보통 다음 3가지 조합을 사용합니다.

### ① 생성자(Constructor) 주입
객체를 생성할 때 모든 필요한 데이터를 주입하고, 필드는 `private final`로 선언하여 변경을 막습니다.

### ② Builder 패턴
필드가 많아지면 생성자의 인자가 너무 많아져 헷갈릴 수 있습니다. 이때 **Lombok의 `@Builder`** 등을 사용하면 가독성 좋게 객체를 생성할 수 있습니다.

```java
// 빌더 패턴 예시
UserDto user = UserDto.builder()
    .name("홍길동")
    .email("hong@example.com")
    .build();
```

### ③ Java 14+ Record 사용
최신 자바 버전에서는 DTO 목적으로 만들어진 **Record** 타입을 사용합니다. Record는 모든 필드가 private final로 선언되며, Getter/Setter 대신 필드명과 동일한 접근자 메서드를 자동으로 생성해 줍니다.

---

## 3. 그럼 Setter는 아예 안 쓰나요?

세상에 절대적인 건 없죠! 다음과 같은 상황에서는 Setter를 쓰기도 합니다.

* **프레임워크의 제약:** 일부 오래된 라이브러리나 특정 직렬화(Serialization) 도구(Jackson 등)에서 기본 생성자와 Setter가 반드시 필요한 경우가 있습니다. (물론 최근엔 설정을 통해 Setter 없이도 대부분 가능합니다.)
* **단계별 데이터 수집:** 데이터를 한 번에 모으지 못하고 여러 곳을 거치며 하나씩 채워야 하는 특수한 경우엔 사용하기도 합니다.

---

### 💡 요약하자면
가장 좋은 설계는 **"생성 시점에 모든 데이터를 넣고, 이후엔 읽기만 가능하게(ReadOnly) 만드는 것"**입니다. 

Lombok을 쓰고 계신다면 클래스 상단에 `@Getter`와 `@NoArgsConstructor`, `@AllArgsConstructor` 혹은 `@Builder`를 조합해 보시고, **`@Setter`는 일단 빼놓고 코딩해보는 습관**을 들여보시는 걸 추천해요! 

DTO 설계를 고민 중이신 걸 보니 코드가 점점 깔끔해지고 계신 것 같네요. 혹시 특정 프레임워크(Spring 등) 환경에서 고민 중이신 건가요?
