##
**CRUD**는 기술적인 '기능의 범위'라기보다는, 데이터라는 **'자원(Resource)'을 다루는 가장 기본적인 4가지 방법**을 의미합니다. 따라서 게시판이냐 댓글이냐는 중요하지 않고, **"해당 데이터를 넣고, 읽고, 고치고, 지울 수 있는가"**가 핵심입니다.

---

## 1. CRUD의 본질: 자원 중심의 사고
질문하신 내용에 대해 명확한 기준을 잡아드릴게요.

* **게시판:** 게시글(Post)이라는 데이터를 **C**(작성), **R**(조회), **U**(수정), **D**(삭제) 하므로 CRUD입니다.
* **댓글:** 댓글(Comment)이라는 데이터를 **C**(작성), **R**(목록 조회), **U**(수정), **D**(삭제) 하므로 이 역시 완벽한 CRUD입니다.
* **회원관리:** 회원(User) 정보를 가입(C), 내 정보 보기(R), 정보 수정(U), 탈퇴(D) 하므로 이것도 CRUD입니다.

즉, **어떤 도메인이든 데이터를 관리하는 기능이 들어간다면 모두 CRUD의 영역**에 속합니다.



---

## 2. 실무에서 "CRUD 게시판"이라고 부르는 이유
보통 개발자들이 "CRUD 게시판 정도는 만들 줄 압니다"라고 말할 때, 이는 단순히 글쓰기만 한다는 뜻이 아닙니다. **웹 애플리케이션의 필수적인 데이터 흐름(Data Flow)**을 한 세트로 구현할 수 있다는 의미로 쓰입니다.

보통 'CRUD 프로젝트'라고 하면 아래 구성 요소들이 다 포함됩니다.

| 구분 | 게시판(Board) 예시 | 댓글(Comment) 예시 |
| :--- | :--- | :--- |
| **Create** | 글쓰기 (POST /posts) | 댓글 달기 (POST /comments) |
| **Read** | 목록 보기/상세 보기 (GET /posts) | 글 하단에 댓글 리스트 노출 (GET /comments) |
| **Update** | 글 수정 (PATCH/PUT /posts/1) | 내가 쓴 댓글 수정 (PATCH /comments/1) |
| **Delete** | 글 삭제 (DELETE /posts/1) | 댓글 삭제 (DELETE /comments/1) |

---

## 3. CRUD를 넘어서는 기능들
"어디까지가 CRUD인가?"라는 질문에 대한 답변으로, **CRUD가 아닌 것**을 보면 경계가 더 명확해집니다.

* **정렬 및 필터링:** 단순히 읽는 것(R)을 넘어, 특정 조건으로 검색하거나 정렬하는 로직.
* **권한 제어 (ACL):** "작성자 본인만 수정(U)/삭제(D)할 수 있게 한다"는 로직 (현재 구현 중이신 세션/JWT가 필요한 이유).
* **상태 변경 (Workflow):** 예를 들어 결제 시스템에서 '결제 대기 -> 결제 완료 -> 배송 시작'처럼 상태가 변하는 로직은 단순 CRUD 이상의 비즈니스 로직으로 봅니다.
* **좋아요/추천:** 이것도 넓게 보면 '좋아요 데이터'를 생성(C)하고 삭제(D)하는 것이지만, 보통은 '비즈니스 기능'으로 분류합니다.
