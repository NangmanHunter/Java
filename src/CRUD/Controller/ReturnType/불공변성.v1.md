##
- `List<Object>`
- `List<?>`.`List<Object>`


##
자바 제네릭의 불공변성(Invariance)은 처음 접하면 매우 직관적이지 않은 개념입니다. 상식적으로는 `String`이 `Object`의 자식(하위 타입)이니까, `List<String>`도 `List<Object>`의 자식이어야 할 것 같지만, 자바는 이를 엄격히 금지합니다.

왜 그런지 이해하기 쉽게 설명해 드릴게요.

---

### 1. 왜 안 될까? (안전성 문제)

만약 자바가 `List<String>`을 `List<Object>`에 대입하는 것을 허용한다고 가정해 봅시다. 그러면 다음과 같은 **대참사**가 발생할 수 있습니다.

```java
List<String> strings = new ArrayList<>();
strings.add("안녕");

// 만약 이게 허용된다면? (실제로는 컴파일 에러 발생)
List<Object> objects = strings; 

// objects는 List<Object>이므로 Integer를 넣는 게 문법적으로 가능해짐
objects.add(12345); 

// 다시 원래의 strings 리스트를 꺼내보면?
String str = strings.get(1); // 펑! (ClassCastException 발생)

```

`strings`는 분명 "문자열만 담는 리스트"라고 선언했는데, `List<Object>`로 취급하는 순간 **숫자가 들어오는 것을 막을 방법이 없어집니다.** 자바는 이런 런타임 오류를 방히하기 위해 아예 컴파일 단계에서 대입을 막아버리는 것이죠.

---

### 2. 공변 vs 불공변 용어 정리

자바 제네릭의 핵심 성질 3가지를 비교하면 더 명확해집니다.

| 용어 | 영어 | 의미 | 예시 |
| --- | --- | --- | --- |
| **공변성** | Covariance | 상속 관계가 그대로 유지됨 | **배열** (`String[]`은 `Object[]`의 하위 타입임) |
| **불공변성** | Invariance | 상속 관계가 유지되지 않음 | **제네릭** (`List<String>`은 `List<Object>`와 남남임) |
| **반공변성** | Contravariance | 상속 관계가 반대로 뒤집힘 | `<? super T>` 와일드카드 사용 시 |

---




### 3. 해결사: 와일드카드 `<?>`
- 질문하셨던 `<?>`가 여기서 등장합니다. `List<String>`을 `List<Object>`에는 못 넣지만, **`List<?>`에는 넣을 수 있습니다.**


* `List<?>`는 "무엇이든 올 수 있지만, **안에 뭐가 들었는지 모르니 쓰기(add)는 제한하겠다**"는 약속입니다.
* 데이터를 꺼낼 때는 최소한 `Object`인 것은 확실하니 안전하게 읽어올 수만 있게 합니다.



### 요약하자면

* **`List<Object>`**: "Object 타입만 다루겠다"는 고집 (다른 리스트 대입 불가).
* **불공변성**: "부모-자식 관계라고 해서 리스트까지 부모-자식은 아니다!"라는 자바 제네릭의 철칙.

배열은 `Object[] obj = new String[5];` 처럼 대입이 가능한데(공변성), 제네릭은 왜 안 되는지 그 차이점이 좀 이해가 가시나요?
