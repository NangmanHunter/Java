## @RequestParam
- [x] 주소창
  - 주소창의 `?id=...`
  - 쿼리 스트링(?id=...)
- [x] Body.form-data




##
궁금해하시는 지점이 아주 날카롭습니다! 결론부터 말씀드리면 **"원래 있습니다. 하지만 용도가 조금 다릅니다."**

우리가 흔히 **'쿼리 스트링'**이라고 부르는 그 주소창의 `?key=value`는 HTTP 메시지의 **시작줄(Start Line)**에 붙는 녀석이라, GET이든 POST든 상관없이 기술적으로는 붙일 수 있습니다.

상황별로 어떻게 다른지 깔끔하게 정리해 드릴게요.

---

### 1. POST 주소에 쿼리 스트링을 쓰는 경우
예를 들어 `POST /api/members/login?type=admin` 같은 주소로 요청을 보냈다고 가정해 봅시다.

* **동작 방식**: 서버는 주소창에 붙은 `type=admin`을 `@RequestParam`으로 읽을 수 있습니다. 동시에 HTTP Body(몸체)에 담긴 JSON 데이터는 `@RequestBody`로 읽습니다.
* **용도**: 보통 데이터의 **'본체'**는 Body에 담고, 그 데이터를 처리하는 **'옵션'**이나 **'카테고리'** 같은 메타 정보를 주소에 살짝 붙여 보낼 때 간혹 사용합니다.

### 2. 왜 GET에서만 쓰는 줄 알고 계셨을까요? (오해의 원인)
보통의 교육 과정이나 입문서에서는 이렇게 가르치기 때문입니다.
* **GET**: "데이터를 **주소**에 실어 보낸다." (쿼리 스트링 전용)
* **POST**: "데이터를 **Body**에 숨겨 보낸다." (Body 전용)

이게 '정석'이자 '약속'이기 때문에, 굳이 POST를 쓰면서 비밀번호 같은 중요한 정보를 주소(쿼리 스트링)에 노출할 이유가 전혀 없어서 잘 안 쓰는 것뿐입니다.

---

### 3. POST 방식에서 데이터를 보내는 2가지 길

POST 요청이 서버에 도착하면, 서버(스프링)는 데이터를 두 군데서 뒤질 수 있습니다.

| 위치 | 방식 | 어노테이션 | 특징 |
| :--- | :--- | :--- | :--- |
| **URL 주소창** | 쿼리 스트링 | `@RequestParam` | 주소에 다 보임. 기록에 남음. |
| **Message Body** | JSON 등 | `@RequestBody` | 주소에 안 보임. 구조적인 데이터에 적합. |
| **Message Body** | Form 데이터 | `@RequestParam` | **주소에는 안 보이지만** 형식이 단순함. |

> **여기서 반전!**
> 아까 유저님이 포스트맨에서 성공하신 방식은 사실 주소에 데이터가 붙은 게 아니라, **Body** 안에 `x-www-form-urlencoded`라는 형식으로 숨겨서 보낸 것일 확률이 높습니다. 스프링의 `@RequestParam`은 **주소창(쿼리 스트링)**에 있는 것도 읽지만, **Body에 담긴 폼 데이터**도 읽을 수 있거든요.



---

### 4. 핵심 요약
1.  **기술적으로**: POST도 쿼리 스트링(`?id=...`)을 가질 수 있습니다. (하지만 보안상 비밀번호는 절대 안 보냅니다.)
2.  **스프링의 `@RequestParam`**: 주소창의 `?id=...`와 Body의 `form-data`를 **둘 다** 읽을 수 있는 만능 도구입니다.
3.  **유저님의 케이스**: 포스트맨에서 Params에 적으셨다면 주소창에 붙은 것이고, Body의 form-data에 적으셨다면 Body에 숨겨서 가되 `@RequestParam`이 가로채서 읽은 것입니다.

**결론적으로**, 보안이 중요한 로그인/회원가입은 주소창에 노출되지 않도록 **Body**에 데이터를 실어 보내는 것이 원칙이며, 요즘은 그 데이터를 **JSON**으로 주고받는 것이 가장 세련된 방식입니다.

이제 쿼리 스트링과 POST의 관계가 좀 정리가 되시나요? 이 개념이 잡히면 나중에 API 설계할 때 "이 값은 주소에 넣을까, Body에 넣을까?"를 명확히 결정하실 수 있을 거예요!
