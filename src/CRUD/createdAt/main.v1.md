개발자들 사이에서 `createdAt`이나 `updatedAt`처럼 필드명 뒤에 **At**을 붙이는 건 거의 '국룰'에 가까운 관습이에요. 여기에는 꽤나 타당한 이유들이 숨어있습니다.

---

### 1. 문법적인 명확성 (Preposition 'At')
영어에서 시간을 나타낼 때 전치사 **at**은 특정 시점(Point in time)을 가리킵니다.
* **Created (동사):** "생성되었다"라는 동작에 집중합니다.
* **Created At (구문):** "~ 시점에 생성됨"이라는 뜻이 되어, 해당 필드에 담긴 값이 **'특정 시각'**임을 명확히 해줍니다.

### 2. 데이터 타입의 암시
프로그래밍을 하다 보면 필드명만 보고도 이게 어떤 종류의 데이터인지 추측할 수 있어야 효율이 올라갑니다.
* `is` / `has`: **Boolean** (예: `isAdmin`, `hasToken`)
* `count` / `amount`: **Number** (예: `userCount`)
* `At`: **Timestamp / DateTime** (예: `deletedAt`)

만약 그냥 `createDate`라고 하면 "날짜(YYYY-MM-DD)만 있는 건가?" 싶을 수 있지만, `createdAt`은 보통 **시, 분, 초까지 포함된 정밀한 시간**임을 시사하는 경우가 많습니다.

### 3. 일관성과 가독성
여러 필드가 섞여 있을 때 `At`으로 끝나면 시간 관련 정보라는 게 한눈에 들어옵니다.

| 필드명 | 의미 |
| :--- | :--- |
| **createdAt** | 생성된 시점 |
| **updatedAt** | 마지막으로 수정된 시점 |
| **deletedAt** | 삭제된 시점 (Soft Delete) |
| **lastLoginAt** | 마지막 로그인 시점 |

---

### 💡 여담: On과 At의 차이
가끔 `createdOn`을 쓰는 곳도 본 적 있으실 텐데, 뉘앙스가 살짝 다릅니다.
* **On:** 보통 날짜(Date) 위주일 때 사용합니다. (예: `createdOn: 2026-04-26`)
* **At:** 더 정밀한 시각(Time)까지 포함할 때 주로 사용합니다. (예: `createdAt: 2026-04-26 18:41:00`)

결국 **"이 데이터는 특정 동작이 일어난 정확한 '그때'를 기록한 것이다"**라는 것을 직관적으로 보여주기 위해 `At`을 붙이는 것이라고 보시면 됩니다!
