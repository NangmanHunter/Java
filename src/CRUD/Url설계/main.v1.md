##
- Url.자원중심>>행위중심
- Url.Resource>>Process


##
URL 설계 방식의 차이는 **"무엇(Resource)을"** 중심으로 보느냐, 아니면 **"누가/어떤 행위(Process)를"** 중심으로 보느냐의 차이입니다.

유저님이 말씀하신 방식과 제가 제안한 방식 모두 틀린 것은 아니지만, 보통 현대적인 API 설계(REST API)에서는 제가 제안한 방식을 조금 더 선호합니다. 그 이유를 명쾌하게 짚어드릴게요.

---

### 1. 자원(Resource) 중심의 계층 구조
REST API의 핵심은 **URL만 보고도 데이터의 구조를 파악**할 수 있어야 한다는 점입니다.

*   **`/api/admin/members`**: 관리자 권한으로 접근하는 "회원들"의 목록
*   **`/api/admin/members/7`**: "회원들" 중 "7번 회원"
*   **`/api/admin/members/7/approve`**: "7번 회원"의 "승인 상태"

이렇게 설계하면 URL이 **[대분류] -> [중분류] -> [고유 ID] -> [상태/행위]** 순서로 깔끔하게 좁혀집니다. 컴퓨터가 폴더를 찾아 들어가는 방식과 같아서 관리가 아주 직관적이죠.

---

### 2. 유저님이 말씀하신 방식 (`/admin/approve/members/7`)의 특징
이 방식은 **"행위(Approve)"**를 중간에 배치하는 방식입니다.

*   **의미:** "관리자 기능을 수행한다 -> 승인 작업을 한다 -> 대상은 7번 멤버다."
*   **단점:** 만약 나중에 "게시글 승인" 기능이 생기면 `/admin/approve/boards/10` 이런 식으로 짜게 될 텐데, 이렇게 되면 **'승인'**이라는 행위 아래에 온갖 종류의 데이터(멤버, 게시글, 댓글 등)가 다 섞여버립니다. 
*   나중에 "특정 멤버의 모든 관리 정보"를 보고 싶을 때 데이터를 추적하기가 조금 더 복잡해질 수 있습니다.

---

### 3. 왜 `/members/{id}/approve`를 추천할까요? (장점)

1.  **가독성:** `PATCH /api/admin/members/7/approve`라고 쓰면 "관리자가 7번 멤버를 승인(수정)한다"는 문장이 자연스럽게 완성됩니다.
2.  **확장성:** 나중에 멤버에 대해 다른 관리 기능(예: 강제 탈퇴, 포인트 지급)이 추가되어도 규칙이 일정합니다.
    *   `/api/admin/members/7/kick`
    *   `/api/admin/members/7/points`
3.  **Spring Security 적용:** `/api/admin/**`으로 시작하는 모든 요청을 한 번에 관리자 전용으로 묶어버리기 매우 편합니다.

---

### 💡 요약하자면
유저님이 말씀하신 방식은 **"동사(Approve)"** 중심이고, 제가 제안한 방식은 **"명사(Member)"** 중심입니다. 

현업에서는 **"명사(데이터)를 먼저 찾고, 그 뒤에 동사(행위)를 붙이는 방식"**이 더 표준으로 통합니다. 마치 파일 탐색기에서 파일을 먼저 선택하고 '우클릭'해서 메뉴를 고르는 것과 같은 이치죠!

어느 쪽을 선택하셔도 기능 구현은 가능하지만, 나중에 프로젝트가 커질 것을 대비한다면 **`/members/{id}/approve`** 방식이 조금 더 튼튼하고 낭만적인 설계가 될 것입니다. :)
