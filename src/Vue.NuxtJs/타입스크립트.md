##
역시 백엔드 개발자다운 예리한 감각이십니다! 자바(Java)로 치면 **인텔리제이(IntelliJ)가 패키지 명칭을 바꾸면 관련 소스들의 `import` 문을 싹 다 리팩토링해 주는 기능**과 정확히 같은 상황을 겪고 계신 겁니다. 🏹

이게 가능한 이유는 **VS Code**와 **TypeScript**가 뒤에서 몰래(?) 엄청난 일을 하고 있기 때문입니다.

---

### 🧐 1. 누가 이 일을 벌이는가? (TypeScript Server)
VS Code 안에는 **'TypeScript Language Service'**라는 일종의 데몬(Daemon)이 백그라운드에서 계속 돌아가고 있습니다.

* **감시 엔진:** 유저님이 폴더 이름을 바꾸거나 파일을 이동하면, VS Code가 이를 감지하고 TypeScript 서버에 "야, 주소가 바뀌었어!"라고 보고합니다.
* **자동 리팩토링:** TypeScript 서버는 프로젝트의 설계도인 `tsconfig.json`을 기준으로 연결된 모든 파일의 `import` 경로를 스캔합니다. 그리고 **"어? 아까 그 파일이 이사 갔네?"** 싶으면 자동으로 경로를 수정해 줄지 물어보거나(팝업), 설정에 따라 알아서 바꿔버립니다.

### ⚙️ 2. VS Code의 자동 설정 (치트키)
유저님이 따로 세팅하지 않아도 이게 되는 이유는 VS Code의 **기본값**이 아래와 같이 설정되어 있기 때문입니다.

* **`javascript.updateImportsOnFileMove.enabled`**: `always` 혹은 `prompt`
* **`typescript.updateImportsOnFileMove.enabled`**: `always` 혹은 `prompt`

> **"파일 이동 시 가져오기(Import) 자동 업데이트"** 기능입니다. 유저님이 폴더명을 바꾸는 순간, VS Code는 프로젝트 전체를 뒤져서 바뀐 경로를 `../../../` 식으로 계산해서 다 꽂아주는 거죠.

### 🛡️ 3. 백엔드 개발자가 주의할 점 (낭만 파괴 방지)
이 기능이 정말 편하지만, 때로는 위험할 때도 있습니다.

* **절대 경로 vs 상대 경로:** 만약 유저님이 `@/...` 같은 절대 경로(Alias)를 쓰고 있다면 에러가 덜 나지만, `../../` 같은 상대 경로를 쓰고 있을 때 폴더 깊이가 깊어지면 경로가 지저분해질 수 있습니다.
* **`node_modules`는 예외:** 폴더 이름은 바꿔주지만, 아까 겪으신 `package.json` 안의 실제 패키지 설치 경로(물리적 위치)까지 `npm`이 자동으로 맞춰주지는 않습니다. 그래서 아까 `ENOENT` 에러가 났던 겁니다.

지금 유저님이 보고 계신 마법은 **"VS Code(IDE) + TypeScript(언어 서버)의 합작 리팩토링"**입니다. 

자바에서 클래스 파일 드래그해서 다른 패키지로 옮길 때 인텔리제이가 **"Update imports?"**라고 묻는 것과 똑같은 기능이 프론트엔드에서도 작동하고 있는 거죠. 덕분에 폴더 구조를 바꿔도 코드가 안 깨지고 유지되는 낭만을 누리고 계신 겁니다!
