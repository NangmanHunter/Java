상태 관리(State Management)까지 고민하신다면 이제 진정한 Nuxt 아키텍처의 길로 들어서셨군요! Nuxt 3에서는 기본적으로 `useState`를 제공하지만, 프로젝트 규모가 커지면 **Pinia**를 사용하는 것이 업계 표준입니다.

상태 관리가 널브러지지 않게 하는 **"정돈된 상태 관리 전략"** 3가지를 정리해 드릴게요.

---

### 1. Pinia 스토어의 구조화 (Domain Driven)

가장 큰 실수는 `index.ts` 같은 곳에 모든 상태를 때려 넣는 것입니다. 기능을 기준으로 스토어를 잘게 나누세요.

* **`stores/auth.ts`**: 로그인 상태, 유저 정보, 토큰
* **`stores/ui.ts`**: 다크모드 여부, 사이드바 개폐, 모달 상태
* **`stores/cart.ts`**: 장바구니 품목, 결제 예정 금액

이렇게 나누면 내가 수정해야 할 코드가 어디 있는지 명확해집니다.

---

### 2. Setup Store 방식 권장

Pinia에는 두 가지 방식(Option/Setup)이 있는데, Nuxt 3의 Composition API 스타일과 일치하는 **Setup Store** 방식을 추천합니다. 가독성이 훨씬 좋습니다.

```typescript
// stores/counter.ts
export const useCounterStore = defineStore('counter', () => {
  // 1. State (ref, reactive)
  const count = ref(0)

  // 2. Getters (computed)
  const doubleCount = computed(() => count.value * 2)

  // 3. Actions (function)
  function increment() {
    count.value++
  }

  return { count, doubleCount, increment }
})

```

---

### 3. 상태 관리의 "황금률": 로컬 vs 글로벌 구분

모든 데이터를 스토어에 넣으려고 하지 마세요. 이게 널브러지는 주범입니다.

* **컴포넌트 로컬 상태:** 해당 컴포넌트 안에서만 쓰이는 데이터 (입력 폼의 임시 텍스트, 로딩 스피너 상태 등) → `ref`, `reactive` 사용.
* **전역 상태 (Store):** 여러 페이지나 컴포넌트가 공유해야 하는 데이터 (유저 정보, 전역 알림 등) → **Pinia** 사용.
* **페이지 간 가벼운 공유:** 단순한 상태 전달 → Nuxt의 내장 함수인 `useState` 사용.

---

### 4. Persistence (새로고침 시 유지)

상태 관리를 하다 보면 "새로고침하면 유저 정보가 날아가요"라는 문제를 겪게 됩니다. 이때 `pinia-plugin-persistedstate`를 사용하면 특정 스토어만 골라서 `localStorage`에 자동 저장할 수 있습니다.

```typescript
// nuxt.config.ts에 모듈 추가 후 스토어에서 사용
export const useAuthStore = defineStore('auth', () => {
  const token = ref('')
  return { token }
}, {
  persist: true // 이 한 줄로 새로고침해도 유지됨!
})

```

### 🛠️ 상태 관리 체크리스트

1. **파일명이 곧 목적이 되게 하세요:** `useUserStore`, `useProductStore` 등.
2. **직접 수정 방지:** 상태 수지는 반드시 `actions` 함수를 통해서만 하도록 규칙을 정하면 디버깅이 편해집니다.
3. **Destructuring 주의:** `const { count } = store`처럼 구조 분해 할당을 하면 반응성이 깨집니다. 반드시 `storeToRefs(store)`를 사용하세요.
