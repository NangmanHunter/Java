##
- `throw new RuntimeException("삭제 권한이 없습니다.");`


##
결론부터 말씀드리면, 별도의 처리를 하지 않았다면 **HTTP 상태 코드 500(Internal Server Error)**과 함께 **JSON 객체** 형태 혹은 **HTML 에러 페이지**로 나갑니다.

스프링 부트(Spring Boot) 컨트롤러에서 예외가 터져서 밖으로 던져졌을 때의 최종 모습은 다음과 같습니다.

---

### 1. REST API (JSON 응답)
가장 일반적인 상황입니다. 클라이언트(모바일 앱, 프론트엔드 React/Vue 등)가 JSON을 기대하고 있다면 스프링의 `BasicErrorController`가 예외를 낚아채서 다음과 같은 규격으로 응답을 만듭니다.

```json
{
  "timestamp": "2026-04-24T15:20:00.000+00:00",
  "status": 500,
  "error": "Internal Server Error",
  "message": "삭제 권한이 없습니다.",
  "path": "/api/post/1"
}
```
* **어떤 형태로?**: `message`라는 키(Key)의 값(Value)으로 문자열이 들어갑니다.
* **중요 포인트**: `RuntimeException`은 기본적으로 "서버 내부에서 알 수 없는 에러가 발생했다"고 판단하기 때문에 상태 코드가 `500`으로 고정됩니다.

---

### 2. 일반 웹 페이지 (HTML 응답)
브라우저에서 직접 URL을 입력해 접속 중이었다면, JSON 대신 우리가 흔히 보는 **"Whitelabel Error Page"**라는 이름의 기본 에러 화면이 뜹니다.

> **Whitelabel Error Page**
> This application has no explicit mapping for /error, so you are seeing this as a fallback.
> ...
> **Message: 삭제 권한이 없습니다.**

---

### 3. 실무에서 주로 하는 처리 (Customizing)
사실 클라이언트 입장에서는 500 에러를 받으면 "서버가 고장 났나?"라고 오해할 수 있습니다. 그래서 보통은 `@RestControllerAdvice`를 사용해서 **상태 코드와 형식을 예쁘게 다듬어서** 보냅니다.

예를 들어, "권한 없음"은 500이 아니라 **403 Forbidden**으로 나가는 게 더 정확하겠죠?

```java
// 이런 식으로 가공하면
@ExceptionHandler(RuntimeException.class)
public ResponseEntity<String> handleRuntime(RuntimeException e) {
    return ResponseEntity.status(HttpStatus.FORBIDDEN).body(e.getMessage());
}
```

이렇게 처리하면 클라이언트는 JSON 객체가 아닌 **단순 문자열("삭제 권한이 없습니다.")**만 받거나, 본인이 정한 에러 규격으로 받게 됩니다.

### 요약
1.  **아무 설정 안 함**: `500 에러` + `JSON 데이터(안에 메시지 포함)`
2.  **설정 함**: 내가 정한 `HTTP 상태 코드` + `내가 원하는 메시지 형식`

지금 프로젝트에서 별도의 `ExceptionResponse` 클래스나 `GlobalExceptionHandler`를 만들어 두셨나요? 그렇다면 그 클래스에서 정의한 변수명(예: `msg`, `error_details` 등)에 담겨서 나갈 겁니다.
