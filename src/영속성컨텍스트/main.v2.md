##
JPA(Java Persistence API)의 구현체인 Hibernate 등은 **영속성 컨텍스트(Persistence Context)**를 통해 일차 캐시, 변경 감지, 쓰기 지연과 같은 강력한 기능들을 모두 지원합니다.


### 1. 1차 캐시 (First-level Cache)
영속성 컨텍스트 내부에는 엔티티를 보관하는 메모리 공간이 있습니다.
* **작동 방식:** 데이터를 조회할 때 먼저 DB로 가지 않고 1차 캐시를 확인합니다. 만약 찾는 데이터가 캐시에 있다면 DB를 거치지 않고 바로 반환합니다.
* **장점:** DB 접근 횟수를 줄여 성능을 높이고, 동일한 트랜잭션 내에서 조회한 객체의 동일성(Identity)을 보장합니다.

### 2. 변경 감지 (Dirty Checking)
JPA는 엔티티를 수정할 때 별도의 `update()` 메서드를 호출할 필요가 없습니다.
* **작동 방식:** JPA는 엔티티를 영속성 컨텍스트에 보관할 때, 최초 상태의 복사본인 **'스냅샷'**을 찍어둡니다. 그리고 트랜잭션이 커밋되는 시점에 현재 엔티티와 스냅샷을 비교하여 변경된 부분이 있다면 자동으로 `UPDATE` SQL을 생성해서 실행합니다.
* **특징:** 마치 자바 컬렉션의 객체를 수정하는 것처럼 자연스럽게 데이터를 다룰 수 있게 해줍니다.

### 3. 쓰기 지연 (Transactional Write-behind)
엔티티를 저장하거나 수정할 때 즉시 DB에 쿼리를 날리지 않습니다.
* **작동 방식:** `em.persist(entity)`를 호출하면 SQL을 생성해서 내부의 **'쓰기 지연 SQL 저장소'**에 차곡차곡 쌓아둡니다. 그러다 트랜잭션을 커밋하는 순간, 쌓여있던 쿼리들을 한꺼번에 DB로 보냅니다(Flush).
* **장점:** 네트워크 통신 횟수를 줄일 수 있고, 여러 쿼리를 모아서 처리하기 때문에 최적화가 가능합니다.

---

### 왜 이런 기능들이 중요한가요?
- 보통 백엔드 개발을 할 때 **객체 지향적인 코드**와 **관계형 데이터베이스(RDB)** 사이에는 괴리가 생기기 마련입니다. 
- JPA는 영속성 컨텍스트라는 중간 관리자를 두어, 개발자가 SQL 중심이 아닌 **객체 중심**으로 비즈니스 로직에만 집중할 수 있게 도와주는 것입니다.
- 비즈니스로직에만 집중. Service에만 집중
- 객체 얘로만 건들면. 알아서들 싹다 반영되게 만든것.
- ←`객체`→
  - 모든주도권을 센터에서 진행하게 잡아놓은격.




현재 진행 중인 프로젝트에서 엔티티를 설계하거나 데이터를 다룰 때 이 원리를 이해하고 있으면, 의도치 않은 `UPDATE` 쿼리가 날아가거나 조회가 안 되는 문제를 훨씬 쉽게 해결하실 수 있을 거예요.
