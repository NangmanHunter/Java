##
비관적 락(Pessimistic Lock)이 **"절대 틀리면 안 되는 금전적 수치"**를 수호한다면, 낙관적 락(Optimistic Lock)은 **"동시에 수정될 확률은 낮지만, 혹시라도 꼬이면 마지막 사람이 앞사람의 노력을 지워버리는 경우"**를 방지하기 위해 사용합니다.

구체적으로 어떤 데이터에 낙관적 락이 어울리는지 예시를 들어드릴게요.

---

### 1. 게시글 또는 공지사항 수정
가장 대표적인 예시입니다. 
* **상황:** 관리자 A와 관리자 B가 동시에 공지사항 내용을 수정하러 들어왔습니다. 
* **문제:** A가 내용을 길게 수정하고 '저장'을 눌렀는데, 1초 뒤에 오타 하나만 고친 B가 '저장'을 누르면 A가 고생해서 쓴 내용이 증발하고 오타만 고쳐진 버전으로 덮어씌워집니다 (Lost Update Problem).
* **해결:** 낙관적 락을 걸면 B가 저장하려 할 때 "이미 다른 사용자에 의해 수정되었습니다"라는 안내를 띄워 A의 노력을 보호할 수 있습니다.

### 2. 사용자 프로필 정보
* **상황:** 사용자가 웹 브라우저 탭을 두 개 띄워놓고 한쪽에서는 '닉네임'을 바꾸고, 다른 한쪽에서는 '자기소개'를 바꿉니다.
* **이유:** 프로필 수정은 빈번하게 일어나지 않으므로 DB에 미리 락을 걸어둘 필요(비관적 락)는 없지만, 데이터의 최종 상태가 엉뚱하게 섞이는 것은 막아야 합니다.

### 3. 설정 값 (App Config/Preferences)
* **상황:** 시스템의 운영 환경 설정(예: 점검 모드 ON/OFF, 배너 노출 여부 등)을 수정할 때입니다.
* **이유:** 이런 설정은 보통 한 명이 수정하곤 하지만, 드물게 운영진 여럿이 동시에 접근할 때 데이터 일관성을 지키기 위해 사용합니다.

### 4. 예약 시스템의 '상태' (확정 전)
* **상황:** 상담 예약이나 회의실 예약에서 '승인 대기' 상태의 데이터를 처리할 때입니다.
* **이유:** 수치(돈)가 아니더라도, 누군가 '승인' 버튼을 누르는 순간 다른 사람이 '반려' 버튼을 눌러버리면 프로세스가 꼬일 수 있습니다.

### 💡 한눈에 보는 선택 기준

| 구분 | 낙관적 락 (Optimistic) | 비관적 락 (Pessimistic) |
| :--- | :--- | :--- |
| **핵심 마인드** | "충돌 안 나겠지? 나면 그때 취소하자." | "무조건 충돌 난다. 일단 문부터 잠그자." |
| **비용** | 낮음 (DB 자원을 점유하지 않음) | 높음 (트랜잭션 동안 DB 자원 점유) |
| **주요 예시** | 게시글, 프로필, 설정 정보 등 일반 데이터 | **계좌 잔액, 상품 재고, 티켓 예매** 등 수량 데이터 |
| **사용자 경험** | 가끔 "다시 시도하세요" 메시지를 봄 | 수정 중일 때 다른 사용자는 대기(Wait)함 |



### 수학적/물리적 관점에서의 직관
물리학에서 **임계 영역(Critical Section)**에 진입할 때, 비관적 락은 입구에서부터 통행을 차단하는 '물리적 장벽'이고, 낙관적 락은 일단 들여보내 주되 나갈 때 '검문(Version 체크)'을 해서 위반 사항이 있으면 그동안의 작업을 무효화(Rollback)하는 방식이라고 보시면 됩니다.

지금 구상 중이신 **게시판 중심의 프로젝트**라면, 게시글 수정이나 사용자 정보 수정에는 **낙관적 락** 정도만 걸어주어도 충분히 안정적인 시스템을 만드실 수 있을 거예요!
