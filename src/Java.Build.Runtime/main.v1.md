
##
- Bytecode.`.class`


- Java.Runtime.`ByteCode👉MachineCode`




##
자바에서 **Bytecode**가 **Machine Code**로 변환되는 시점은 한마디로 "프로그램이 실행되고 있는 중(Runtime)"입니다.

하지만 자바는 효율성을 위해 이 변환 작업을 두 가지 방식(인터프리터와 JIT 컴파일러)으로 섞어서 처리합니다. 구체적인 시점과 원리를 정리해 드릴게요.

---

### 1. 기본 원칙: "필요할 때 바꾼다"

자바 컴파일러(`javac`)가 만든 `.class` 파일은 CPU가 이해할 수 없는 **Bytecode** 상태입니다. 이를 실행하기 위해 JVM(Java Virtual Machine)은 두 가지 단계를 거칩니다.

* **인터프리팅 (프로그램 시작 즉시):**
JVM은 프로그램을 실행하자마자 Bytecode를 한 줄씩 읽어서 바로 기계어로 번역해 실행합니다. 이 방식은 실행은 빠르지만, 매번 번역해야 하므로 전체적인 속도는 느립니다.
* **JIT(Just-In-Time) 컴파일 (실행 도중):**
JVM은 실행 중에 "자주 호출되는 코드(Hotspot)"를 찾아냅니다. 이 코드들은 인터프리터로 매번 번역하는 대신, 통째로 기계어로 컴파일해서 메모리에 캐싱해둡니다. **이때가 진짜 Bytecode가 완전한 Machine Code로 변환되는 핵심 시점입니다.**

---

### 2. 단계별 변환 과정

JVM 내부에서 일어나는 시점별 변화는 다음과 같습니다.

| 단계 | 수행 시점 | 변환 주체 | 결과물 |
| --- | --- | --- | --- |
| **빌드 타임** | 개발자가 코드를 빌드할 때 | `javac` | Bytecode (`.class`) |
| **런타임 (초기)** | 애플리케이션 실행 직후 | 인터프리터 | Machine Code (즉시 실행) |
| **런타임 (중기)** | 특정 메서드가 반복 실행될 때 | **JIT 컴파일러** | **Native Machine Code** (최적화됨) |

---

### 3. 왜 한꺼번에 안 바꾸나요?

모든 코드를 처음부터 기계어로 바꾸지 않는 이유는 두 가지입니다.

1. **플랫폼 독립성:** 미리 기계어로 다 바꿔버리면 윈도우용, 맥용, 리눅스용 파일을 따로 만들어야 합니다. Bytecode 상태로 두어야 어떤 환경의 JVM에서도 실행될 수 있습니다.
2. **최적화:** JIT 컴파일러는 프로그램이 실제로 돌아가는 상황을 지켜보며 코드를 최적화합니다. 미리 컴파일하는 것보다 **실행 중에(Runtime)** 분석해서 기계어로 바꾸는 것이 특정 상황에서 더 빠른 성능을 낼 때가 많습니다.

> **요약하자면:**
> 자바 코드는 실행되는 순간부터 조금씩 기계어로 바뀌기 시작하며, 특히 **성능이 중요한 '뜨거운(Hot)' 코드들은 JIT 컴파일러에 의해 실행 도중에 실시간으로 기계어로 완전히 변환**되어 메모리에 저장됩니다.

혹시 JIT 컴파일러가 구체적으로 어떤 기준으로 코드를 선별하는지(Hotspot 탐지 방식) 더 궁금하신가요?
