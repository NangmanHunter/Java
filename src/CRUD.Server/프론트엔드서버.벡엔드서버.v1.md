##
프론트엔드와 백엔드 서버를 분리해서 운영할 때, 클라이언트(브라우저)와 각 서버 간의 통신 흐름은 크게 **[정적 파일 요청]**과 **[데이터 요청/응답]**의 두 단계로 나뉩니다.

순서대로 정리하면 다음과 같습니다.

---

### 1단계: 프론트엔드 정적 리소스 로드
사용자가 브라우저 주소창에 URL을 입력했을 때 일어나는 과정입니다.

1.  **Request (브라우저 → 프론트 서버):** 브라우저가 프론트엔드 서버에 접속하여 HTML, CSS, JS 파일을 요청합니다.
2.  **Response (프론트 서버 → 브라우저):** 프론트 서버는 빌드된 정적 파일(Index.html 등)을 브라우저로 보냅니다.
3.  **렌더링:** 브라우저는 받은 파일을 해석해 화면을 그립니다. 이때까지는 백엔드와 통신이 발생하지 않을 수 있습니다.

### 2단계: 백엔드 API 데이터 통신 (핵심)
화면이 뜬 후, 실제 데이터를 채우기 위해 백엔드 서버와 주고받는 과정입니다.

1.  **Request (브라우저 → 백엔드 서버):** 브라우저에서 실행 중인 JavaScript(Fetch API, Axios 등)가 백엔드 서버의 특정 API 주소(예: `https://api.example.com/posts`)로 데이터를 요청합니다.
    * 이때 **CORS(Cross-Origin Resource Sharing)** 체크가 발생할 수 있습니다. (주소가 다르기 때문)
2.  **Processing (백엔드 서버):** 백엔드 서버는 요청을 받아 DB에서 데이터를 조회하거나 비즈니스 로직을 처리합니다.
3.  **Response (백엔드 서버 → 브라우저):** 백엔드는 처리 결과를 보통 **JSON** 형식으로 브라우저에 전달합니다.
4.  **Update:** 브라우저의 JavaScript가 받은 JSON 데이터를 기반으로 화면의 일부분만 동적으로 업데이트합니다.



---

### 전체 흐름 요약 표

| 순서 | 주체 | 동작 | 내용 |
| :--- | :--- | :--- | :--- |
| **1** | **브라우저 → 프론트** | **Request** | "화면 그릴 파일(HTML/JS) 좀 줘." |
| **2** | **프론트 → 브라우저** | **Response** | 정적 파일 전달 (화면 나타남) |
| **3** | **브라우저 → 백엔드** | **Request** | "게시물 목록 데이터(JSON) 좀 줘." |
| **4** | **백엔드 → 브라우저** | **Response** | 실제 데이터(JSON) 전달 |

### 핵심 차이점
전통적인 방식(SSR, 서버 사이드 렌더링)은 서버가 데이터가 포함된 HTML을 통째로 구워서 주지만, 이렇게 서버를 분리하면 **프론트엔드는 "틀"만 주고, 백엔드는 "알맹이(데이터)"만 주는 구조**가 됩니다.

이 과정에서 프론트엔드 서버와 백엔드 서버는 서로 직접 통신하기보다, **사용자의 브라우저가 중간에서 양쪽 서버에 번갈아 요청**을 보내는 형태가 기본입니다. (물론 서버 환경에 따라 프론트 서버가 백엔드에 직접 요청을 보내는 경우도 있습니다.)
