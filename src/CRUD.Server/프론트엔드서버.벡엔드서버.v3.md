##

프록시(Proxy)를 어디서 돌리느냐는 **"어떤 목적"**으로 프록시를 쓰느냐에 따라 위치가 달라집니다. 

하지만 지금처럼 프론트와 백엔드를 나누어 개발하는 상황에서 말하는 프록시는 보통 **프론트엔드 쪽**에서 설정을 시작하는 경우가 많습니다.



---

### 1. 개발 환경에서의 프록시 (프론트엔드 서버)
React나 Vue 같은 프레임워크로 로컬에서 개발할 때 발생하는 **CORS(Cross-Origin Resource Sharing) 문제**를 해결하기 위해 사용합니다.

* **위치:** 프론트엔드 개발 서버 (Node.js 환경)
* **원리:** 브라우저가 백엔드(`:8080`)로 직접 요청을 보내면 보안상 차단될 수 있으니, 일단 프론트 서버(`:3000`)로 요청을 보냅니다. 그러면 프론트 서버가 대신 백엔드 서버에 데이터를 물어보고 결과를 전달해 줍니다.
* **설정 예시:** `package.json`에 `"proxy": "http://localhost:8080"`를 추가하거나 `setupProxy.js` 파일을 만듭니다.


### 2. 운영 환경에서의 프록시 (Nginx 등 별도 서버)
실제 서비스를 배포했을 때는 프론트엔드와 백엔드 **'앞'**에 별도의 웹 서버를 두어 프록시 역할을 맡깁니다.

* **위치:** Nginx, Apache, 또는 Cloudflare 같은 인프라 계층
* **원리:** 사용자가 하나의 주소(예: `example.com`)로 접속하면, 프록시 서버가 경로를 보고 판단하여 교통정리를 해줍니다.
    * `/api/*` 로 오는 요청 → **백엔드 서버(Spring)**로 전달
    * 그 외 일반 요청 → **프론트엔드 정적 파일** 전달
* **장점:** 사용자는 서버가 여러 개인지 알 필요가 없고, 보안이 강화되며 설정이 깔끔해집니다.



### 3. 백엔드 내부의 프록시
백엔드 서버 내부에서도 프록시를 쓰기도 합니다. (예: Spring Cloud Gateway)

* **위치:** 백엔드 입구
* **원리:** 여러 개의 마이크로서비스(MSA)가 있을 때, 어떤 서비스가 이 요청을 처리할지 결정하는 관문 역할을 합니다.
