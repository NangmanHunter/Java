## Origin
- Protocol + Host + Port


##
서버 단위를 **프로세스(포트)**로 이해하셨다면, CORS를 이해하기가 훨씬 수월해집니다. 브라우저 입장에서 **"다른 서버(다른 출처)"**를 판단하는 기준이 바로 그 프로세스가 점유한 **포트 번호**를 포함하기 때문입니다.

---

### 1. CORS의 핵심: "너 출처(Origin)가 어디야?"
CORS(Cross-Origin Resource Sharing)는 브라우저가 구현한 **보안 정책**입니다. 여기서 '출처(Origin)'를 결정하는 3요소는 다음과 같습니다.

1.  **Protocol** (http, https)
2.  **Host** (naver.com, localhost 등)
3.  **Port** (80, 8080, 3000 등)

**이 셋 중 하나라도 다르면 브라우저는 "남의 서버"라고 간주합니다.**



---

### 2. 왜 서버를 여러 개 띄우면 CORS가 발생할까?
앞서 말씀하신 것처럼 **포트 번호만 다르게 해서 서버를 여러 개(여러 프로세스) 띄운 상황**을 가정해 봅시다.

* **프론트엔드:** `http://localhost:3000` (React, Vue 등)
* **백엔드:** `http://localhost:8080` (Spring Boot, Node.js 등)

사용자가 브라우저에서 `3000`번 포트의 페이지를 보고 있는데, 그 페이지가 자바스크립트로 `8080`번 서버에 데이터를 달라고 요청합니다. 이때 브라우저는 이렇게 판단합니다.
> "어? 너는 지금 **3000번** 동네(Origin)에 살고 있는데, 왜 갑자기 **8080번** 동네에 가서 데이터를 가져오려고 해? 이거 위험할 수 있으니 일단 막을게!"

이게 바로 CORS 에러입니다.

---

### 3. 서버는 잘못이 없다 (브라우저의 참견)
재밌는 점은, **서버(프로세스)끼리는 아무런 문제가 없다는 것**입니다.

1.  브라우저가 백엔드 서버(`8080`)에 요청을 보냅니다.
2.  백엔드 서버는 아무 의심 없이 데이터를 잘 처리해서 응답을 보냅니다. (200 OK)
3.  하지만 응답을 받은 **브라우저**가 응답 헤더를 까보더니, "백엔드가 내(3000번) 이름을 허락 목록에 안 넣어줬네?"라며 **받은 데이터를 중간에서 버려버립니다.**

즉, 서버는 일을 다 했는데 브라우저가 보안을 이유로 가로채는 상황인 거죠.

---

### 4. 해결 방법: "얘는 우리 편이야"라고 말해주기
이 문제를 해결하려면 **백엔드 서버(프로세스)가 응답을 줄 때**, 브라우저에게 "3000번 포트에서 오는 요청은 내가 허락한 거니까 안심하고 통과시켜줘"라는 메시지를 담아서 보내야 합니다.

* **HTTP 응답 헤더에 추가:** `Access-Control-Allow-Origin: http://localhost:3000`



---

### 요약
* 서버를 포트별로 여러 개 띄우면(멀티 프로세스), 브라우저는 이들을 **완전히 남남**으로 봅니다.
* **포트가 1이라도 다르면** 브라우저의 보안 정책(CORS)이 발동합니다.
* 이를 해결하려면 백엔드 코드에서 특정 포트(프론트엔드)를 **허용(Allow)**하는 설정을 명시적으로 해줘야 합니다.

결국 서버를 **프로세스/포트 단위**로 쪼개서 관리하기 시작하면, 이 CORS라는 녀석은 반드시 만나게 되는 통과 의례 같은 존재입니다. 현재 자바(Spring)나 다른 프레임워크에서 백엔드 서버를 띄워서 테스트 중이신가요? 어느 쪽 설정을 도와드릴까요?
