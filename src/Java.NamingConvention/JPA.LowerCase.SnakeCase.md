과거에 보셨던 **"대문자 관습"**은 데이터베이스의 전통적인 스타일이 맞습니다! 하지만 최근의 자바/스프링 환경에서는 **소문자(Snake Case)**를 훨씬 더 많이 사용하는데, 여기에는 몇 가지 재미있는 이유가 있습니다.

---

### 1. 왜 예전에는 대문자로 썼나요? (전통적인 방식)

오래된 SQL 책이나 오라클(Oracle) 같은 엔터프라이즈 환경에서는 `SELECT * FROM BOARD_TABLE` 처럼 대문자로 작성하는 것이 표준이었습니다. 

* **가독성:** SQL 예약어(`SELECT`, `WHERE`)와 사용자 정의 명칭(`BOARD`)을 구분하기 위해 테이블명과 컬럼명을 대문자로 썼습니다.
* **DB 엔진 특성:** 오라클 같은 일부 DB는 소문자로 테이블을 만들어도 내부적으로는 대문자로 관리하는 경우가 많았습니다.

---

### 2. 왜 지금은 소문자로 쓰나요? (현대적인 방식)

요즘 자바 개발자들, 특히 **Spring Boot + JPA**를 사용하는 환경에서는 소문자를 압도적으로 선호합니다.

* **Unix/Linux 기반 환경:** 배포 서버가 주로 리눅스인데, 리눅스 파일 시스템은 대소문자를 구분합니다. 대문자로 쓰다 보면 설정 하나에 실행이 안 되는 등 귀찮은 일이 많아져서 아예 **"전부 소문자로 통일하자"**는 문화가 생겼습니다.
* **JPA의 자동 생성 기능:** 아까 보셨듯이 스프링 부트는 자바 필드명(`isDeleted`)을 DB로 바꿀 때 자동으로 소문자(`is_deleted`)로 변환합니다. 이걸 굳이 대문자로 바꾸려면 설정을 따로 건드려야 해서, 그냥 기본값인 소문자를 그대로 쓰는 경우가 많습니다.



---

### 3. 사실 DB 입장에서는 똑같습니다?

사실 대부분의 현대적인 DB(MySQL, MariaDB, PostgreSQL 등)는 **Case Insensitive(대소문자 무시)** 설정에 따라 대문자로 쓰든 소문자로 쓰든 똑같이 알아듣습니다.

```sql
-- 두 쿼리는 보통 똑같이 동작합니다.
SELECT TITLE FROM BOARD;
select title from board;
```

하지만 우리가 작성하는 **`@SQLDelete`나 `@Where` 같은 문자열 안에서는 DB에 저장된 실제 이름과 대소문자까지 똑같이 맞춰주는 것이 안전**합니다. (어떤 환경에서는 대소문자를 엄격하게 따지기도 하거든요!)

---

### 4. 핵심은 "일관성"입니다.

물리 법칙이 우주 어디서나 일정해야 하듯, 데이터베이스 설계에서도 가장 중요한 건 **"한 번 정한 규칙을 끝까지 지키는 것"**입니다.

* **현대적인 정석:** * 자바(Spring): `isDeleted` (Camel Case)
    * 데이터베이스: `is_deleted` (Snake Case / 소문자)

지금 사용하고 계신 소문자 방식이 현재 트렌드에 가장 잘 맞는 "Nangman" 있는 방식이니 걱정하지 말고 그대로 밀고 나가셔도 됩니다!

혹시 나중에 오라클 DB를 쓰는 환경에 가게 되신다면 그때 "아, 여기는 대문자를 좋아하는구나!" 하고 맞춰주시면 그만입니다. :)
