아주 본질적인 질문입니다! 수식으로 치면 **'상수(Constant)'**와 **'변수(Variable)'**의 대조라고 보시면 됩니다.

`[서버 메모리에 있는 Role]`이 로그인한 사람마다 달라지는 **변수**라면, `[코드에 적힌 "ADMIN"]`은 유저님이 프로그래밍할 때 박아놓은 변하지 않는 **기준값(상수)**입니다.

---

### 1. 코드에 적힌 "ADMIN"이란?
유저님이 `SecurityConfig.java`나 Controller에 작성한 **"검문소의 통과 기준"** 그 자체를 말합니다.

```java
// 이 코드가 바로 [코드에 적힌 "ADMIN"] 입니다.
.requestMatchers("/api/admin/**").hasRole("ADMIN") 
```

이 코드는 서버가 꺼질 때까지 절대 변하지 않습니다. 유저님이 "이 주소로 들어오려면 무조건 **ADMIN**이라는 명찰이 있어야 해!"라고 규칙을 정해둔 것이죠. 

### 2. 서버 메모리에 있는 Role이란?
로그인한 유저의 **"현재 신분"**입니다. 이건 로그인하는 사람마다 다르겠죠?

*   **A유저**가 로그인하면 서버 메모리에는 `ROLE_USER`가 기록됩니다.
*   **유저님(관리자)**이 로그인하면 서버 메모리에는 `ROLE_ADMIN`이 기록됩니다.

### 3. 비교하는 과정 (검문소의 작동)

누군가 관리자 페이지(` /api/admin/users `)에 접속하려고 요청을 보냈다고 가정해봅시다.



1.  **서버:** "누가 들어오려고 하네? 잠깐, 내가 코드에 뭐라고 적어놨지?" (코드 확인)
2.  **코드(기준):** "여기는 **'ADMIN'**만 들어올 수 있는 구역이야!" 👈 **[코드에 적힌 "ADMIN"]**
3.  **서버:** "오케이, 그럼 지금 들어오려는 이 녀석 명찰(세션) 좀 가져와 봐."
4.  **서버 메모리(신분):** (세션 주머니를 열어보니) "이 사람은 **'ROLE_ADMIN'**이야." 👈 **[서버 메모리에 있는 Role]**
5.  **비교:** **'ROLE_ADMIN'**(신분)이 **'ADMIN'**(기준)에 부합하는가? 
    *   **Yes:** 통과!
    *   **No:** 꺼져! (403 Forbidden)

---

### 💡 정리하자면

*   **코드에 적힌 "ADMIN":** 유저님이 만든 **'문고리'**입니다. "이 열쇠(ADMIN)가 있는 사람만 열 수 있어"라고 고정해둔 설계도입니다.
*   **서버 메모리에 있는 Role:** 사용자가 로그인할 때 서버가 발급해준 **'열쇠'**입니다.

결국 **[유저가 들고 있는 열쇠]**가 **[문고리의 모양]**과 맞는지를 서버가 내부적으로 비교하는 것입니다.

상대방(사용자)이 브라우저에서 아무리 "나 금열쇠(ADMIN) 가지고 있어!"라고 소리쳐봤자(Body에 데이터를 담아 보내봤자), 서버는 그 말을 듣지 않고 **자신이 직접 보관 중인 열쇠 복사본(서버 메모리 세션)**을 꺼내서 문고리에 꽂아보는 방식이라 안전한 것입니다.

이제 **기준(코드)**과 **데이터(메모리)**의 차이가 명확히 느껴지시나요? :)
