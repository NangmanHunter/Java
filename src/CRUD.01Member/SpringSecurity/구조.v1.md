






## SpringSecurity
- DelegatingFilterProxy
- FilterChainProxy


##
* **DelegatingFilterProxy**: 서블릿 컨테이너(Tomcat 등)와 스프링 컨테이너 사이를 연결하는 다리 역할을 합니다.
* **FilterChainProxy**: 스프링 시큐리티가 관리하는 실제 보안 필터 리스트를 가지고 있는 본체입니다.


---

## 2. 인증(Authentication)의 핵심 구성 요소
"이 사용자가 누구인가?"를 확인하는 과정에서 일하는 주요 객체들입니다.

| 구성 요소 | 역할 |
| :--- | :--- |
| **SecurityContextHolder** | 보안 주체(Principal)의 세부 정보를 포함하여 응용 프로그램의 보안 컨텍스트에 대한 세부 정보가 저장되는 곳입니다. |
| **Authentication** | 현재 접근 주체의 정보와 권한을 담는 인터페이스입니다. |
| **AuthenticationManager** | 인증을 어떻게 처리할지 결정하는 API입니다. 보통 `ProviderManager`가 이 구현체 역할을 합니다. |
| **AuthenticationProvider** | 실제 인증 로직(비밀번호 검증 등)이 담긴 곳입니다. |
| **UserDetailsService** | 유저 정보를 DB나 메모리에서 로드하는 서비스입니다. |

---

## 3. 인증 처리 흐름 (Flow)
로그인 요청이 들어왔을 때 내부적으로 일어나는 순서입니다.

1.  **Http Request 수신**: 사용자가 아이디/비밀번호를 보냅니다.
2.  **토큰 생성**: `UsernamePasswordAuthenticationFilter`가 정보를 가로채서 `UsernamePasswordAuthenticationToken`을 만듭니다.
3.  **Manager에게 전달**: 생성된 토큰을 `AuthenticationManager`에게 넘깁니다.
4.  **Provider 선택**: Manager는 이 토큰을 처리할 수 있는 적절한 `AuthenticationProvider`를 찾아 인증을 위임합니다.
5.  **유저 정보 조회**: Provider는 `UserDetailsService`를 통해 DB에서 사용자 정보를 가져옵니다.
6.  **검증**: DB의 정보와 사용자가 입력한 정보를 비교합니다.
7.  **결과 저장**: 인증이 성공하면 `SecurityContextHolder`에 인증 객체를 저장하고 다음 로직으로 넘어갑니다.



---

## 4. 인가(Authorization)
인증이 완료된 후, "이 사용자가 이 페이지에 들어갈 권한이 있는가?"를 체크합니다.

* **FilterSecurityInterceptor**: 인가 처리를 담당하는 가장 마지막 필터입니다.
* **AccessDecisionManager**: 접근 허용 여부를 최종 결정합니다. (투표 기반으로 결정하는 경우가 많습니다.)

### 💡 요약하자면
스프링 시큐리티는 **필터 체인**을 통해 요청을 가로채고, **Manager**와 **Provider**를 통해 인증을 수행한 뒤, 그 결과를 **SecurityContext**에 보관하여 애플리케이션 전반에서 사용자 정보를 사용할 수 있게 해주는 구조입니다.
