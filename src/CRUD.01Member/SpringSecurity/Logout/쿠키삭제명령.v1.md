##
`.deleteCookies("JSESSIONID")`를 명시적으로 넣는 이유는 **"클라이언트(브라우저)에 남아있는 열쇠를 확실하게 폐기하기 위해서"**입니다.

이게 왜 필요한지 조금 더 자세히 설명해 드릴게요.

---

### 1. 세션 무효화 vs 쿠키 삭제
보통 로그아웃을 하면 서버에서 다음 두 가지 작업이 일어납니다.

1.  **세션 무효화 (`invalidateHttpSession(true)`)**: 서버 메모리에 저장된 "이 사용자는 로그인 상태다"라는 정보를 삭제합니다. (서버 측 작업)
2.  **쿠키 삭제 (`deleteCookies`)**: 사용자의 브라우저에 저장된 세션 ID(`JSESSIONID`) 쿠키를 삭제하라고 명령합니다. (클라이언트 측 작업)

### 2. 왜 명시적으로 삭제해야 할까?
서버에서 세션을 무효화하면, 사실 브라우저에 쿠키가 남아있더라도 그 쿠키(열쇠)는 더 이상 서버에서 쓸모가 없습니다. 하지만 다음과 같은 이유로 명시적 삭제를 권장합니다.

* **보안상의 깔끔함**: 로그아웃 후에도 브라우저 개발자 도구(F12)의 Application 탭에 세션 쿠키가 그대로 남아있으면 보안상 찜찜합니다. 명시적으로 지워주는 것이 정석입니다.
* **브라우저의 혼동 방지**: 간혹 브라우저가 만료된 쿠키를 계속 들고 있다가 다음 로그인 시도 때 서버로 보내서 예기치 않은 오류(세션 충돌 등)를 일으키는 경우가 있습니다.
* **사용자 프라이버시**: 공용 PC 등에서 다른 사용자가 내 세션 ID를 확인하는 것조차 방지하기 위함입니다.

---

### 3. 작동 원리 (HTTP 응답 헤더)
`.deleteCookies("JSESSIONID")`를 설정하면, 로그아웃 요청에 대한 응답을 보낼 때 서버가 브라우저에게 다음과 같은 헤더를 보냅니다.

> **Set-Cookie**: JSESSIONID=; Max-Age=0; Expires=Thu, 01 Jan 1970 00:00:00 GMT

브라우저는 이 헤더를 받으면 **"아, 이 쿠키는 유효기간이 끝났구나!"** 하고 즉시 삭제해 버립니다.

---

### 4. 주의사항 (중요!)
만약 커스텀 쿠키를 사용하고 계신다면(예: 자동 로그인 토큰 등), 여기에 그 이름도 같이 넣어주어야 합니다.

```java
.logout(logout -> logout
    .logoutUrl("/api/auth/logout")
    .deleteCookies("JSESSIONID", "remember-me", "any-custom-cookie") // 여러 개 등록 가능
    .logoutSuccessHandler(...)
)
```

### 요약하자면
`invalidateHttpSession`이 **서버 안의 기록**을 지우는 것이라면, `deleteCookies`는 **사용자 컴퓨터에 남은 흔적**까지 깔끔하게 지우는 "뒷정리" 과정이라고 보시면 됩니다!

지금 프론트엔드와 연동 중이시라면, 사용자 브라우저에 쿠키가 남아있지 않도록 이 설정을 넣어주시는 것이 훨씬 깔끔한 로그아웃 구현이 됩니다.
