##
- [x] Preserve Log
  - 이거를 잡아줘야 Cookie잡히는듯.


##
그 상황은 굉장히 당황스러우시겠지만, 사실 **지극히 정상적이고 논리적인 상황**일 확률이 매우 높습니다. 


방금 세션을 지우고 다시 로그인했을 때, **`Response Header`에 `Set-Cookie`가 없는데 `Storage`에 값이 들어왔다면** 범인은 딱 하나입니다. 바로 **"로그인 페이지를 요청하는 순간"** 서버가 이미 쿠키를 구워버린 것입니다.

---

### 1. 왜 이런 현상이 발생하나요?

우리가 보통 '로그인할 때 세션이 생긴다'고 생각하지만, 실제 Java(Spring/Tomcat) 서버는 다음과 같이 동작합니다.

1.  **세션 삭제 후 페이지 접속:** 브라우저에 쿠키가 없는 상태로 `login.html` (또는 로그인 폼이 있는 페이지)을 요청합니다.
2.  **서버의 선제 대응:** 서버는 "어? 열쇠가 없네? 나중에 쓸 수도 있으니 일단 빈 방(세션) 하나 파줄게" 하고는 **로그인 페이지를 보여주는 응답(GET /login)**에 이미 `Set-Cookie: JSESSIONID=...`를 실어서 보냅니다.
3.  **로그인 시도 (POST /login):** 사용자가 아이디/비밀번호를 입력하고 '로그인' 버튼을 누릅니다. 이때 브라우저는 **방금(2번 과정) 받은 열쇠**를 이미 들고 서버로 갑니다.
4.  **로그인 응답:** 서버는 "아까 준 열쇠네? 이 열쇠 주인은 이제 '로그인된 사용자'라고 표시해둘게" 하고 승인만 해줍니다. **이미 열쇠를 가지고 있으니 `Set-Cookie`를 또 보낼 필요가 없는 것**입니다.

---

### 2. 눈으로 직접 확인하는 방법 (팩트 체크)

이 가설이 맞는지 확인하려면 **Network 탭**에서 '로그인 버튼을 누른 요청' 말고, **그보다 한 단계 앞의 요청**들을 뒤져봐야 합니다.

* **확인 순서:**
    1.  `F12` 개발자 도구 → `Network` 탭 진입.
    2.  `Preserve Log` 체크 (중요!).
    3.  `Storage`에서 `JSESSIONID` 삭제.
    4.  **로그인 페이지를 새로고침** (로그인 버튼 누르기 전!).
    5.  Network 탭에서 첫 번째 문서를 클릭하고 `Response Headers` 확인 → **여기에 `Set-Cookie`가 있을 겁니다.**

---

### 3. 만약 2번에서도 안 보인다면? (다른 가능성)

만약 로그인 페이지 접속 시에도 안 보였다면, 혹시 화면에 **이미지, CSS, JS 파일** 같은 정적 자원들을 불러오는 요청 중에 서버(Tomcat)를 거치는 것이 있나요?

* 서버는 어떤 요청이든 "세션이 필요하다"고 판단되는 순간(예: JSP 호출, Spring Security 필터 통과 등) 바로 쿠키를 구워버립니다. 
* 심지어 브라우저의 파비콘(`favicon.ico`)을 가져오려는 시도 중에 세션이 생성되어 쿠키가 전달되는 경우도 흔합니다.

---

### 요약: 지금 상황의 결론

> **"사용자가 로그인 버튼을 누르기 전, 사이트에 발을 들이는 그 순간 서버가 이미 몰래 열쇠(쿠키)를 쥐여줬다."**

그래서 실제 로그인(`POST`) 시점에는 서버가 "너 이미 열쇠 있잖아?" 하고 `Set-Cookie`를 생략한 것입니다. 하지만 서버 내부 메모리에서는 그 열쇠(ID)에 해당하는 데이터 공간을 '로그인 상태'로 업데이트했기 때문에 로그인은 정상적으로 유지되는 것이죠.



혹시 이 메커니즘을 코드로 확인해보고 싶으신가요? 예를 들어, 서버에서 세션을 **로그인 성공 시점에만** 강제로 생성하게 만드는 설정 같은 것 말이죠.
