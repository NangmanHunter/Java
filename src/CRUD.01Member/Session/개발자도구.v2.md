오, 아주 예리한 발견입니다! 분명히 서버가 보내줘야 브라우저에 저장되는 건데, 왜 **Network 탭의 Response Header**에는 안 보이고 **Application 탭(저장소)**에만 들어있는지 의아하실 거예요.

여기에는 몇 가지 기술적인 이유가 있습니다. 가장 확률이 높은 순서대로 짚어드릴게요.

---

### 1. 이미 '이전 요청'에서 받아두었기 때문 (가장 유력)
세션 ID(JSESSIONID)는 **로그인하는 그 순간에만 생기는 게 아닙니다.**

* 보통 서버(Tomcat 등)는 로그인을 안 했더라도, 사용자가 우리 사이트에 **처음 접속하는 순간** 일단 세션 방을 하나 파고 세션 ID를 쿠키로 던져줍니다. (비로그인 장바구니 기능을 위해서나, 그냥 접속자 식별을 위해서요.)
* 즉, **로그인 페이지를 요청했을 때** 이미 서버로부터 `Set-Cookie`를 받아버린 상태일 수 있습니다.
* 이미 브라우저에 쿠키가 저장되어 있다면, 로그인을 눌렀을 때는 서버가 굳이 `Set-Cookie`를 다시 보낼 필요가 없습니다. (기존 열쇠를 그대로 쓰면 되니까요.)

> **테스트 방법:** 크롬 개발자 도구의 Application 탭에서 해당 쿠키를 **강제로 삭제(우클릭 -> Delete)** 하신 뒤, 새로고침을 해보세요. 그러면 Network 탭의 첫 번째 요청 Response Header에 `Set-Cookie`가 다시 나타날 겁니다.

---

### 2. '로그인 성공' 후 리다이렉트(Redirect) 되었기 때문
로그인 버튼을 눌렀을 때 일어나는 일입니다.

1.  사용자: `POST /login` (ID/PW 전송)
2.  서버: 로그인 성공! 응답으로 `302 Found`와 함께 `Set-Cookie: JSESSIONID=...`를 보냄. (이게 우리가 찾아야 할 헤더입니다.)
3.  브라우저: 0.1초 만에 `Set-Cookie`를 읽어 저장하고, 서버가 알려준 메인 페이지(`/main`)로 이동함.

이 과정이 너무 빨라서 Network 탭에서 `login` 요청 기록을 놓치고, 이미 쿠키가 저장된 후의 `main` 요청 헤더만 보고 계실 가능성이 큽니다.

---

### 3. 브라우저의 '보안 정책' (Rare case)
최근 브라우저들은 보안상 `Set-Cookie` 헤더를 네트워크 탭에서 가끔 숨기거나 별도의 경고 표시와 함께 보여주는 경우가 있습니다. 하지만 이 경우에도 **Application** 탭에는 데이터가 남습니다.

---

### 확인 사격: 이렇게 한번 해보세요!

가장 확실하게 눈으로 확인하는 방법입니다.

1.  **Application** 탭에 가서 떠 있는 `JSESSIONID`를 마우스 우클릭으로 **Delete** 합니다. (완전 초기화)
2.  **Network** 탭으로 가서 상단의 **Preserve Log**(로그 보존) 체크박스를 체크합니다.
3.  페이지를 새로고침하거나 로그인을 다시 시도합니다.
4.  Network 탭 리스트 중 가장 위쪽이나 `login`이라 적힌 항목의 **Response Headers**를 꼼꼼히 보세요. `Set-Cookie`가 아주 반갑게 서 있을 겁니다.

### 요약하자면:
**"Application 탭에 이미 있다"**는 건, 이전에 서버랑 통신할 때 **이미 열쇠를 받아두었다**는 뜻입니다. 서버는 한 번 준 열쇠를 뺏거나 바꾸지 않는 이상 다시 보내지 않거든요!
