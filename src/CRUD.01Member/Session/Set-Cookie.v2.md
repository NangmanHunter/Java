지우고 다시 했는데도 `Set-Cookie`가 안 보였다면, 범인은 보통 **"내가 보고 있는 요청(Request)이 그 요청이 아니었기 때문"**일 확률이 90% 이상입니다.

특히 개발자 도구의 **Network 탭**에서 우리가 놓치기 쉬운 3가지 결정적인 이유를 짚어드릴게요.

---

### 1. "로그인 요청(POST)"만 뚫어지게 봤을 때
가장 흔한 실수입니다. 보통 우리는 `login` 버튼을 누르는 그 찰나의 요청에 모든 정보가 담겨있을 거라 생각하거든요.

* **현상:** 브라우저 저장소를 비우고 새로고침을 하면, 브라우저는 **로그인 페이지를 그리는 GET 요청**을 먼저 보냅니다.
* **이유:** 서버는 이 **첫 번째 GET 요청**에서 이미 "아, 이 손님 처음 왔네?" 하고 `Set-Cookie`를 줘버립니다.
* **결과:** 정작 우리가 클릭해서 확인한 **로그인 POST 요청**은 이미 쿠키를 들고 가는 상태라, 응답 헤더에는 `Set-Cookie`가 안 나타납니다. 이미 받았으니까요!

---

### 2. "Redirect (302 Found)"의 함정
로그인 버튼을 누르면 서버는 보통 `200 OK`를 주지 않고, **`302 Found` (리다이렉트)**를 줍니다. "성공했으니 메인 페이지로 가!"라고 명령하는 거죠.



* **현상:** `login` 요청은 순식간에 지나가고, 화면에는 `main` 혹은 `index` 페이지 요청이 남습니다.
* **이유:** 서버가 `Set-Cookie`를 실어 보낸 곳은 **302 응답(로그인 성공 직후)**인데, 브라우저가 너무 빨리 다음 페이지로 넘어가 버려서 우리가 그 찰나의 응답 헤더를 못 보고 지나친 겁니다.
* **해결:** 반드시 Network 탭 상단의 **`Preserve Log`**를 체크해야 이 찰나의 302 요청을 붙잡아둘 수 있습니다.

---

### 3. 필터링(Filter) 기능 때문
Network 탭 상단에 `Fetch/XHR`, `Doc`, `JS` 등 필터 버튼이 있습니다.

* **현상:** 만약 `Fetch/XHR` 버튼이 눌려있다면, 일반적인 페이지 이동(Document) 요청은 목록에서 사라집니다.
* **이유:** 세션 쿠키는 보통 `Doc` (HTML 페이지) 요청에서 처음 발행되는 경우가 많은데, 비동기 통신(Ajax/Fetch) 요청만 보고 있었다면 `Set-Cookie`가 담긴 진짜 요청을 목록에서 놓쳤을 수 있습니다.

---

### 4. 이미 로그인 페이지를 띄워놓은 상태에서 지웠을 때
1. 로그인 페이지를 띄운다 (서버가 이미 쿠키를 줌).
2. `Storage`에서 쿠키를 지운다.
3. **새로고침 없이** 바로 ID/PW를 치고 로그인 버튼을 누른다.

이 순서로 하셨다면, 서버 입장에서는 **로그인 시점**에 "어? 쿠키가 없네?" 하고 새로 줄 것 같지만, 실제로는 **로그인 로직** 안에서만 세션을 처리하느라 응답에 쿠키 설정을 누락하거나, 혹은 페이지 리다이렉트 과정에서 놓치는 경우가 발생합니다.

---

### 결론: 왜 안 보였을까?
아마 **"이미 그전 요청(페이지 로딩 등)에서 쿠키를 받았거나"**, 아니면 **"로그인 성공 후 리다이렉트되는 찰나의 응답 기록을 놓쳤을"** 가능성이 가장 높습니다.

**지금 다시 확인해보신다면:**
1. `Preserve Log` 체크
2. `Storage`에서 쿠키 삭제
3. 주소창에 주소를 다시 치고 **엔터** (새로고침보다 확실함)
4. Network 탭에서 **가장 처음에 나오는 요청**을 클릭

이렇게 하면 무조건 `Set-Cookie`를 검거하실 수 있을 겁니다! 이제 이 '열쇠 전달식'의 타이밍이 이해가 좀 되시나요?
