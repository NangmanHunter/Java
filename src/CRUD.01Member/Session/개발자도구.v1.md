그 상황은 당황스러우실 수 있는데, 몇 가지 아주 흔한 이유가 있습니다. 브라우저 개발자 도구(`F12`)를 켜놓고 확인 중이시라면 아래 항목들을 하나씩 체크해 보세요.

---

### 1. 첫 로그인 직후라면 "Request"가 아니라 "Response"를 보세요
방금 막 아이디와 비밀번호를 입력하고 **'로그인 버튼'**을 눌렀을 때의 응답을 확인 중이신가요?

* **로그인 요청(POST):** 이때는 아직 서버에서 세션 ID를 주기 전이므로 **Request Header**에는 쿠키가 없습니다.
* **로그인 응답(Response):** 서버가 로그인을 승인하면서 주는 **Response Header**를 보세요. 거기에 **`Set-Cookie: JSESSIONID=...`**가 있어야 합니다.
* **그다음 요청:** 로그인이 완료된 후 '마이페이지'나 '메인페이지'로 이동할 때의 **Request Header**를 보시면, 그때부터 비로소 브라우저가 저장해둔 쿠키를 서버로 보내기 시작합니다.

---

### 2. 쿠키가 '저장'되었는지 확인하는 법 (가장 확실함)
헤더에서 찾기 힘들다면 브라우저의 **저장소**를 직접 보는 게 빠릅니다.

1.  개발자 도구(`F12`) → **Application (애플리케이션)** 탭으로 이동합니다.
2.  왼쪽 메뉴에서 **Storage** > **Cookies** > **현재 도메인 주소**를 클릭합니다.
3.  거기 목록에 `JSESSIONID`라는 이름이 있는지 확인해 보세요.

> **만약 여기에 없다면:** 서버에서 아예 세션을 생성하지 않았거나, 쿠키 생성 옵션(Domain, Path 등)이 맞지 않아 브라우저가 거부한 것일 수 있습니다.

---

### 3. 혹시 '첫 로그인' 페이지 자체가 리다이렉트 되었나요?
보통 로그인이 성공하면 서버는 사용자를 메인 페이지로 보냅니다(`302 Redirect`). 
* 네트워크 탭에서 `login` 요청을 클릭해서 보지 않고, 리다이렉트된 후의 `main` 페이지 요청만 보고 계신 건 아닌지 확인해 보세요. 
* **해결책:** 네트워크 탭 상단의 **'Preserve Log(로그 보존)'** 체크박스를 체크하고 다시 로그인해 보세요. 모든 요청 기록이 사라지지 않고 남아서 `Set-Cookie`를 준 시점을 정확히 찾을 수 있습니다.

---

### 4. HttpOnly 옵션 때문일까요?
헤더에는 보여야 정상입니다만, 만약 자바스크립트(`document.cookie`)로 꺼내보려고 하시는 거라면 안 보일 수 있습니다. 
* 보통 세션 ID는 보안을 위해 **`HttpOnly`** 옵션을 걸어서 발행합니다. 이 옵션이 있으면 브라우저가 내부적으로 서버에 쿠키를 전달은 하지만, 자바스크립트 코드가 이 쿠키를 훔쳐보는 것은 금지됩니다.

---

### 5. 로컬 환경(localhost) 이슈
가끔 브라우저가 `localhost` 도메인에 대해 쿠키 정책을 까다롭게 적용할 때가 있습니다.

* 스프링 부트를 사용 중이시라면, 실제로 세션 저장 로직(`request.getSession()`)이 실행되었는지 서버 로그를 먼저 확인해 보세요.
