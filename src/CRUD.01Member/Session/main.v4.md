로그인 세션 방식의 핵심은 **"서버가 사용자를 기억하기 위해 열쇠(ID)를 발행하고, 이를 금고(메모리)에 보관하는 것"**입니다.

로그인이 성공했을 때 서버가 클라이언트(브라우저)에게 보내주는 것은 정확히 **세션 ID(Session ID)**입니다. 이 과정을 단계별로 뜯어보면 메커니즘이 명확해집니다.

---

## 1. 로그인 성공 직후의 과정

사용자가 아이디와 비밀번호를 보내서 인증에 성공하면, 서버는 내부적으로 다음 작업을 수행합니다.

1.  **세션 생성:** 서버의 메모리(또는 DB/Redis)에 해당 사용자를 위한 전용 공간을 만듭니다.
2.  **세션 ID 발행:** 그 공간에 접근할 수 있는 무작위의 고유한 문자열(예: `JSESSIONID=ab123...`)을 생성합니다.
3.  **응답 전송:** 이 **세션 ID**를 HTTP 응답 헤더의 **`Set-Cookie`**라는 필드에 담아서 브라우저로 보냅니다.

> **서버가 보내는 것:** `Set-Cookie: SESSION_ID=xyz123; HttpOnly; Path=/`

---

## 2. 세션 방식의 전체 메커니즘



전체 흐름을 보면 왜 세션 ID가 중요한지 알 수 있습니다.

* **브라우저의 역할:** 서버로부터 받은 세션 ID를 자신의 **쿠키 저장소**에 자동으로 저장합니다.
* **이후의 요청:** 사용자가 다른 페이지(예: 마이페이지)로 이동할 때마다, 브라우저는 쿠키 저장소에 있던 세션 ID를 꺼내서 요청 헤더에 자동으로 포함해 보냅니다.
* **서버의 확인:** 서버는 들어온 세션 ID를 보고 "아, 이 열쇠는 아까 내가 만든 1번 사용자(홍길동) 꺼네!"라고 확인하며 로그인을 유지해 줍니다.

---

## 3. 핵심 요약: 서버는 무엇을 주는가?

| 항목 | 내용 |
| :--- | :--- |
| **전달 매체** | HTTP 응답 헤더의 `Set-Cookie` |
| **실제 데이터** | 암호화되거나 무작위로 생성된 **Session ID** (문자열) |
| **상태 저장소** | **서버 메모리** (서버가 사용자 정보를 들고 있음) |

### 왜 이렇게 할까?
서버는 사용자의 민감한 정보(비밀번호, 개인정보 등)를 브라우저에 직접 보내고 싶지 않기 때문입니다. 대신 아무 의미 없는 **'번호표(세션 ID)'**만 주고, 실제 중요한 정보는 서버 본인의 안전한 메모리에 보관하는 방식인 것이죠.

혹시 이 과정에서 쿠키와 세션의 차이점이나, 서버 메모리(영속성) 관련해서 더 궁금한 부분이 있으신가요?
