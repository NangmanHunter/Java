그래서 다음과 같은 규칙이 브라우저 엔진에 **본능(기본 동작)**으로 탑재되었습니다.

*   **규칙 1:** 서버 응답에 `Set-Cookie`가 있으면, 브라우저는 묻지도 따지지도 않고 로컬 저장소에 저장한다.
*   **규칙 2:** 이후 그 서버(도메인)에 다시 요청을 보낼 때는, 저장된 쿠키를 **자동으로** `Cookie` 헤더에 담아서 보낸다.


### 3. 그런데 여기서 "자동"의 범위가 중요합니다
아무 데나 다 던지는 건 아닙니다. 쿠키는 나름의 **'조건'**을 보고 자동으로 던집니다.

*   **도메인 조건:** `naver.com`에서 받은 쿠키는 `google.com`에 갈 때는 절대 던지지 않습니다.
*   **경로 조건:** 특정 폴더(`/마이페이지`)에서만 쓰라고 설정된 쿠키는 다른 폴더로 갈 때 안 던질 수도 있습니다.
*   **유효 기간:** 유효 기간이 지난 쿠키는 브라우저가 스스로 쓰레기통에 버리고 던지지 않습니다.

---

### 4. 이 "자동성" 때문에 생기는 문제 (보안)
쿠키가 너무 알아서 잘 던져지다 보니 보안 취약점도 생겼습니다. 그게 바로 유명한 **CSRF(Cross-Site Request Forgery)** 공격입니다.

> **예시:** 내가 은행 사이트에 로그인된 상태에서, 나쁜 놈이 만든 낚시 사이트에 접속만 해도 브라우저는 "어? 은행 서버에 요청 보내네? 그럼 내가 가진 은행 쿠키(세션 ID)도 같이 던져야지!" 하고 자동으로 신분증을 제출해 버립니다.

이런 부작용을 막기 위해 최근에는 `SameSite` 같은 옵션을 추가해서 **"진짜 그 사이트에서 직접 보낸 요청일 때만 자동으로 던져라"** 같은 제약 조건을 더 정교하게 발전시키고 있습니다.

### 요약
쿠키는 태생부터 **"개발자가 직접 챙기지 않아도 브라우저가 알아서 신분증을 제출하는 자동 시스템"**으로 설계되었습니다. 이 '자동화' 덕분에 우리는 로그인 한 번으로 웹서핑을 편하게 할 수 있는 것이죠!
