## 1. 세션(Session) vs JWT(JSON Web Token) 핵심 차이

가장 큰 차이는 **"로그인 정보를 어디에 저장하느냐"**입니다.

### 세션 방식 (Server-side Storage)
서버의 메모리나 DB에 로그인 상태를 저장합니다.
* **흐름:** 사용자가 로그인하면 서버는 '세션 ID'를 발행해 브라우저(쿠키)로 보내고, 서버 메모리에는 그 ID에 대응하는 정보를 저장합니다.
* **장점:** 보안성이 비교적 높고, 서버에서 특정 사용자의 로그아웃을 강제로 제어(세션 만료)하기 쉽습니다.
* **단점:** 사용자가 많아지면 서버 메모리 부하가 커지며, 서버가 여러 대일 경우 세션을 공유하기 위한 별도의 설정(Redis 등)이 필요합니다.

### JWT 방식 (Client-side Storage)
로그인 정보를 토큰 자체에 담아 클라이언트가 들고 있게 합니다.
* **흐름:** 서버는 로그인 성공 시 유저 정보를 암호화한 '토큰'을 전달합니다. 이후 클라이언트는 모든 요청에 이 토큰을 실어 보내고, 서버는 별도의 저장소 확인 없이 토큰의 유효성만 검증합니다.
* **장점:** 서버가 상태를 들고 있지 않아(Stateless) 확장이 매우 자유롭습니다. 마이크로서비스 아키텍처(MSA)나 모바일 앱 환경에서 선호됩니다.
* **단점:** 토큰 내부에 정보를 담으므로 크기가 커질 수 있고, 한 번 발행된 토큰은 유효기간이 만료될 때까지 서버가 강제로 회수하거나 제어하기 어렵습니다.

## 2. 비교 요약표

| 구분 | 세션 (Session) | JWT (Token) |
| :--- | :--- | :--- |
| **저장 위치** | 서버 메모리 또는 DB | 클라이언트 (로컬 스토리지/쿠키) |
| **상태 유지** | Stateful (상태 유지) | Stateless (무상태) |
| **확장성** | 서버 확장 시 세션 공유 설정 필요 | 서버가 늘어나도 검증 로직만 있으면 OK |
| **보안성** | 상대적으로 높음 (세션 탈취 시 서버에서 무효화 가능) | 보안 주의 필요 (토큰 탈취 시 유효기간 내 통제 불가) |
| **주요 용도** | 일반적인 웹 서비스, 단일 서버 환경 | 모바일 앱, 다중 서버 환경, API 중심 서비스 |

## 3. 이번 프로젝트에는 무엇을 쓸까요?

현재 만들고 계신 프로젝트의 성격에 따라 결정하시면 됩니다.

1.  **세션 방식을 추천하는 경우:**
    * Spring Security의 기본 동작 방식이라 학습 자료가 많습니다.
    * 사용자 수가 관리 가능한 수준이고, 구현이 직관적입니다.
    * 로그아웃 처리를 서버에서 확실하게 제어하고 싶을 때 적합합니다.

2.  **JWT 방식을 추천하는 경우:**
    * 나중에 프론트엔드를 React나 Vue로 분리하여 REST API 서버로 운영할 계획이 있을 때.
    * 서버를 여러 대 띄우는 환경을 고려하고 싶을 때.
    * 최신 트렌드인 토큰 기반 인증 방식을 직접 경험해보고 싶을 때.

**개인적인 추천:** 처음 Member 부분을 구현하신다면 **세션 방식**으로 로그인/로그아웃의 기본 원리(인터셉터나 필터 활용)를 먼저 익히시는 것을 권장합니다. 그 후에 필요에 따라 JWT로 리팩토링해보는 과정이 실력 향상에 큰 도움이 됩니다.

둘 중 어떤 방식으로 구현해보고 싶으신가요? 선택하시면 그에 맞는 Spring 구현 로직을 가이드해 드릴게요.
