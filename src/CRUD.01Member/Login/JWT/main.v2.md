##
쿠키.헤더
JWT.헤더


##
쿠키와 JWT는 둘 다 HTTP 메시지에 실려 전송되지만, 그 방식과 규칙에는 명확한 차이가 있습니다.

결론부터 말씀드리면, **쿠키 역시 HTTP 헤더에 담겨 전송됩니다.** 다만, 우리가 직접 코드로 넣는 것이 아니라 브라우저가 **자동**으로 관리한다는 점이 핵심입니다.

---

### **1. 쿠키의 전송 방식 (HTTP Header)**

서버와 브라우저가 쿠키를 주고받을 때 사용하는 전용 헤더가 정해져 있습니다.

* **서버 → 브라우저 (쿠키 저장):** 서버가 응답할 때 `Set-Cookie`라는 헤더에 값을 담아 보냅니다.
    * `Set-Cookie: sessionId=abc1234; Domain=example.com; Path=/; HttpOnly`
* **브라우저 → 서버 (쿠키 전송):** 브라우저는 저장된 쿠키 중 해당 도메인에 맞는 것이 있으면 요청 헤더의 `Cookie` 항목에 자동으로 이어 붙입니다.
    * `Cookie: sessionId=abc1234; userSetting=darkmode`



---

### **2. JWT와 쿠키의 전송 비교**

두 방식 모두 헤더를 쓰지만, "누가, 어떤 이름으로" 보내는지가 다릅니다.

| 구분 | 쿠키 (Session ID 등) | JWT (일반적인 방식) |
| :--- | :--- | :--- |
| **사용 헤더** | `Cookie` | `Authorization` |
| **전송 값 형식** | `key=value` | `Bearer <Token>` |
| **전송 주체** | **브라우저가 자동**으로 포함 | **개발자가 코드**로 직접 포함 |
| **상태 관리** | 브라우저가 관리 (Stateful) | 클라이언트(앱/웹)가 관리 (Stateless) |

---

### **3. 왜 JWT는 직접 넣고 쿠키는 자동일까?**

* **쿠키의 자동성:** 쿠키는 웹의 초창기부터 브라우저의 표준 규약으로 만들어졌습니다. 그래서 브라우저는 "A 사이트에서 준 쿠키는 A 사이트에 요청 보낼 때 무조건 다시 보낸다"라는 약속이 철저히 지켜집니다.
* **JWT의 수동성:** JWT는 브라우저 표준이라기보다 **토큰 기반 인증 방식**의 데이터 포맷입니다. 보통 보안상의 이유(CSRF 공격 방지 등)로 브라우저 자동 전송에 의존하지 않고, 자바스크립트 코드에서 `Header`에 직접 실어 보내는 방식을 선호합니다.

---

### **4. "쿠키에 JWT를 담을 수도 있다?"**
이 부분이 헷갈릴 수 있는데, **JWT라는 내용물을 쿠키라는 봉투에 담아** 보낼 수도 있습니다.

1.  서버가 `Set-Cookie: access_token=eyJ...` 라고 보냅니다.
2.  그러면 브라우저는 이 JWT가 담긴 쿠키를 다음 요청부터 **자동으로** 보냅니다.
3.  이 경우, 개발자는 헤더에 JWT를 넣는 코드를 따로 짤 필요가 없게 됩니다.

**요약하자면:**
쿠키는 **`Cookie`라는 이름의 헤더**에 브라우저가 알아서 쏙 집어넣어 보낸다고 이해하시면 됩니다!
