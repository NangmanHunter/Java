운영 중인 서비스에서 인증 방식을 세션(Session)에서 JWT(JSON Web Token)로 전환하는 것은 비행기 운행 중에 엔진을 교체하는 것만큼이나 까다로운 작업입니다. 말씀하신 대로 서비스를 멈추지 않고 진행하기 위해 보통 **'하이브리드(Hybrid) 운영 기간'**을 거치는 것이 정석입니다.

전환 과정에서 주로 사용하는 전략들을 정리해 드립니다.

---

### **1. 과도기적 공존 (Dual Authentication Strategy)**
가장 흔히 사용하는 방식으로, 서버가 **두 가지 인증 수단을 모두 수용**하도록 설정합니다.

* **동작 원리:** 서버의 인터셉터(Interceptor)나 필터(Filter) 계층에서 요청을 검사합니다.
    1. 먼저 HTTP 헤더에 `Authorization: Bearer <JWT>`가 있는지 확인합니다.
    2. JWT가 없다면 기존 방식대로 쿠키(Cookie)에 세션 ID가 있는지 확인합니다.
    3. 둘 중 하나라도 유효하면 인증된 사용자로 처리합니다.
* **장점:** 구형 클라이언트(업데이트 안 된 앱 등)와 신형 클라이언트를 동시에 지원할 수 있습니다.

---

### **2. 단계적 전환 (Step-by-Step Transition)**
한 번에 전체를 바꾸는 것이 아니라, 특정 API 그룹부터 차례대로 적용합니다.

1.  **신규 기능 우선 적용:** 새로 만드는 기능이나 특정 도메인(예: 마이페이지 외의 일반 조회 API)에만 JWT를 먼저 적용합니다.
2.  **로그인 시 두 가지 발급:** 사용자가 로그인할 때 서버는 세션도 생성하고, 동시에 JWT도 발급하여 응답에 실어 보냅니다.
3.  **클라이언트의 선택적 사용:** 프론트엔드에서는 점진적으로 세션 쿠키 대신 JWT를 헤더에 담아 보내도록 코드를 수정해 나갑니다.

---

### **3. API 게이트웨이를 활용한 추상화**
마이크로서비스 아키텍처(MSA) 환경이라면 **API Gateway**가 큰 역할을 합니다.

* **변환 계층:** 클라이언트는 무조건 JWT로 요청을 보냅니다. 하지만 아직 세션 기반으로 돌아가는 레거시 내부 서버들을 위해, API 게이트웨이가 중간에서 JWT를 확인한 뒤 **세션 정보로 변환하여 내부 서버에 전달**해 줍니다.
* 이 방식을 쓰면 내부 로직을 한 번에 다 고치지 않아도 외부 인터페이스를 먼저 JWT로 통일할 수 있습니다.

---

### **4. 세션 데이터를 JWT로 마이그레이션**
기존 세션에 담겨 있던 유저의 권한 정보나 상태 값을 JWT의 **Claim** 영역으로 옮기는 과정이 필요합니다.

* **주의점:** 세션은 서버 메모리에 데이터를 저장하므로 크기 제한이 거의 없지만, JWT는 매 요청마다 전송되므로 너무 많은 정보를 담으면 네트워크 부하가 커집니다. 따라서 핵심 식별자(User ID, Role) 위주로 구성하는 설계 변경이 동반됩니다.

---

### **결론적으로 어떤 모습이었을까?**
실제 운영 환경에서는 아마 다음과 같은 순서를 밟았을 가능성이 높습니다.

1.  **1단계:** 기존 세션 로직은 그대로 두고, JWT 검증 로직을 추가하여 **서버가 두 방식을 모두 이해하게 만듦.**
2.  **2단계:** 로그인 API 수정. 로그인 성공 시 세션 생성과 함께 **JWT 토큰을 추가로 반환.**
3.  **3단계:** 클라이언트 코드를 순차적으로 업데이트하여 API 호출 시 JWT를 사용하도록 변경.
4.  **4단계:** 일정 기간(모든 사용자의 세션이 만료될 충분한 시간)이 지난 후, 서버에서 **세션 관리 로직을 제거.**

이런 방식은 사용자 입장에서는 로그아웃되는 불편함 없이 자연스럽게 인증 체계가 바뀌는 경험을 제공합니다. 결국 '2개를 다 띄워 놓고 진행'하는 것이 안정적인 전환의 핵심이라고 볼 수 있습니다.
