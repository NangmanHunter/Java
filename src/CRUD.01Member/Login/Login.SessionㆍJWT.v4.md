##
- Session.메모리위에




##
세션(Session)과 JWT(JSON Web Token)의 가장 큰 차이점을 아주 명확하게 짚어드릴게요. 특히 질문하신 **'세션이 서버에서 어떻게 유지되는지'**에 대한 궁금증부터 해결해 봅시다.

---

## 1. 세션(Session): "내가 너 기억하고 있을게"

세션 방식에서 서버가 정보를 '쥐고 있다'는 것은 **서버의 메모리(RAM)나 별도의 데이터베이스에 저장**하고 있다는 뜻이 맞습니다.

* **저장 방식:** * 기본적으로는 서버의 **RAM(메모리)** 위에 객체 형태로 들고 있습니다. 
    * 하지만 서버가 꺼지면 로그인이 다 풀려버리겠죠? 그래서 실제 서비스에서는 **Redis** 같은 아주 빠른 메모리 기반 데이터베이스나 일반 DB에 저장하기도 합니다.
* **동작 원리:**
    1.  사용자가 로그인하면 서버는 **세션 ID**라는 랜덤한 열쇠를 하나 만듭니다.
    2.  서버는 이 열쇠를 저장소(RAM 등)에 보관하고, 사용자(브라우저)에게는 이 열쇠만 보내줍니다.
    3.  브라우저는 이 열쇠(쿠키)를 들고 있다가 매번 서버에 보여줍니다.
    4.  서버는 "오, 내 메모리에 있는 이 열쇠 주인 맞네!" 하고 통과시켜 줍니다.



---

## 2. JWT (Token): "네 정보는 네가 들고 다녀"

JWT는 서버가 정보를 들고 있지 않습니다. 대신 사용자에게 **증명서**를 발급해 주는 방식입니다.

* **저장 방식:** 서버는 아무것도 저장하지 않습니다. (Stateless)
* **동작 원리:**
    1.  사용자가 로그인하면 서버는 사용자의 정보(ID, 권한 등)를 담은 **토큰**을 만듭니다.
    2.  이 토큰이 조작되지 않았음을 증명하기 위해 서버만 아는 비밀키로 **디지털 서명**을 해서 사용자에게 줍니다.
    3.  사용자는 이 토큰을 보관하다가 서버에 보여줍니다.
    4.  서버는 자기 메모리를 뒤지는 게 아니라, **토큰의 서명이 유효한지만 계산**해서 확인합니다. "내가 만든 도장이 맞네? 통과!" 하는 식이죠.



---

## 3. 한눈에 비교하기

| 구분 | 세션 (Session) | JWT (Token) |
| :--- | :--- | :--- |
| **저장소** | **서버 메모리/DB** | **클라이언트(브라우저)** |
| **상태 유지** | Stateful (서버가 상태를 기억함) | Stateless (서버는 기억력 0) |
| **서버 부담** | 사용자가 많아지면 메모리 사용량 증가 | 메모리 부담은 없으나 검증 계산 필요 |
| **장점** | 보안성이 높음 (강제 로그아웃 가능) | 서버 확장성(Scale-out)에 매우 유리 |
| **단점** | 서버가 여러 대면 세션 공유 설정이 복잡함 | 한 번 발급된 토큰은 뺏기가 어려움 |

---

### 요약하자면!
* **세션**은 "서버가 장부를 들고 있는 방식"입니다. 질문하신 대로 **RAM(메모리)**에 주로 두며, 필요에 따라 DB에 넣기도 합니다.
* **JWT**는 "서버는 장부가 없고, 사용자가 위조 불가능한 신분증을 들고 다니는 방식"입니다.

서버가 여러 대로 늘어나는 현대적인 클라우드 환경에서는 세션을 공유하는 게 까다롭기 때문에 JWT를 더 많이 선호하는 편이에요.
