결론부터 말씀드리면, **"로직 자체를 다 갈아엎을 정도는 아니지만, 통신 방식의 패러다임을 바꾸는 작업"**이라 에너지가 어느 정도 들긴 합니다. 하지만 지금 세션 방식으로 기반을 잘 닦아두면 나중에 전환하는 게 불가능한 작업은 아닙니다.

전환 시 구체적으로 어떤 부분에 에너지가 들어가는지 핵심 요소를 짚어드릴게요.

---

## 1. 서버 측: "상태 저장"에서 "상태 없음(Stateless)"으로
현재 세션 방식은 서버의 메모리나 DB(지금 쓰시는 H2 등)에 세션 정보를 저장하고 관리합니다.
* **세션:** 서버가 "아, 이 사람은 로그인한 1번 사용자구나"라고 기억해줌.
* **JWT:** 서버는 아무것도 기억하지 않음. 사용자가 들고 온 '토큰'이 진짜인지 검증만 함.
* **작업량:** 세션 관리 로직(Session Manager 등)을 걷어내고, JWT를 생성(Issue)하고 검증(Verify)하는 **Filter 또는 Interceptor**를 새로 구현해야 합니다.

## 2. 보안 설정 (Spring Security)
이미 Spring Security를 사용 중이라면 설정 클래스를 수정해야 합니다.
* **세션:** `http.sessionManagement().sessionCreationPolicy(SessionCreationPolicy.IF_REQUIRED)`
* **JWT:** `http.sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)`
* 이 설정 하나로 서버의 세션 생성 메커니즘을 꺼버려야 하므로, 기존에 세션에 의존하던 모든 보안 로직을 검토해야 합니다.

## 3. 프론트엔드 통신 방식의 변화 (가장 큰 변화)
사실 서버보다 프론트엔드(또는 API 호출부)에서 수정할 게 더 많을 수 있습니다.
* **세션:** 브라우저가 알아서 쿠키를 주고받으므로 신경 쓸 게 적음.
* **JWT:** 로그인이 성공하면 서버가 준 토큰을 **Local Storage나 별도 변수**에 저장해야 하고, 이후 모든 API 요청 헤더에 `Authorization: Bearer <TOKEN>`을 직접 실어 보내는 코드를 추가해야 합니다.

## 4. 로그아웃 및 만료 처리
이 부분이 가장 골치 아픈 '에너지 소비' 구간입니다.
* JWT는 한 번 발급하면 서버가 강제로 만료시키기 어렵습니다. (서버가 상태를 안 들고 있으니까요.)
* 이를 해결하려면 **Refresh Token** 개념을 도입해야 하고, 이를 저장할 별도의 저장소(Redis 등)가 결국 다시 필요해지는 경우가 많습니다.

---

### 💡 지금 세션으로 개발할 때의 팁 (나중을 위해)
나중에 JWT로 바꿀 계획이 있다면, 지금 코드를 짤 때 **"비즈니스 로직"과 "인증 로직"을 철저히 분리**해두세요.

1.  **Controller에서 세션을 직접 뒤지지 않기:** `HttpSession`을 컨트롤러 인자로 직접 받아서 쓰지 마세요. 대신 `@AuthenticationPrincipal` 같은 어노테이션을 사용하여 "현재 로그인한 사용자 정보"를 추상화해서 받아쓰면, 나중에 인증 방식이 세션에서 JWT로 바뀌어도 비즈니스 로직(Service, Repository)은 수정할 필요가 없습니다.
2.  **DTO 기반 통신:** 데이터를 주고받을 때 세션 객체 자체를 넘기지 말고, 필요한 정보만 담은 DTO를 사용하세요.

### 요약하자면
현재 공부하시면서 기반을 잡는 단계라면 **세션 방식으로 끝까지 구현해 보시는 것을 추천**합니다. 인증의 흐름을 정확히 이해하고 나면, 나중에 JWT로 바꾸는 것은 "인증 수단만 교체하는 작업"이 되기 때문에 아주 좋은 공부가 될 거예요.

에너지는 좀 들겠지만, 구조만 잘 잡아두면 **'재건축'이 아니라 '인테리어 공사' 수준**으로 끝낼 수 있습니다.

혹시 지금 프로젝트에서 사용자 정보를 세션에 어떻게 담고 계신가요? (예: 직접 세션에 setAttribute를 하시는지, 아니면 Spring Security의 인증 객체를 쓰시는지 등) 그 방식에 따라 나중에 바꿀 때의 난이도가 결정됩니다.
