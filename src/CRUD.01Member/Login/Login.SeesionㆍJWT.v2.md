대형 서비스들은 단순히 세션이나 JWT 중 하나만 선택하기보다, **각 방식의 장점을 결합하거나 서비스의 아키텍처(MSA 등)에 최적화된 하이브리드 방식**을 주로 사용합니다.

구체적으로 어떤 전략을 쓰는지 핵심 3가지로 정리해 드릴게요.

---

## 1. 하이브리드 방식 (Access Token + Refresh Token)
JWT의 장점(무상태성)을 살리되 단점(제어 불가능)을 보완하기 위해 가장 많이 쓰는 방식입니다.

* **Access Token (JWT):** 유효기간을 매우 짧게(예: 30분~1시간) 설정하여 클라이언트가 API를 호출할 때 사용합니다. 탈취되어도 금방 만료되므로 위험을 최소화합니다.
* **Refresh Token (Database/Redis 저장):** 유효기간을 길게(예: 2주) 설정하고, 서버측 DB나 **Redis**에 저장합니다. Access Token이 만료되면 이 토큰을 대조해 새로운 Access Token을 발급해 줍니다.
* **특징:** 평소에는 JWT처럼 서버 부하 없이 검증하다가, 사용자를 강제 로그아웃시켜야 할 때만 서버 저장소에서 Refresh Token을 삭제하여 제어권을 가집니다.



---

## 2. 중앙 집중형 세션 관리 (Redis 사용)
대형 서비스는 서버가 수백 대입니다. 각 서버 메모리에 세션을 저장하면 로그인이 유지되지 않기 때문에, 별도의 **세션 서버(In-Memory DB)**를 둡니다.

* **공유 저장소:** 모든 서버가 사용자 세션 정보를 **Redis**나 **Memcached** 같은 초고속 메모리 DB 한곳에서 조회합니다.
* **장점:** 세션 방식의 장점(즉각적인 제어)을 유지하면서도 서버 확장성 문제를 해결합니다. 전통적인 금융권이나 대형 커머스에서 보안을 위해 여전히 선호하는 방식입니다.



---

## 3. 서비스 성격에 따른 분리 (Gateway 인증)
대형 서비스는 내부적으로 수많은 마이크로서비스(MSA)로 쪼개져 있습니다.

* **API Gateway에서 인증:** 사용자가 요청을 보내면 입구인 'Gateway'에서만 인증을 수행합니다. 
* **내부 전달:** Gateway가 인증을 마치면 내부 서비스들(주문, 장바구니, 회원정보 등)에는 인증된 사용자 정보를 가벼운 토큰에 담아 넘겨줍니다. 
* **장점:** 내부 서비스마다 일일이 로그인을 확인할 필요가 없어 효율적입니다.

## 💡 요약: 어떤 걸 선택할까?

| 서비스 규모 | 추천 방식 | 이유 |
| :--- | :--- | :--- |
| **중소규모 웹** | **세션(Session)** | 구현이 가장 빠르고 보안 관리가 직관적임. |
| **모바일 앱 병행** | **JWT (Basic)** | 쿠키 관리가 까다로운 모바일 환경에 적합. |
| **대규모/글로벌** | **JWT + Redis (Refresh)** | 서버 부하 분산과 사용자 제어라는 두 마리 토끼를 잡음. |

### 프로젝트를 위한 조언
지금 만드는 게시판 프로젝트가 **"나중에 포트폴리오로 확장하고 싶다"**면 **1번(JWT + Refresh Token)** 방식을 구현해 보시는 걸 추천합니다. 

현업에서 가장 많이 요구하는 역량 중 하나가 **"Token 만료 시나리오를 어떻게 처리했는가"**와 **"Redis를 어떻게 활용했는가"**이기 때문입니다.

직접 코드를 짤 때 어떤 방식이 더 끌리시나요? 결정하시면 Spring Security와 MariaDB를 연동한 구체적인 구현 설계를 도와드릴 수 있습니다.
