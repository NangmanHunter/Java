아주 정확한 기억입니다! 예전(특히 JSP/Servlet 시절이나 초기 스프링)에는 로그인 상태를 유지하고 화면에 띄우기 위해 크게 두 가지 방식을 썼습니다. 물리적으로 "계속 요청을 보냈다"기보다는 **"요청마다 서버가 누군지 확인했다"**는 표현이 더 정확할 것 같네요.

과거에 어떤 흐름으로 진행됐는지, 그리고 지금은 어떻게 바뀌었는지 논리적으로 짚어드릴게요.

---

### 1. 예전 방식: HttpSession (서버 세션)
가장 전통적인 방식입니다. 서버(Java)가 메모리(RAM)의 일정 구역을 떼어서 로그인 정보를 들고 있는 방식이죠.

* **동작 원리:** 1. 사용자가 로그인하면 서버 메모리에 세션을 만들고 유저 정보를 저장합니다.
    2. 서버는 브라우저에게 `JSESSIONID`라는 딱지를 쿠키로 구워줍니다.
    3. 이후 브라우저가 **어떤 요청을 보낼 때마다** 이 딱지를 같이 보냅니다.
    4. 서버는 요청이 들어올 때마다 "이 딱지가 내 메모리에 있는 걔 맞나?" 하고 **계속 확인**해서 로그인 상태를 유지했습니다.
* **단점:** 서버 메모리를 쓰기 때문에 사용자가 많아지면 서버가 힘들어하고(물리적 한계), 서버를 껐다 켜면 로그인이 다 풀려버렸죠.



---

### 2. 페이지 이동 시마다 쿼리 (Parameter)
정말 오래된 방식이나 특정 보안 환경에서는 페이지를 이동할 때마다 URL 뒤에 `?userId=...` 같은 정보를 붙여서 **계속 넘겨주며** 서버에 "나 얘야!"라고 말해줘야 했던 시절도 있었습니다. (매우 번거롭고 보안에 취약했죠.)

---

### 3. 지금 우리가 하는 방식: SPA + JWT/Cookie
지금 Nuxt 3와 Spring Boot로 하고 계신 방식은 훨씬 세련된 방식입니다.

* **프론트엔드 (Nuxt):** 페이지를 이동한다고 해서 서버에 매번 "화면 전체"를 달라고 하지 않습니다. 대신 **메모리(Pinia나 useState)**에 유저 정보를 한 번 저장해두면, 새로고침하기 전까지는 서버에 묻지 않고도 내 이름을 화면에 띄울 수 있습니다.
* **백엔드 (Spring):** 서버는 이제 세션 메모리에 정보를 저장하지 않는 **Stateless(무상태)** 방식을 선호합니다. 대신 **JWT(토큰)**를 발행하거나 보안 쿠키를 써서, 요청이 올 때만 그 토큰이 유효한지 "수학적으로 검증"만 합니다.

---

### 💡 왜 "계속 요청을 보낸다"고 느끼셨을까요?

아마 **"인증 필터(Filter/Interceptor)"** 때문일 겁니다. 
로그인 상태를 띄우기 위해 우리가 명시적으로 코드를 짜지는 않았지만, 내부적으로는 **모든 요청이 서버로 갈 때마다** 서버의 Filter가 "이 녀석 로그인 한 놈인가?"를 **계속 체크**하고 있었거든요. 

사용자님 입장에서는 "내가 누군지 계속 서버에 증명해야 하네?"라고 느끼실 수 있는 대목입니다.

### 🛠️ 지금 적용할 "Nangman" 포인트

지금 작성하신 사이드바 코드(`const user = ref(...)`)를 로그인 상태와 연동할 때, 예전처럼 매번 서버에 "내 이름 뭐야?"라고 물어볼 필요가 없습니다.

1.  **최초 로그인 시:** 서버가 `name`, `email` 등을 줍니다.
2.  **저장:** 이걸 프론트의 전역 상태(`useState`)에 딱 한 번 저장합니다.
3.  **사용:** 사이드바는 그 저장된 값을 바라보기만 하면 됩니다. (서버 요청 0회!)

결국 예전에는 **서버가 주도권**을 쥐고 매번 확인했다면, 지금은 **프론트가 정보를 쥐고** 필요할 때만 서버와 통신하는 구조로 진화한 셈입니다. 수학의 독립변수가 서버에서 클라이언트로 옮겨온 느낌이랄까요? :)

혹시 예전에 Java로 개발하실 때 세션(`session.setAttribute`)을 쓰셨던 기억이 나시나요? 그게 바로 그 방식입니다!
