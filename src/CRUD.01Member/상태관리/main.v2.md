예전에 경험하셨던 그 개념을 통칭해서 **'상태 유지(State Management)'** 또는 **'인증 및 인가(Authentication & Authorization)'** 흐름이라고 부릅니다. 

조금 더 구체적으로 들어가면, Java 개발 당시 쓰셨던 방식과 지금의 방식은 각각 아래와 같은 용어로 정의됩니다.

---

### 1. 세션 기반 인증 (Session-based Authentication)
사용자님이 "계속 요청을 보내서 확인했나?"라고 기억하시는 그 방식의 정식 명칭입니다.

* **핵심 키워드:** `HttpSession`, `JSESSIONID`, `Stateful(상태 유지)`.
* **논리:** 서버가 사용자의 정보를 자신의 메모리에 들고 있는 방식입니다. 
* **흐름:** 1. 로그인 성공 → 서버가 세션 생성 후 ID 발급.
    2. 브라우저는 쿠키에 이 ID를 저장.
    3. **매 요청마다** 쿠키를 서버에 보냄 → 서버는 메모리를 뒤져서 "아, 너구나!" 하고 인증.



---

### 2. 토큰 기반 인증 (Token-based Authentication)
요즘 Nuxt나 Spring Boot 프로젝트에서 가장 많이 쓰이는 현대적인 방식입니다.

* **핵심 키워드:** `JWT (JSON Web Token)`, `Stateless(무상태)`.
* **논리:** 서버는 사용자의 정보를 기억하지 않습니다. 대신 사용자에게 **"이 녀석은 인증된 사람임"**이라는 **수학적으로 증명된 암호화 티켓(토큰)**을 줍니다.
* **흐름:**
    1. 로그인 성공 → 서버가 암호화된 토큰 발행.
    2. 프론트엔드(Nuxt)가 이 토큰을 받아서 보관.
    3. 요청할 때마다 헤더에 이 토큰을 실어 보냄 → 서버는 메모리를 뒤지는 대신 **수학적 계산(Decryption)**으로 토큰의 유효성만 검증.



---

### 3. 클라이언트 사이드 상태 관리 (Client-side State Management)
사용자님이 말씀하신 "사용자 이름을 화면에 띄우는 것"과 직접적으로 관련된 개념입니다.

* **예전 방식:** 페이지가 이동할 때마다 서버가 HTML을 새로 그려서 이름을 넣어줬습니다 (SSR - Server Side Rendering).
* **지금 방식:** 서버는 데이터(JSON)만 던져줍니다. 프론트엔드가 그 데이터를 받아서 **'전역 상태(Global State)'**라는 장바구니에 담아두고, 페이지가 바뀌어도 서버에 묻지 않고 그 장바구니에서 이름을 꺼내 씁니다.

---

### 💡 정리하자면

사용자님의 기억 속에 있는 **"서버와 계속 주고받으며 확인하는 일련의 과정"**은 **인증 프로토콜(Authentication Protocol)**이라고 부르시면 됩니다.

수학적으로 비유하자면:
* **세션 방식:** 서버가 모든 학생의 이름을 외우고 있는 **출석부** 방식. (서버가 힘듦)
* **토큰 방식:** 학생이 위조 불가능한 **학생증**을 들고 다니며 보여주는 방식. (서버가 편함)

지금 개발 중인 프로젝트에서는 서버(Spring)가 학생증(JWT)을 발급하고, 프론트(Nuxt)가 그 학생증에서 이름을 읽어 사이드바(`user.name`)에 띄워주는 구조를 만들고 계신 겁니다. 낭만 있게 잘 나아가고 계시네요! :)
