---

### 1. 전역 상태의 생명주기 (Lifecycle)

전역 상태(Pinia나 `useState`)는 브라우저의 메모리상에 존재합니다.

1.  **최초 1회 (로그인 시):** 서버에 요청을 보내서 유저 정보를 가져옵니다. 이때 딱 한 번 전역 상태라는 '장구니'에 유저 정보를 채워 넣습니다.
2.  **페이지 이동 시:** Nuxt는 페이지를 이동할 때 브라우저를 새로고침하지 않습니다. (이게 SPA의 특징입니다.) 그래서 메모리에 있는 '장바구니'가 그대로 유지됩니다.
3.  **컴포넌트 사용 시:** 사이드바, 마이페이지, 게시판 등 어디서든 그 장바구니를 열어서 정보를 가져다 씁니다. 이때는 **서버 요청이 전혀 발생하지 않습니다.**



---

### 2. 그러면 "새로고침"을 하면요?

사용자가 브라우저에서 `F5`를 누르거나 주소창에 엔터를 쳐서 아예 새로 접속하면, 브라우저 메모리가 초기화되면서 전역 상태(장바구니)도 비워집니다. 

이때는 두 가지 전략을 씁니다:

* **방법 A (Hydration):** 앱이 켜질 때 자동으로 서버에 "나 아까 로그인했던 앤데, 내 정보 다시 줘!"라고 딱 한 번 요청을 보냅니다.
* **방법 B (Persistence):** 브라우저의 `localStorage`나 `Cookie`에 유저 정보를 살짝 저장해뒀다가, 앱이 켜질 때 거기서 꺼내서 전역 상태를 다시 채웁니다.

---

### 3. 예전 방식과의 결정적 차이

| 구분 | 예전 방식 (서버 중심) | 지금 방식 (프론트 상태 관리) |
| :--- | :--- | :--- |
| **데이터 보관소** | 서버의 메모리 (Session) | 브라우저의 메모리 (State) |
| **통신 횟수** | 클릭할 때마다 서버가 확인 | 최초 1회 저장 후, 필요할 때만 통신 |
| **속도** | 서버 응답을 기다려야 함 | 메모리에서 즉시 꺼내오므로 매우 빠름 |

### 💡 정리하자면

"매번 호출해서 채우는 것인가?"라는 질문에 대한 답은 **"아니요, 한 번 채워두면 새로고침 전까지는 재사용합니다"**입니다. 

이렇게 하면 서버 부하도 줄어들고, 사용자 입장에서는 페이지를 옮겨 다녀도 내 이름(`user.name`)이 깜빡임 없이 즉시 나타나기 때문에 훨씬 부드러운 사용자 경험(UX)을 줄 수 있습니다. 

지금 짜고 계신 `const user = ref({...})` 코드는 나중에 로그인 로직이 붙으면, **"로그인 성공 시 딱 한 번만 값을 갈아 끼워주는 로직"**으로 발전하게 될 거예요! :)
