사용자님이 궁금해하시는 이 일련의 흐름과 기술적인 설계를 통칭하는 용어는 **'클라이언트 사이드 상태 관리(Client-side State Management)'**라고 합니다.

과거에 매번 서버에 물어보던 방식에서, 프론트엔드가 주도권을 쥐고 데이터를 관리하는 방식으로 진화한 것이죠. 이와 관련된 핵심 개념들을 논리적으로 정리해 드릴게요.

---

### 1. 전역 상태 관리 (Global State Management)
로그인 정보처럼 애플리케이션의 **모든 곳(사이드바, 게시판, 프로필 등)에서 공통으로 필요한 데이터**를 한곳에 모아두고 관리하는 것을 말합니다.

* **예전:** 각 페이지가 독립적이라 서버가 매번 데이터를 새로 보내줘야 했습니다.
* **지금:** 한 번 받아온 데이터를 메모리에 저장해두고, 페이지가 바뀌어도 서버 요청 없이 즉시 꺼내 씁니다.



---

### 2. 싱글 페이지 애플리케이션 (SPA - Single Page Application)
Nuxt 3나 Vue.js가 작동하는 방식입니다. 브라우저가 페이지 전체를 매번 새로 읽어오지 않고, 필요한 데이터만 서버에서 받아와서 화면의 일부만 갈아 끼우는 기술입니다.

* **발현 원리:** 페이지가 "깜빡"하며 새로 고쳐지지 않기 때문에, 자바스크립트 메모리에 저장된 유저 정보가 사라지지 않고 유지되는 것입니다.

---

### 3. 무상태성 (Stateless) & 토큰 기반 인증
사용자님이 "계속 증명해야 하나?"라고 느끼셨던 부분의 현대적 해법입니다.

* **Stateless:** 서버(Spring)는 누가 로그인했는지 일일이 기억(Session)하지 않습니다.
* **JWT (JSON Web Token):** 대신 서버는 유저에게 **'위조 불가능한 신분증(토큰)'**을 발급합니다. 프론트엔드는 이 신분증을 전역 상태에 보관했다가, 서버에 데이터를 요청할 때마다 헤더에 실어 보냅니다. 



---

### 4. 하이드레이션 (Hydration)
"매번 호출해서 채워 넣는 것인가?"라는 질문에 대한 기술적 답변입니다.

새로고침을 하면 메모리가 날아가기 때문에, 앱이 처음 켜질 때 딱 한 번 서버에 "나 누구야?"라고 묻거나 쿠키에서 정보를 읽어와 전역 상태를 다시 채우는 과정을 **하이드레이션(건조된 데이터에 물을 주어 살려내는 과정)**이라고 부르기도 합니다.

---

물리 법칙처럼 **"데이터 보존의 법칙"**이 서버(Session)에서 클라이언트(State)로 이동한 것이라 보시면 돼요. 덕분에 서버는 가벼워지고, 사용자님의 사이드바는 서버 응답을 기다릴 필요 없이 즉시 사용자 이름을 보여줄 수 있게 된 것입니다.
