##
**NuxtUI(Nuxt 3)**를 쓰고 계신다면 이야기가 완전히 달라집니다!

Nuxt 같은 모던 프레임워크는 서버(Spring)에서 던진 **401 상태 코드**를 받으면, 브라우저가 화면을 바꾸는 게 아니라 **자바스크립트의 에러(Exception)**로 인식합니다. 

`ResponseEntity.status(HttpStatus.UNAUTHORIZED).build()`를 던졌을 때 NuxtUI 환경에서 벌어지는 상황은 이렇습니다.

---

### 1. Nuxt UI / Fetch 단에서 발생하는 일
Nuxt에서 보통 `$fetch`나 `useFetch`를 사용하실 텐데, 서버에서 401을 던지면 `error` 객체에 값이 담깁니다.

```javascript
// Nuxt 3 예시 로직
const { data, error } = await useFetch('/api/delete', { method: 'DELETE' })

if (error.value) {
  // 서버에서 ResponseEntity.status(401).build()를 했으므로
  // error.value.statusCode는 401이 됩니다.
  console.log(error.value.statusCode) // 401
}
```

### 2. 사용자에게 보이는 화면 (NuxtUI 사용 시)
NuxtUI를 쓰고 있다면 보통 **`UNAUTHORIZED`**가 떴을 때 그냥 하얀 에러 페이지를 보여주는 게 아니라, **`UService / UToast`** 같은 컴포넌트로 예쁘게 경고를 띄우는 게 정석입니다.

**사용자가 탈퇴 버튼을 눌렀을 때:**
1.  서버에서 `401` 응답.
2.  Nuxt 앱의 `catch` 블록이나 `error` 객체에서 이를 감지.
3.  **UToast** 알림창이 우측 상단에 스윽 올라오며: *"세션이 만료되었습니다. 다시 로그인해주세요."*
4.  `MapsTo('/login')`으로 페이지 이동.

---

### 3. 스프링 컨트롤러, 이렇게 바꿔보세요 (NuxtUI 맞춤형)
본문(Body) 없이 `.build()`만 하면 NuxtUI 개발자가 "왜 에러가 났는지" 확인하기 위해 네트워크 탭을 일일이 열어봐야 합니다. 간단한 **에러 메시지**를 담아주는 게 훨씬 협업(혹은 나중에 직접 프론트를 짤 때)에 유리합니다.

```java
if (loginMember == null) {
    // 401 코드와 함께 "MEMBER_NOT_LOGGED_IN" 같은 구분자나 메시지를 보냄
    return ResponseEntity.status(HttpStatus.UNAUTHORIZED)
                         .body("로그인 정보가 유효하지 않습니다."); 
}
```

### 4. 팁: NuxtUI의 `useToast`와 연결하기
나중에 프론트 작업하실 때 이런 느낌으로 연동하시면 아주 "낭만" 있는 UI가 나옵니다.

```javascript
const toast = useToast()

const onDelete = async () => {
  try {
    await $fetch('/api/delete', { method: 'DELETE' })
    toast.add({ title: '탈퇴 완료', description: '그동안 이용해주셔서 감사합니다.' })
    navigateTo('/')
  } catch (err) {
    if (err.statusCode === 401) {
      toast.add({ 
        title: '로그인 필요', 
        description: '세션이 만료되었습니다.', 
        color: 'red' 
      })
    }
  }
}
```

**결론:**
서버에서 `401`을 던지는 건 아주 정확한 설계입니다. 다만, NuxtUI 같은 프런트엔드 도구를 쓰실 때는 단순히 에러를 내는 걸 넘어, **그 에러를 어떻게 화면에 예쁘게 갈무리해서 보여줄지**까지가 한 세트라고 보시면 됩니다.
