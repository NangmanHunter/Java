
## required
- [ ] `required = true`
- [x] `required = false`


##
그 부분은 **사용자 경험(UX)보다는 시스템의 안정성과 예외 처리** 때문에 그렇습니다.

`required = true`(기본값)로 설정하면, 세션에 해당 값이 없을 경우 스프링이 컨트롤러에 진입하기도 전에 `HttpSessionRequiredException`이라는 에러를 던지며 바로 **400 Bad Request** 페이지를 띄워버립니다.

### 왜 `required = false`를 권장할까요?

**1. 화이트 에러 페이지 방지**
사용자가 로그인이 풀린 줄 모르고 탈퇴 버튼을 눌렀을 때, 갑자기 브라우저에 시스템 에러 화면이 뜨는 것보다 "로그인이 필요합니다" 같은 깔끔한 메시지를 보여주거나 로그인 페이지로 리다이렉트 시키는 것이 훨씬 매끄럽습니다.

**2. 세밀한 로직 제어**
컨트롤러 내부에서 `if (loginMember == null)`로 체크하면, 개발자가 원하는 방식대로 후속 처리를 할 수 있습니다. 
* 예: "이미 세션이 만료되어 자동으로 로그아웃되었습니다"라는 알림을 주거나, 특정 로그를 남기는 등 유연한 대응이 가능하죠.

---

### 사실 더 권장되는 방식: 인터셉터(Interceptor)
질문하신 것처럼 "어차피 로그인 안 한 사람은 탈퇴 로직 근처에도 오면 안 되는 것 아니냐"는 시각이 정확합니다. 그래서 실무에서는 이 체크 로직을 컨트롤러마다 넣지 않고 **인터셉터(Interceptor)**나 **필터(Filter)**에서 미리 걸러냅니다.



```java
// 이런 식으로 인터셉터에서 미리 거르면 컨트롤러는 깔끔해집니다.
if (session == null || session.getAttribute("loginMember") == null) {
    response.sendRedirect("/login");
    return false;
}
```

이렇게 하면 컨트롤러에서는 `required = true`를 써도 안전합니다. 인터셉터라는 '문지기'가 이미 로그인 안 한 사람을 다 집으로 돌려보냈을 테니까요.

### 요약하자면
* **`required = false`**: 컨트롤러 안에서 "우아하게" 에러를 처리하고 싶을 때 사용합니다.
* **`required = true`**: 인터셉터 등으로 이미 로그인 보장이 된 상태이거나, 에러 페이지가 떠도 상관없을 때 사용합니다.

논리적인 흐름을 중요하게 생각하시는 스타일이시라, 아마 나중에는 인터셉터로 모든 권한을 관리하는 방식이 훨씬 편하게 느껴지실 겁니다.

이제 회원 탈퇴 시 세션을 날리는 로직(`session.invalidate()`)도 같이 추가해 보실 건가요?
