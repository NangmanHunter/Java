아, 그 당혹스러움 이해합니다. 코드를 보면 분명히 `@Where(clause = "is_deleted = false")`를 걸어두셨는데, `findAll()`을 했을 때 삭제된 데이터까지 다 나온다면 정말 귀신이 곡해할 노릇이죠.

이 현상이 발생하는 이유는 크게 두 가지 가능성이 있습니다. 하나씩 짚어보시죠.

---

### 1. 하이버네이트 6.3 버전 이상의 변경점 (가장 유력)
사용 중인 스프링 부트 버전이 최신(3.2.x 이상)이라면, 하이버네이트 6.3 이상이 탑재되어 있을 겁니다. 이때부터는 **`@Where` 어노테이션의 사용법이 바뀌었습니다.**

* **기존:** `@Where(clause = "is_deleted = false")`
* **현재:** **`@Where(clause = "is_deleted = false")`는 Deprecated(사용 중단 권장)** 되었고, 구문 해석 방식이 달라졌을 수 있습니다.

**해결책:** 최신 방식인 **`@SQLRestriction`**으로 바꿔보세요.
```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@SQLDelete(sql = "UPDATE member SET is_deleted = true WHERE member_id = ?")
// @Where 대신 이 어노테이션을 사용해 보세요
@SQLRestriction("is_deleted = false") 
public class Member { ... }
```

### 2. 영속성 컨텍스트(1차 캐시)의 '장난'
혹시 테스트 코드나 로직에서 **방금 `delete`를 수행한 직후에 같은 트랜잭션 안에서 `findAll`을 호출**하셨나요?



* **원인:** JPA의 `delete`를 호출해도, 해당 트랜잭션이 끝나기 전까지는 DB에만 `UPDATE` 쿼리가 예약될 뿐, **메모리(영속성 컨텍스트)에 있는 객체는 여전히 `isDeleted = false` 상태로 살아있을 수 있습니다.**
* `findAll()`을 호출할 때 DB에서 긁어오더라도, 동일한 ID의 객체가 이미 메모리에 있으면 JPA는 DB 데이터 대신 메모리에 있는 (아직 삭제 안 된 것처럼 보이는) 객체를 반환합니다.

**해결책:** 조회를 하기 전에 `em.flush()`와 `em.clear()`를 호출해서 메모리를 비우거나, 트랜잭션을 분리해서 확인해 보세요.

### 3. @Where의 스펠링이나 컬럼명 확인
의외로 단순한 실수일 때도 있습니다.
* DB의 실제 컬럼명이 `is_deleted`가 맞는지 확인해 보세요. (JPA의 필드명인 `isDeleted`가 아니라 **DB 테이블의 물리적인 컬럼명**을 써야 합니다.)
* 하이버네이트 버전에 따라 `is_deleted = 0` (숫자형) 혹은 `is_deleted = 'false'` 등으로 명시해야 할 때도 있습니다.
