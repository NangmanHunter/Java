이 코드를 실행하면 벌어지는 일은 **"화면(프론트엔드)"**과 **"서버(스프링)"** 사이의 소통 방식에 따라 달라집니다.

### 1. 서버(Spring Boot)에서 일어나는 일
`ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();`가 실행되면, 스프링은 HTTP 응답 헤더에 **상태 코드 401 (Unauthorized)**을 담아서 브라우저로 보냅니다. 

* **Body(데이터):** `.build()`를 호출했으므로 본문 내용(JSON 등)은 비어 있습니다.
* **Header:** `HTTP/1.1 401 Unauthorized`가 찍힙니다.

---

### 2. 브라우저(화면)에서 보이는 결과
사용자가 어떤 방식으로 요청했느냐에 따라 결과가 판이하게 다릅니다.

#### A. 브라우저 주소창에 직접 입력하거나 일반적인 페이지 이동(`<a>` 태그 등)일 때
보통 스프링 부트의 기본 설정에 따라 **"White Label Error Page"**라고 불리는 회색빛 시스템 에러 페이지가 뜹니다.
* 거기에 **"401 Unauthorized"**라는 문구가 크게 박혀 나오겠죠. 
* 사용자 입장에서는 "어? 사이트가 고장 났나?"라고 느낄 수 있는 아주 불친절한 화면입니다.

#### B. 자바스크립트(Fetch, Axios)를 이용한 비동기 요청(AJAX)일 때
요즘 대부분의 회원 탈퇴 버튼은 페이지 이동 없이 비동기로 처리하죠. 이럴 때는 화면이 바뀌지 않습니다. 대신 개발자 도구(F12)의 **Network 탭**에 해당 요청이 **빨간색**으로 뜨면서 **401** 에러가 찍히는 걸 볼 수 있습니다.

```javascript
// 프론트엔드 코드 예시
fetch('/delete', { method: 'DELETE' })
  .then(response => {
    if (response.status === 401) {
        alert("로그인 세션이 만료되었습니다. 다시 로그인해주세요.");
        window.location.href = "/login";
    }
  });
```
위와 같이 프론트엔드 개발자가 **401이라는 숫자**를 받아서 "로그인해주세요"라는 알림창을 띄우는 식으로 처리를 하게 됩니다.

---

### 3. 더 나은 방법은 없을까?
서버 개발자 입장에서 "나 로그인 안 된 거 확인했어!"라고 숫자(401)만 던지는 것도 명확하지만, 때로는 **이유**를 같이 담아주는 게 친절합니다.

```java
if (loginMember == null) {
    return ResponseEntity.status(HttpStatus.UNAUTHORIZED)
                         .body("로그인 정보가 없습니다. 다시 로그인해 주세요."); 
}
```
이렇게 `.body()`에 메시지를 실어 보내면, 화면 쪽에서 그 메시지를 그대로 받아서 `alert()` 창에 띄워줄 수 있어 협업하기가 훨씬 좋아집니다.
