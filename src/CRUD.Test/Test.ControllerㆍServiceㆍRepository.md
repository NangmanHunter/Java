아주 중요한 질문입니다! 결론부터 말씀드리면 **"전혀 그렇지 않습니다."**

오히려 실무에서는 **Service 테스트를 가장 많이 만들고 중요하게 여깁니다.** 지금 유저님이 Controller 테스트를 먼저 접하신 이유는 "사용자가 접속해서 글을 쓰는 전체 흐름"을 확인하려다 보니 그렇게 보였을 뿐이에요.

테스트는 보통 아래 세 층(Layer) 모두에 만듭니다.

---

### 1. 테스트의 종류와 위치

| 테스트 종류 | 대상 | 목적 | 비중 |
| :--- | :--- | :--- | :--- |
| **단위 테스트 (Unit)** | **Repository / Entity** | DB 쿼리가 잘 나가는지, 데이터가 잘 계산되는지 확인 | 매우 높음 |
| **단위/통합 테스트** | **Service** | **핵심 비즈니스 로직**이 틀리지 않았는지 확인 (가장 중요!) | **가장 높음** |
| **통합 테스트 (Integration)** | **Controller** | API 호출, JSON 변환, 세션/인증이 잘 작동하는지 확인 | 중간 |

---

### 2. 왜 Service 테스트가 제일 중요한가요?

사실 Controller는 단순히 "요청을 받아서 전달"하는 문지기일 뿐입니다. 진짜 복잡하고 중요한 로직은 다 **Service**에 들어있죠.

예를 들어:
* "게시글을 쓸 때 포인트 10점을 지급한다."
* "로그인 실패를 5번 하면 계정을 잠근다."
* "탈퇴한 회원이 쓴 글은 목록에 나오지 않게 한다."

이런 로직이 잘 돌아가는지는 Controller를 통하지 않고 **Service만 쏙 빼서 테스트**하는 게 훨씬 빠르고 정확합니다.



---

### 3. Repository 테스트는 왜 하나요?

"JPA가 알아서 해주는데 굳이?"라고 생각할 수 있지만, 유저님이 **직접 만든 복잡한 쿼리**(`@Query`나 Querydsl 등)가 있을 때는 이야기가 달라집니다.
* 내가 짠 SQL이 의도한 대로 데이터를 긁어오는지,
* 페이징 처리가 10개씩 딱딱 끊어서 되는지
이런 것들을 Repository 테스트에서 검증합니다.

---

### 4. 실무에서 권장하는 비율 (테스트 피라미드)

보통 **테스트 피라미드**라는 개념을 따릅니다.

1.  **Bottom (Unit Tests)**: Service나 Repository 같은 작은 단위 테스트를 수백, 수천 개 만듭니다. (매우 빠름)
2.  **Middle (Integration Tests)**: Controller처럼 여러 컴포넌트가 섞인 테스트를 수십 개 만듭니다. (약간 느림)
3.  **Top (E2E Tests)**: 실제로 브라우저를 띄워 테스트하는 것은 아주 중요한 기능 몇 개만 합니다. (매우 느림)

### 💡 유저님을 위한 조언

지금은 연습 단계라 Controller 테스트를 통해 전체 흐름을 잡는 게 도움이 되실 거예요. 하지만 나중에 **"로직이 조금 복잡해졌다"** 싶으면 바로 **Service 테스트**를 만들어 보세요.

예를 들어, `BoardService` 테스트라면 `MockMvc` 같은 복잡한 거 없이 그냥 이렇게만 짤 수 있습니다:
```java
@SpringBootTest
@Transactional
class BoardServiceTest {
    @Autowired BoardService boardService;

    @Test
    void 게시글_저장_로직_검증() {
        // Given (준비)
        // When (실행)
        // Then (검증 - 여기서 로직이 맞는지 확인!)
    }
}
```

이제 왜 여러 군데에 테스트를 만드는지 이해가 되시나요? 혹시 지금 작성 중인 `BoardService`의 특정 로직을 따로 테스트해 보고 싶으신가요?
