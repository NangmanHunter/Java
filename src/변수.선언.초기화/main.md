##
자바에서 **"초기화(Initialization)"**와 **"대입(Assignment)"**의 개념을 명확히 구분하면 답이 보입니다. 자바는 변수의 **'종류'**에 따라 초기화에 대해 아주 다른 태도를 취합니다.

---

### 1. 전역 변수(필드): 초기화 없이 진행 가능!
클래스 레벨에서 선언하는 변수(인스턴스 변수, 정적 변수)는 초기화를 하지 않아도 **컴파일 에러가 나지 않습니다.**

* **이유:** 자바가 **기본값(Default Value)**으로 자동 초기화를 해주기 때문입니다.
* **동작:** `int`는 `0`, `boolean`은 `false`, 객체 참조(String 등)는 `null`로 채워집니다.
* **메모리:** 아까 말씀드린 **BSS(또는 자바의 Heap/Method Area 내 기본값 영역)**에 일단 0으로 자리를 잡습니다.

```java
public class Test {
    int a; // 초기화 안 함! 하지만 에러 안 남.
    
    void print() {
        System.out.println(a); // 0이 출력됨
    }
}
```

### 2. 지역 변수: 사용 전 반드시 초기화 필수!
메서드 안에서 선언하는 변수는 초기화를 하지 않고 **'사용'**하려고 하면 무조건 **컴파일 에러**가 납니다.

* **이유:** 스택 영역은 함수가 호출될 때마다 빠르게 생성되고 사라지는데, 이때 자바가 일일이 0으로 닦아줄 시간이 없습니다(성능 때문). 그래서 "사용자가 직접 값을 넣기 전까지는 쓰레기 값이 들어있을 수 있으니 읽지 마!"라고 차단하는 것입니다.
* **예외:** 선언만 해두고 **사용(읽기)**을 안 한다면 에러는 나지 않습니다. 하지만 그 변수를 가지고 계산을 하거나 출력하려 하면 바로 에러가 뜹니다.

```java
public void myMethod() {
    int a; // 선언만 함 (OK)
    
    // System.out.println(a); // 여기서 바로 사용하면 컴파일 에러!
    
    a = 10; // 값을 넣음 (초기화 완료)
    System.out.println(a); // 이제는 사용 가능 (OK)
}
```

---

### 3. "그럼 무조건 에러인가요?"에 대한 묘수
만약 어떤 로직에 따라 값이 들어갈 수도, 안 들어갈 수도 있는 상황이라면 자바 컴파일러는 매우 깐깐하게 검사합니다.

```java
public void logic(boolean flag) {
    int a;
    if (flag) {
        a = 100;
    }
    // System.out.println(a); // 컴파일 에러! (flag가 false면 a가 초기화 안 될 수도 있으니까)
}
```

이럴 때는 보통 **`int a = 0;`** 처럼 의미 없는 기본값이라도 넣어주는 것이 자바의 관례입니다.

### 4. 정리: 자바의 초기화 규칙

| 변수 종류 | 자동 초기화 여부 | 사용 시 제약 | 저장 메모리 위치 |
| :--- | :--- | :--- | :--- |
| **전역/정적 변수** | **O** (0, null 등) | 자유롭게 사용 가능 | Data/BSS (Heap, Method Area) |
| **지역 변수** | **X** | **사용 전 반드시 명시적 초기화** | Stack |
