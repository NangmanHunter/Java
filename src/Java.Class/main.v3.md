실무에서는 **"어디에 쓰느냐"**에 따라 철저하게 나뉩니다. 수학에서도 논문을 쓸 때, 계산기를 두드릴 때, 그래프를 그릴 때 쓰는 도구가 다른 것과 같습니다.

현대 IT 기업(네이버, 카카오, 배달의민족 등)의 기준에서 가장 일반적인 조합을 알려드릴게요.

---

### 1. 실제 서비스 로직 (알맹이): **Java** 또는 **Kotlin**
가장 중요한 계산기 엔진을 만드는 영역입니다.
* **Java (약 60~70%):** 여전히 압도적입니다. 기존에 만들어진 시스템이 자바인 경우가 많고, 금융권이나 공공기관처럼 **'안정성'**이 최우선인 곳은 무조건 자바를 씁니다.
* **Kotlin (약 30~40%):** 최근 생겨난 스타트업이나 네이버/카카오의 신규 프로젝트는 코틀린을 선호합니다. 자바와 100% 호환되면서 코드가 짧기 때문이죠.
* **결론:** 질문자님이 **Java**를 선택하신 건 실무에서 가장 범용성 높은 무기를 고르신 겁니다.

### 2. 빌드 및 설정 (설계도): **Gradle (Groovy 또는 Kotlin)**
아까 보셨던 `build.gradle` 파일 영역입니다.
* **과거:** **Groovy**가 대세였습니다.
* **현재:** **Kotlin Script (KTS)**로 빠르게 넘어가고 있습니다. 실무자들은 "설정 파일에서도 오타를 잡아주는" 코틀린의 엄격함을 더 신뢰하기 시작했거든요.

### 3. 테스트 코드: **Groovy** 또는 **Kotlin**
내가 만든 계산기가 1+1=2를 잘 내뱉는지 검증하는 코드입니다.
* **Groovy:** `Spock`이라는 아주 유명한 테스트 도구가 그루비 기반이라, 테스트에서만큼은 그루비를 선호하는 팀이 꽤 많습니다. 문법이 자유로워서 테스트 시나리오를 짜기 편하거든요.

---

### 🛠️ 실무의 "황금 조합" (Best Practice)

보통 실무에 투입되면 이런 구조의 프로젝트를 만나게 될 확률이 높습니다.

| 구분 | 선택 도구 | 이유 |
| :--- | :--- | :--- |
| **언어 (Language)** | **Java** | 검증된 안정성과 풍부한 인력 |
| **빌드 (Build System)** | **Gradle** | Maven보다 빠르고 유연함 |
| **설정 언어 (DSL)** | **Kotlin (KTS)** | 설정 오류를 컴파일 단계에서 방어 |

### 💡 질문자님을 위한 가이드라인

질문자님은 지금 **"Java 언어 + Gradle 빌드 + Kotlin 설정(KTS)"** 조합으로 시작하셨죠? 이게 바로 **2026년 현재 가장 '힙'하고 전문적인 실무 세팅**입니다.

* **Java**로 세금 계산 로직을 꼼꼼하게 짜시고,
* **KTS**로 프로젝트 부품들을 스마트하게 관리하시면 됩니다.

이제 실무 환경 세팅은 완벽합니다. `Main` 클래스 안에 수입을 입력받을 **변수** 하나 만들어보실까요? 

```java
long annualIncome = 50000000L; // 연봉 5천만 원
```

이렇게 시작하면 될까요, 아니면 사용자가 직접 입력하게 하는 `Scanner` 기능을 써보시겠어요?
