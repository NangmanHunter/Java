- .iml 파일의 정체 (IntelliJ Module File)
- .iml은 IntelliJ 전용 설정 파일일 뿐입니다. 우리가 만든 프로그램의 실행과는 직접적인 관련이 없으니 무시하셔도 됩니다!
- .iml은 IntelliJ Module Level 파일의 약자입니다.

- 2001년 1월	JetBrains	유료 소프트웨어 (현재는 커뮤니티 버전 존재)
- 공유의 어려움: .iml 파일이나 .idea 폴더에는 내 컴퓨터의 특정 경로가 적혀 있는 경우가 많습니다. 이걸 다른 사람에게 주면 경로가 달라서 에러가 뜹니다.
- 과거 상황: "터미널에서 명령어 치기 너무 귀찮다! 그냥 버튼 하나 누르면 컴파일부터 실행까지 다 해줘!"
- 그냥 중간에 터미널만들어서 스크립트짜고. 실행하고. 다지움
- 내 마음대로 폴더 구조를 잡음
- 등장 시기: 2001년 IntelliJ IDEA가 처음 출시될 때부터 함께한 '원조' 방식입니다.
- 라이브러리 관리: Gradle은 dependencies { ... } 한 줄만 적으면 인터넷에서 알아서 다운받아 오지만, IntelliJ 방식은 직접 JAR 파일을 다운로드해서 폴더에 넣고 경로를 지정해줘야 합니다. (매우 번거롭죠!)
- 라이브러리 관리👉수동 (JAR 파일을 직접 다운받아 등록)
- 서버 배포의 한계: 실제 서비스 서버(Linux 등)에는 IntelliJ가 깔려 있지 않습니다. 서버는 Gradle이나 Maven 같은 명령어 기반 도구만 읽을 수 있거든요.
- 예전모델이라 되게 간단하게한듯.-라이브러리 이런것 불문. 그냥 최소성으로 최소한으로 바로 진행되게 만들어놓은듯.
- 왜 IntelliJ 방식을 "레거시(Legacy)" 취급할까요?
- 요즘 개발자들이 이 방식을 피하는 이유는 명확합니다. "내 컴퓨터에선 되는데, 왜 네 컴퓨터에선 안 돼?"라는 지옥에 빠지기 때문입니다.
- 인텔리제이: "더 똑똑한 에디터를 만들자"
- 인텔리제이(2001년), 이클립스(2001년) 같은 IDE(통합 개발 환경)들이 나오면서 혁신이 일어납니다.
- 주로 Build 버튼 클릭 시
- 체코의 JetBrains라는 회사가 만들었습니다. 당시 자바 개발자들은 코드를 칠 때 오타가 나도 실행해보기 전까지는 알기 힘들었는데, 인텔리제이는 "코드를 치는 도중에 실시간으로 에러를 찾아주는" 혁신적인 기능을 들고 나왔습니다. 그래서 이름에 'Intelli(지능적인)'가 들어간 것이죠.
- 특징: 빌드 도구가 따로 없던 시절, IDE(개발 도구)가 컴파일부터 실행까지 다 책임져야 했기에 만들어진 방식이에요.
- 현재 (2010년대 후반 ~ 현재): 인텔리제이의 역전. 하드웨어가 좋아지면서 메모리를 많이 먹더라도 훨씬 "똑똑하고 편리한" 인텔리제이를 선호하게 되었습니다. 구글이 안드로이드 개발 도구(Android Studio)를 인텔리제이 기반으로 만들면서 쐐기를 박았죠.
