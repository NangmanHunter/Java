왜 매번 선언해야 하는지, 어떤 마법을 부리는지 쉽게 핵심만 짚어드릴게요.

---

### 1. "귀찮은 일은 내가 할게, 너는 로직만 써"

기존 Vue(Options API나 초기 Composition API)에서는 변수나 함수를 선언한 뒤, 반드시 "이거 템플릿(HTML)에서 쓸 거야!"라고 일일이 등록해줘야 했습니다.

* **`<script setup>`이 없을 때:**
```javascript
export default {
  setup() {
    const count = ref(0)
    const increment = () => count.value++

    // 템플릿에 내보내기 위해 일일이 리턴해줘야 함 (매우 귀찮음)
    return { count, increment } 
  }
}

```


* **`<script setup>`을 쓸 때:**

```javascript
    const count = ref(0)
    const increment = () => count.value++
    // 리턴? 그게 뭐죠? 그냥 선언하면 템플릿에서 바로 사용 가능!
```

### 2. 컴파일러가 해주는 마법
우리가 `<script setup>` 안에 코드를 짜면, Nuxt/Vue의 빌드 도구가 코드를 읽어서 **"아, 이 변수는 템플릿에서 쓰고 있네? 자동으로 노출시켜줄게"**라며 코드를 뒤에서 다시 조립해줍니다. 덕분에 코드가 훨씬 간결해지고 가독성이 좋아지죠.

### 3. 왜 매번 써야 하나요?
자바스크립트 파일(`useAuth.ts` 같은 컴포저블)과 뷰 파일(`.vue`)의 차이 때문입니다.

*   **일반 JS/TS 파일:** 그냥 함수를 만들고 `export`만 하면 됩니다.
*   **`.vue` 파일:** "데이터(Script) + 화면(Template) + 스타일(Style)"이 한 파일에 섞여 있습니다. 이 둘(데이터와 화면)을 **연결해주는 다리**가 바로 `setup`입니다. 이 다리가 없으면 템플릿은 스크립트 안에 뭐가 있는지 전혀 모르는 상태가 됩니다.

---

### 4. `setup`이 붙으면 달라지는 점들
*   **자동 임포트(Auto-import):** 컴포넌트를 가져와서 쓸 때 원래는 `components: { MyComponent }`라고 등록해야 했지만, `setup`에서는 그냥 `import`만 하거나(Nuxt는 심지어 생략 가능) 선언만 하면 바로 태그로 쓸 수 있습니다.
*   **성능 최적화:** 컴파일러가 코드를 더 효율적으로 분석해서 실행 시점에 성능이 더 빠릅니다.
*   **상수 선언의 자유:** `const`나 `function`을 평범한 자바스크립트 짜듯이 선언해도 뷰의 반응형 시스템과 찰떡같이 붙습니다.

---

### 한마디로 요약하자면?
