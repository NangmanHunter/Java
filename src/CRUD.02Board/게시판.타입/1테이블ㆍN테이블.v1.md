##
게시판 기능을 구현할 때 테이블을 합칠지(공통 테이블), 나눌지(개별 테이블)는 서비스의 확장성과 데이터 성격에 따라 결정됩니다. 결론부터 말씀드리면, **대부분의 경우 '1개의 공통 테이블'로 관리하고 `type` 컬럼으로 구분하는 방식이 훨씬 효율적**입니다.

각 방식의 장단점을 비교해 드릴게요.

---

### 1. 하나의 테이블로 합치는 방식 (추천)
`board` 테이블 하나에 `category` 또는 `type` (NOTICE, COMMUNITY 등) 컬럼을 추가하여 구분하는 방식입니다.

* **장점:**
    * **코드 재사용성:** 게시글 조회, 작성, 수정, 삭제 로직을 하나의 API/서비스 코드로 공통화할 수 있습니다.
    * **통합 검색:** 공지사항과 일반 게시글을 한 번에 검색해야 할 때 쿼리가 단순합니다.
    * **인프라 효율:** 테이블 개수가 적어 관리 포인트가 줄어들고, 인덱스 관리도 효율적입니다.
* **단점:**
    * 특정 게시판에만 필요한 컬럼이 생길 경우(예: 공지사항에만 필요한 '상단 고정 기한' 등), 다른 타입에서는 해당 컬럼이 `NULL`이 됩니다. (하지만 큰 문제는 아닙니다.)


### 2. 개별 테이블로 나누는 방식
`notice_board`, `community_board`처럼 별도의 테이블을 만드는 방식입니다.

* **장점:**
    * **데이터 독립성:** 각 게시판의 성격이 완전히 달라 컬럼 구성이 80% 이상 차이 난다면 관리가 명확합니다.
    * **성능 분리:** 특정 게시판에 데이터가 수백만 건씩 쌓여도 다른 게시판 조회 성능에 영향을 주지 않습니다.
* **단점:**
    * **코드 중복:** 비슷한 CRUD 로직을 테이블마다 따로 작성하거나 복잡한 상속 구조를 설계해야 합니다.
    * **통계의 어려움:** '내가 쓴 모든 글 보기' 같은 기능을 구현할 때 여러 테이블을 `UNION`으로 묶어야 해서 쿼리가 무거워집니다.

### 💡 판단 기준 정리



| 고려 요소 | 추천 방식 | 이유 |

| **컬럼의 유사도** | **1개 테이블** | 제목, 내용, 작성자, 조회수 등 기본 구조가 같다면 합치는 게 이득입니다. |
| **관리의 편의성** | **1개 테이블** | 새로운 게시판(Q&A, FAQ 등)이 추가될 때 테이블 생성 없이 `type` 값만 추가하면 됩니다. |
