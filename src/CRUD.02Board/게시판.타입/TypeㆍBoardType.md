필드명은 사소해 보이지만 나중에 API 문서를 만들거나 프론트엔드와 협업할 때 의외로 큰 영향을 줍니다. 결론부터 말씀드리면, 저는 **`type`**을 조금 더 추천하지만, 상황에 따라 **`boardType`**이 나은 경우도 있습니다.

비교해 보시고 본인의 스타일이나 프로젝트 성격에 맞는 걸 골라보세요.

---

### 1. `type` (간결함 중시)
객체 지향적인 관점에서는 클래스 이름이 이미 `Board`이기 때문에, `board.getType()`이라고 부르는 것이 자연스럽습니다.

* **장점:**
    * **가독성:** 코드가 간결해집니다. `board.getType()`은 "게시판의 타입"이라는 의미가 직관적입니다.
    * **일관성:** 만약 다른 엔티티(예: `Member`)에도 타입을 나눈다면, 모두 `type`이라는 공통 필드명을 쓸 수 있어 규칙이 생깁니다.
    * **프론트엔드 편의성:** JSON 데이터로 나갈 때 `{ "type": "NOTICE" }`로 표현되어 데이터 구조가 깔끔합니다.
* **단점:**
    * **DB 쿼리 모호함:** SQL 쿼리를 직접 짤 때 `SELECT * FROM board WHERE type = ...` 라고 하면, 여러 테이블을 조인할 때 어느 테이블의 `type`인지 헷갈릴 수 있습니다. (물론 Alias를 쓰면 해결됩니다.)

### 2. `boardType` (명확함 중시)
데이터베이스 관점이나 대규모 프로젝트에서 필드명의 고유성을 중시할 때 사용합니다.

* **장점:**
    * **검색 용이성:** IDE(IntelliJ 등)에서 `boardType`으로 검색하면 딱 게시판 관련 로직만 나옵니다. `type`은 너무 흔한 단어라 검색 결과가 너무 많을 수 있습니다.
    * **조인 시 명확함:** 다른 테이블과 조인했을 때 별칭 없이도 `board_type`이라는 컬럼명만 보고 용도를 바로 알 수 있습니다.
* **단점:**
    * **중복 느낌 (Redundancy):** `board.getBoardType()`은 "게시판.게시판타입()"이 되어 의미가 중복되는 느낌을 줍니다.

### 💡 저의 추천: `type`

현재 진행 중인 `bitcopark-base` 프로젝트는 **도메인 모델**을 기반으로 하고 있으므로, 객체 내부에서는 **`type`**이라고 부르는 것이 훨씬 "객체답게" 느껴집니다.

만약 DB 컬럼명이 겹치는 게 걱정된다면, 엔티티 필드명은 `type`으로 하되 DB 컬럼명만 `@Column`으로 지정해 줄 수도 있습니다.

```java
@Enumerated(EnumType.STRING)
@Column(name = "board_type", nullable = false) // DB에는 명확하게 board_type으로 저장
private BoardType type; // 자바 코드에서는 간결하게 type으로 사용
```

이렇게 하면 **자바 코드의 간결함**과 **DB의 명확함**을 동시에 잡을 수 있습니다.
