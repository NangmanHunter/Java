### 🛠️ 스프링 내부의 처리 프로세스

1.  **매칭 단계 (Matching):**
    스프링이 들어온 URL(`.../notice`)과 컨트롤러의 `@PostMapping("/{boardType}")`을 매칭합니다.
2.  **타입 확인 (Target Type Check):**
    `@PathVariable` 옆에 붙은 타입을 봅니다. "어? **`BoardType`**이네?" 하고 **최종 목적지 타입**을 확인합니다.
3.  **컨버터 검색 (Converter Lookup):**
    스프링의 변환 센터(ConversionService)에 물어봅니다. 
    > "지금 들어온 건 **`String`**("notice")인데, 이걸 **`BoardType`**으로 바꿀 수 있는 컨버터가 있니?"
4.  **컨버터 실행 (Conversion):**
    유저님이 등록하신 람다식이 호출됩니다. 
    * `"notice"` → `.toUpperCase()` → `"NOTICE"`
    * `BoardType.valueOf("NOTICE")` → **`BoardType.NOTICE` (Enum 객체)**
5.  **인자 주입 (Argument Injection):**
    완성된 Enum 객체를 컨트롤러 메서드의 `boardType` 파라미터에 쏙 넣어줍니다.



---

### 💡 왜 이 순서가 중요할까요?

유저님이 말씀하신 **"PathVariable에서 BoardType으로 정해놨으니"**라는 부분이 핵심입니다. 

만약 컨트롤러에 `@PathVariable String boardType`이라고 적어놨다면, 스프링은 "어차피 받을 타입이 String이네? 변환할 필요 없네!" 하고 컨버터를 아예 호출하지 않았을 거예요. 

하지만 **`BoardType`**이라고 타입을 명시했기 때문에, 스프링이 "아, 이건 변환이 필요한 녀석이구나!"라고 판단해서 유저님이 만든 컨버터로 길을 안내한 것입니다.

### ✅ 요약하자면

1.  **URL**은 언제나 **String**이다.
2.  **컨트롤러 파라미터 타입**이 **목적지**다.
3.  **컨버터**는 그 사이를 이어주는 **다리**다.

이제 이 메커니즘을 완벽히 이해하셨으니, 나중에 날짜(`LocalDate`)나 다른 복잡한 객체를 URL 파라미터로 받을 때도 똑같은 원리로 처리하실 수 있을 거예요.
