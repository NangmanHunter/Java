##
오, 아주 날카로운 지적입니다! 정확하게 짚으셨어요.

원래 자바의 `registry.addConverter` 메서드는 기본적으로 `Converter` **인터페이스 객체**를 인자로 받습니다. 그런데 제가 제안한 두 번째 방식(`String.class, BoardType.class, ...`)은 단순히 람다만 쓴 게 아니라, **스프링이 제공하는 또 다른 "편의용 메서드"**를 활용한 것입니다.

이게 어떻게 가능한지, 그 원리를 깔끔하게 정리해 드릴게요.

---

### 1. 왜 두 코드가 "결과적으로" 같을까요?

사실 `addConverter`는 **오버로딩(Overloading)**이 되어 있어서 여러 가지 형태로 호출할 수 있습니다.

* **방식 1 (인터페이스 직접 구현):** `addConverter(Converter<?, ?> converter)` 메서드를 사용하는 방식입니다.
* **방식 2 (타입 명시 + 람다):** `addConverter(Class<S> sourceType, Class<T> targetType, Converter<? super S, ? extends T> converter)` 메서드를 사용하는 방식입니다.

즉, 두 번째 방식은 **"입력은 String이고 출력은 BoardType이야"**라고 타입을 미리 알려줬기 때문에, 세 번째 인자로 들어가는 람다(`source -> ...`)가 자연스럽게 `Converter` 인터페이스의 `convert` 메서드 역할을 수행할 수 있게 되는 것입니다.

---

### 2. 람다식이 가능한 근본적인 이유: @FunctionalInterface

자바에서 람다식을 쓰려면 해당 인터페이스가 **"추상 메서드를 딱 하나만 가진 인터페이스"**여야 합니다. 이를 **함수형 인터페이스**라고 부르죠.

`Converter` 인터페이스를 열어보면 이렇게 생겼습니다:

```java
@FunctionalInterface // <-- 이 어노테이션이 람다를 허용한다는 증표입니다!
public interface Converter<S, T> {
    T convert(S source); // 추상 메서드가 이거 하나뿐입니다.
}
```



---

### 3. 비교 분석: 어느 게 더 유리할까요?

| 구분 | 익명 객체 방식 (방식 1) | 람다 + 타입 명시 방식 (방식 2) |
| :--- | :--- | :--- |
| **가독성** | 코드가 길고 복잡함 | 한 줄로 끝나서 깔끔함 |
| **타입 추론** | 제네릭을 직접 적어야 함 | 메서드 인자에서 타입을 명확히 선언함 |
| **동작 원리** | 클래스를 새로 정의해서 객체 생성 | 함수형 인터페이스를 람다로 구현 |

### 💡 정리하자면
질문하신 대로 **"람다식 자체가 지원해줘야 한다"**는 말씀이 맞습니다! 스프링의 `Converter`는 `@FunctionalInterface`이기 때문에 자바 8부터 람다 문법을 완벽하게 지원합니다.

두 번째 방식이 더 짧은 이유는 **스프링이 "타입 두 개 알려주면 람다는 내가 알아서 Converter로 포장해줄게!"**라고 미리 메서드를 만들어놨기 때문이에요. 

이제 안심하고 두 번째 방식을 쓰셔도 됩니다. 훨씬 '모던'한 자바 코드 느낌이 나거든요! 어떤 방식이 더 직관적으로 느껴지시나요?
