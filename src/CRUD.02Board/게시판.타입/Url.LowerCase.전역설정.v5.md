맞습니다! 통찰력이 정말 좋으시네요. **HTTP 통신 수준에서 URL은 100% 문자열(String)입니다.** 브라우저 주소창에 치는 모든 것은 결국 텍스트니까요.

그럼에도 불구하고 왜 굳이 `String.class`를 적어줘야 하는지, 그 **"백엔드 내부의 속사정"**을 알려드릴게요.

---

### 1. 스프링은 '만능 번거로움 해결사'이기 때문입니다.
스프링의 `FormatterRegistry`(또는 `ConversionService`)는 단순히 URL 문자열만 처리하는 곳이 아닙니다. 이 녀석은 애플리케이션 곳곳에서 발생하는 **모든 타입 변환**을 담당하는 거대한 중앙 터미널이에요.

예를 들어:
* **문자열** → **Enum** (지금 유저님이 하시는 것)
* **숫자(Integer)** → **문자열(String)**
* **날짜(Date)** → **Calendar**
* **커스텀 객체 A** → **커스텀 객체 B**

이 터미널에는 수많은 변환 규칙이 등록되어 있습니다. 만약 출발지(`String.class`)를 알려주지 않으면, 스프링은 이 수천 개의 규칙 중에서 어떤 상황에 유저님의 람다식을 꺼내 써야 할지 결정하지 못합니다.

### 2. "URL은 String이다"라는 건 우리만 아는 사실입니다.
컴퓨터 입장에서 `addConverter` 메서드는 그저 **"어떤 데이터를 변환하는 로직을 등록하는 행위"**일 뿐입니다.

* **우리 생각:** "어차피 URL은 다 String이니까 당연히 String을 BoardType으로 바꾸는 거겠지?"
* **자바/스프링 생각:** "내가 가진 변환 지도에 등록하려면 **'재료(Source)'**가 무엇인지 확실히 적어줘야 해. 그래야 나중에 `String`이 들어왔을 때만 이 로직을 실행하지!"

### 3. 메서드 설계의 규칙 (오버로딩)
자바 문법적으로도 `addConverter` 메서드가 그렇게 설계되어 있습니다.

```java
// 스프링 내부 코드 (개념적)
void addConverter(Class sourceType, Class targetType, Converter converter);
```

이 메서드를 호출하려면 인자 3개를 다 채워야 합니다. 자바는 "대충 String이겠거니" 하고 넘어가는 법이 없거든요. **"String이라는 원재료를 넣으면 BoardType이라는 결과물이 나온다"**는 공식을 완성하기 위해 `String.class`가 필요한 것입니다.
