##
아주 좋은 설계 포인트입니다. "기능을 따로 팔 것인가, 아니면 기존 것을 재사용할 것인가"의 고민이죠.

결론부터 말씀드리면, **기존의 `findOne` 로직은 그대로 두고, DTO(응답 객체)에 `isWriter` 같은 플래그 값을 하나 추가해서 처리하는 방식**을 가장 추천합니다.

왜 그런지 이유와 함께 깔끔한 구현 전략을 짜드릴게요.

---

## 1. 왜 별도의 API를 파지 않는 게 좋을까?

* **네트워크 낭비:** 프론트엔드 입장에서는 게시글 상세 정보를 가져오기 위해 API를 호출했는데, "내가 작성자인가?"를 알기 위해 또 다른 API를 호출하는 것은 비효율적입니다.
* **유지보수:** 게시글 정보를 가져오는 로직이 두 군데(`findOne`, `checkWriter`)로 나뉘면, 나중에 게시글 관련 로직이 바뀔 때 두 곳을 다 고쳐야 할 수도 있습니다.

---

## 2. 추천하는 구현 전략: DTO에 필드 추가

게시글 상세 데이터를 줄 때, **"이 글을 요청한 사람이 작성자인가?"**에 대한 결과를 아예 포함해서 보내주는 것입니다.

### (1) 백엔드: BoardResponse DTO 수정
```java
@Getter
public class BoardResponse {
    private Long id;
    private String title;
    private String content;
    private String writerName;
    private boolean isWriter; // <-- 이 필드가 핵심입니다!

    // 생성자에서 로그인한 사용자 ID와 작성자 ID를 비교합니다.
    public BoardResponse(Board board, Long loginMemberId) {
        this.id = board.getId();
        this.title = board.getTitle();
        this.content = board.getContent();
        this.writerName = board.getMember().getName();
        // 로그인한 ID가 작성자 ID와 같으면 true, 아니면 false
        this.isWriter = board.getMember().getId().equals(loginMemberId);
    }
}
```

### (2) 백엔드: Controller 수정
상세 조회 시 현재 로그인한 사용자의 ID를 넘겨줍니다.
```java
@GetMapping("/{id}")
public ResponseEntity<BoardResponse> getBoard(
        @PathVariable Long id,
        @SessionAttribute(name = "loginMember", required = false) Member loginMember) {
    
    Board board = boardService.findOne(id);
    Long loginId = (loginMember != null) ? loginMember.getId() : null;
    
    return ResponseEntity.ok(new BoardResponse(board, loginId));
}
```

## 3. 프론트엔드: 버튼 노출 제어

이렇게 API 응답에 `isWriter`가 포함되면, 프론트엔드 코드는 놀라울 정도로 단순해집니다.

```javascript
async function loadDetail(id) {
    const response = await fetch(`/api/boards/${id}`);
    const board = await response.json();

    // 데이터 바인딩
    document.getElementById('title').innerText = board.title;
    document.getElementById('content').innerText = board.content;

    // 권한 확인 (서버가 보내준 결과 그대로 사용)
    const updateBtn = document.getElementById('board-update');
    if (!board.isWriter) {
        updateBtn.style.display = 'none'; // 내가 쓴 글이 아니면 버튼 숨기기
    }
}
```

---

## 4. 이 방식의 장점

1.  **일관성:** "작성자 확인 로직"을 백엔드의 **한 곳(DTO 생성 시점)**에서 관리할 수 있습니다. 
2.  **보안:** 프론트엔드에서 ID 두 개를 비교하는 것보다, 서버에서 "작성자 여부"를 판별해서 넘겨주는 것이 로직 노출 면에서 더 안전합니다.
3.  **UI/UX:** 페이지가 로딩될 때 게시글 내용과 버튼 노출 여부가 동시에 결정되므로 화면 깜빡임이 적습니다.
