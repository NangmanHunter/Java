Repository에서 `@Query`를 이용해 DB에 직접 업데이트를 치는 방식으로 가면, 엔티티 안에 만들었던 `addViewCount()` 메서드는 사실상 **쓸모가 없어집니다.**

이 상황을 어떻게 받아들여야 할지, 그리고 왜 실무에서도 이런 고민을 하는지 핵심만 짚어 드릴게요.

---

### 1. 왜 `addViewCount()`가 필요 없어질까?
`addViewCount()`는 **애플리케이션 메모리**에 올라온 객체의 숫자를 올리는 용도입니다. 하지만 Repository 방식은 객체를 거치지 않고 **DB로 바로 화살을 쏘는 것**과 같습니다.

* **엔티티 메서드 방식:** DB에서 데이터를 '가져와서' -> 메모리에서 '수정하고' -> 다시 DB에 '저장'
* **Repository 방식:** DB에 있는 데이터를 '그 자리에서 바로 +1'

결과적으로 비즈니스 로직이 엔티티에서 Repository(쿼리)로 옮겨간 셈이죠.

---

### 2. 그럼에도 불구하고 엔티티 메서드를 남겨두는 경우
하지만 어떤 개발자들은 여전히 엔티티에 메서드를 남겨두기도 합니다. 그 이유는 **'단위 테스트'** 때문입니다.

* **DB 연결 없이** 엔티티만 가지고 로직이 잘 작동하는지 테스트하고 싶을 때 `addViewCount()`가 있으면 테스트 코드를 짜기가 아주 편합니다.
* 만약 나중에 "조회수를 올릴 때 포인트도 같이 지급한다" 같은 복잡한 **객체 지향적 로직**이 추가된다면, 쿼리 하나로 해결하기 어려워지므로 다시 엔티티 메서드가 중요해집니다.

---



### 3. Repository 방식 채택 시 주의할 점 (영속성 컨텍스트)
이게 가장 중요한 기술적 포인트입니다. 

`@Modifying` 쿼리는 JPA를 건너뛰고 DB에 바로 실행되기 때문에, **이미 메모리에 올라와 있는 `Board` 객체의 조회수**는 여전히 0일 수 있습니다. (DB는 1이 됐는데 말이죠.)

그래서 아래처럼 처리하는 게 정석입니다.

```java
@Modifying(clearAutomatically = true) // 쿼리 실행 후 메모리(영속성 컨텍스트)를 비워줌
@Query("update Board b set b.viewCount = b.viewCount + 1 where b.id = :id")
int updateViewCount(@Param("id") Long id);
```

`clearAutomatically = true`를 붙여주면, JPA가 메모리를 비우고 다음번 조회 때 DB에서 **실제로 1이 올라간 최신 데이터**를 다시 가져오게 됩니다.
