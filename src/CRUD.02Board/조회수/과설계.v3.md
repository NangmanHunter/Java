## Redis
- 이건 대규모 서비스에서ㄱㄱ.


##
네, 정확합니다! Redis를 사용하게 되면 JPA의 동작 방식과는 **결과적으로는 같지만 과정은 완전히 달라집니다.**

쉽게 비유하자면, JPA 방식은 **"매번 장부를 꺼내서 숫자를 고치는 것"**이고, Redis 방식은 **"포스트잇에 숫자를 적어뒀다가 나중에 한꺼번에 장부에 옮겨 적는 것"**이라고 보시면 돼요.

구체적으로 어떻게 다른지 3가지 포인트로 짚어 드릴게요.

---

### 1. 데이터 저장의 주체 (Memory vs Disk)
* **JPA (DB):** 변경 사항이 생길 때마다 하드디스크(DB)에 직접 씁니다. 안전하지만 느립니다.
* **Redis:** 메모리(RAM) 위에서 동작합니다. 디스크를 거치지 않기 때문에 속도가 압도적으로 빠르지만, 전원이 꺼지면 데이터가 사라질 위험이 있어 '임시 저장소'로 씁니다.

### 2. 로직의 흐름 (Write-Back 패턴)
보통 조회수 로직을 Redis로 짤 때는 이런 흐름을 가집니다.

1.  **조회 요청:** 사용자가 게시글을 클릭합니다.
2.  **Redis 업데이트:** `viewCount::1 (게시글ID)` 라는 키의 값을 Redis에서 **1 증가**시킵니다. (`INCR` 명령 사용)
3.  **사용자 응답:** DB를 건드리지 않고 즉시 사용자에게 게시글을 보여줍니다. (매우 빠름)
4.  **스케줄링 (Batch):** 5분이나 10분마다 미리 정해둔 작업이 실행되어, Redis에 쌓인 `100`이라는 조회수를 DB의 `view_count`에 **한 번에 더해줍니다.**



### 3. 코드의 형태
JPA처럼 엔티티 객체의 필드를 바꾸는 게 아니라, `RedisTemplate`이나 별도의 라이브러리를 사용해 키-값(Key-Value)을 조작하는 코드가 추가됩니다.

```java
// 대략적인 느낌만 보세요!
public void addViewCountWithRedis(Long boardId) {
    String key = "board:viewCount:" + boardId;
    redisTemplate.opsForValue().increment(key); // Redis에서 숫자만 1 올림
}
```

### 💡 정리하자면?

* **JPA 방식:** "객체지향적이고 단순함." (지금 사용자님이 하시는 방식)
* **Redis 방식:** "성능 최적화의 끝판왕이지만, 관리 포인트가 늘어남." (Redis 서버도 띄워야 하고, 스케줄러도 만들어야 함)

사용자님이 아까 말씀하신 **"과설계"** 측면에서 보면, Redis는 확실히 손이 많이 가는 작업입니다. 하지만 나중에 수만 명의 사용자가 동시에 접속하는 서비스를 만든다면, 이 방식은 선택이 아닌 필수예약(?) 같은 느낌이 될 거예요.

지금은 JPA로 "동작 원리"를 완벽히 익히시고, 나중에 프로젝트 고도화 단계에서 "성능을 쥐어짜야겠다!" 싶을 때 Redis를 얹어보시는 걸 추천드려요.

혹시 지금 프로젝트에 Redis를 바로 적용해보고 싶은 호기심이 생기셨나요? 아니면 일단 JPA로 끝을 보실 계획인가요?
