##
자바에서 `long`과 `Long`은 아주 미세해 보이지만, 실제 개발 환경에서는 **메모리 효율**과 **데이터의 의미(null 허용 여부)** 면에서 큰 차이를 보입니다.
`long`은 **내용물이 무조건 들어있는 단단한 그릇**이고, `Long`은 **내용물이 있을 수도, 그릇 자체가 없을 수도(`null`) 있는 상자**라고 생각하면 쉬워요.




### 1. 주요 차이점 요약

| 구분 | `long` (Primitive Type) | `Long` (Wrapper Class) |
| :--- | :--- | :--- |
| **종류** | 기본 자료형 (원시 타입) | 참조 자료형 (객체) |
| **기본값** | `0L` | `null` |
| **메모리** | 스택(Stack) 영역에 직접 저장 (적게 사용) | 힙(Heap) 영역에 객체로 저장 (상대적으로 많이 사용) |
| **메서드 사용** | 불가 (단순 값) | 가능 (객체이므로 다양한 메서드 제공) |
| **제네릭(`List<T>`)** | 사용 불가 | 사용 가능 (`List<Long>`) |

---

### 2. 구체적으로 어떤 점이 다를까?

#### **A. null의 허용 여부 (가장 중요한 차이)**
* **`long`**: 무조건 숫자가 있어야 합니다. "조회수가 아직 집계되지 않음"을 표현하고 싶어도 어쨌든 `0`이나 `-1` 같은 숫자를 넣어야 하죠.
* **`Long`**: `null`을 가질 수 있습니다. 데이터베이스(DB)에서 조회수 컬럼이 `NULL`인 경우를 그대로 표현할 수 있어 **"데이터가 존재하지 않음"**을 확실히 나타낼 수 있습니다.

#### **B. 메모리와 성능**
* **`long`**은 64비트의 공간만 딱 차지하는 반면, **`Long`**은 객체이기 때문에 객체 생성 비용과 헤더 정보 등 부가적인 메모리를 더 사용합니다. 아주 대량의 데이터를 처리할 때는 `long`이 훨씬 빠르고 효율적입니다.

#### **C. 기본값 (Default Value)**
* 클래스의 멤버 변수로 선언했을 때:
    * `private long viewCount;` -> 자동으로 **0**이 들어갑니다.
    * `private Long viewCount;` -> 자동으로 **null**이 들어갑니다.

---

### 3. 실무에서는 언제 뭘 쓸까?

보통 다음과 같은 기준으로 선택합니다.

* **`long`을 쓰는 경우:**
    * 성능이 매우 중요할 때.
    * 조회수가 `null`일 리가 없고, 기본적으로 `0`부터 시작하는 게 자연스러울 때.
    * 계산이 빈번하게 일어날 때.


* **`Long`을 쓰는 경우:**
    * **DB와 연동할 때 (Entity 클래스):** DB 테이블의 ID(PK)나 Null이 허용된 컬럼에는 `Long`을 쓰는 것이 관례입니다.
    * **제네릭이나 컬렉션:** `List<Long>`, `Map<String, Long>` 처럼 리스트나 맵에 담아야 할 때.
