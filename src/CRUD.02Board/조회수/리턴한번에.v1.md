##
`void`를 반환하는 메서드는 호출과 동시에 `return`할 수 없어서 흐름이 끊기는 느낌이 들 수 있죠. 이를 깔끔하게 해결하면서 **한 줄로 처리하는 세 가지 방법**을 소개해 드릴게요.

---

## 1. 체이닝(Chaining) 방식 (엔티티 수정)
엔티티의 메서드가 자기 자신(`this`)을 반환하도록 수정하는 방식입니다.

```java
// Board 엔티티 내부
public Board addViewCount() {
    this.viewCount++;
    return this; // void 대신 자기 자신을 반환
}

// Service 레이어
return BoardResponseDto.from(board.addViewCount());
```
* **장점:** 가장 직관적이고 코드가 짧아집니다.
* **참고:** Setter나 비즈니스 로직 메서드에서 `return this;`를 사용하는 것은 빌더 패턴과 유사한 방식이라 자주 쓰입니다.

## 2. 함수형 인터페이스 활용 (Peek)
만약 엔티티 코드를 수정하고 싶지 않다면, Java의 `Optional`에서 제공하는 기능을 활용할 수 있습니다.

```java
// Service 레이어
return boardRepository.findById(boardId)
        .map(board -> {
            board.addViewCount(); // 여기서 증가시키고
            return board;         // 다시 board를 반환
        })
        .map(BoardResponseDto::from)
        .orElseThrow(() -> new EntityNotFoundException("게시글 없음"));
```
* **장점:** 스트림 흐름을 깨지 않고 물 흐르듯 코드를 작성할 수 있습니다.

## 3. 메서드 내부에서 변환 처리
서비스의 가독성을 위해 엔티티나 DTO에 변환 로직을 위임하는 방식입니다.

```java
// BoardResponseDto 내부
public static BoardResponseDto fromWithIncrement(Board board) {
    board.addViewCount(); // 변환하기 전에 하나 올림
    return new BoardResponseDto(...);
}

// Service 레이어
return BoardResponseDto.fromWithIncrement(board);
```
* **주의:** 이 방식은 "조회(Read)를 하는데 데이터가 변경(Write)되는 부수 효과(Side Effect)"가 발생하므로, 메서드 이름을 `fromWithIncrement`처럼 명확하게 지어주는 것이 중요합니다.

---

### 어떤 것을 추천하나요?

가장 추천하는 방식은 **1번(Chaining)**입니다. 

수학적인 함수 호출 관점에서 보면 `f(g(x))` 형태가 되어 훨씬 깔끔하죠. `bitcopark-base` 프로젝트의 코드 스타일이 간결한 것을 선호하신다면 엔티티의 반환 타입을 `Board`로 바꾸는 것만으로도 충분히 만족스러운 코드가 나올 것 같습니다.

혹시 엔티티를 수정하는 방식이 프로젝트의 설계 원칙에 어긋난다고 생각하시나요? 아니면 위 방법 중 하나를 바로 적용해 보시겠어요?
