## [다른 관점] 정합성과 성능을 모두 고려한 접근법

### 1. 벌크 연산을 통한 동시성 제어 (`@Modifying`)
JPA의 변경 감지(Dirty Checking)는 엔티티를 조회한 시점의 값을 기준으로 업데이트하기 때문에, 동시에 여러 요청이 오면 '수정 손실(Lost Update)'이 발생할 수 있습니다. 이를 방지하기 위해 DB 레벨에서 직접 연산하는 방식을 사용합니다.

```java
// BoardRepository
@Modifying
@Query("update Board b set b.viewCount = b.viewCount + 1 where b.id = :id")
int updateViewCount(@Param("id") Long id);
```

* **장점:** `update ... set count = count + 1` 쿼리가 나가기 때문에 여러 명의 사용자가 동시에 클릭해도 DB가 순차적으로 하나씩 더해줍니다. 데이터 유실이 없습니다.
* **단점:** 영속성 컨텍스트를 거치지 않고 DB를 직접 수정하므로, 수정 후 해당 엔티티를 다시 조회할 때 `clearAutomatically = true` 같은 설정이 없으면 DB와 객체 간 데이터 불일치가 발생할 수 있습니다.

### 2. 하이브리드 방식 (Redis 활용)
조회수가 폭증하는 서비스(예: 인기 커뮤니티, 뉴스)에서는 조회 시마다 DB를 업데이트하는 것이 성능에 치명적입니다.

* **구조:** 조회 발생 시 **Redis**의 `INCR` 명령어로 카운트를 올리고, 일정 시간마다(예: 10분) 스케줄러를 통해 DB에 한꺼번에 반영(Batch Update)합니다.
* **장점:** DB 쓰기 부하를 획기적으로 줄일 수 있고, 속도가 매우 빠릅니다.
