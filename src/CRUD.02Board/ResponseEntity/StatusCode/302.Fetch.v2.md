핵심은 **"누가 요청을 보냈고, 그 응답을 누가 처리하느냐"**의 차이입니다. 서버가 보내주는 302 응답과 내용물(Location 헤더)은 똑같지만, 그걸 받는 **브라우저의 태도**가 달라집니다.

---

### 1. 일반적인 요청 (브라우저 주소창, `<form>` 태그)
사용자가 주소창에 직접 입력하거나 HTML 폼을 전송하는 경우입니다.

* **동작**: 브라우저가 **'페이지 전체'**를 바꿀 준비를 하고 요청을 보냅니다.
* **302 발생 시**: 서버가 `Location: /login`을 주면, 브라우저는 주소창의 주소를 즉시 `/login`으로 바꾸고 **화면 전체를 새로고침**하여 이동합니다.
* **사용자 경험**: 화면이 번쩍하며 로그인 페이지로 바뀝니다.

### 2. 비동기 요청 (fetch, axios, useFetch)
자바스크립트 코드 안에서 데이터를 가져오려고 요청을 보내는 경우입니다.

* **동작**: 브라우저는 **'데이터만'** 몰래 가져오려고 배경(Background)에서 요청을 보냅니다.
* **302 발생 시**: 
    1.  서버가 `Location: /login`을 줍니다.
    2.  브라우저는 자바스크립트 몰래 `/login`에 다시 요청을 보내서 그 페이지의 **HTML 내용물**을 일단 받아옵니다.
    3.  그리고 자바스크립트(`fetch`)에게 **"자, 여기 네가 요청한 결과(HTML)야"**라고 던져줍니다.
* **결과**: 자바스크립트 변수(`boards` 등)에는 게시글 리스트 대신 **로그인 페이지의 HTML 코드**가 담기게 됩니다. 하지만 자바스크립트가 "주소를 바꿔라"라는 명령을 내리지 않았으므로 **브라우저 주소창과 화면은 그대로 유지**됩니다.

---

### 3. 내용물이 어떻게 다른가요?

서버가 최종적으로 주는 내용물은 결국 **로그인 페이지의 HTML**로 동일합니다. 하지만 이 내용물을 어디에 쓰느냐가 다릅니다.

| 요청 방식 | 최종적으로 받는 것 | 브라우저의 행동 | 결과 |
| :--- | :--- | :--- | :--- |
| **주소창 입력** | 로그인 페이지 HTML | **화면 전체를 이 HTML로 렌더링** | 페이지 이동 성공 |
| **fetch 요청** | 로그인 페이지 HTML | **변수에 이 HTML 텍스트를 저장** | 화면은 그대로, 데이터 파싱 에러 발생 |



---

### 💡 이해를 돕기 위한 비유

* **일반 요청**: 당신이 직접 식당(서버)에 갔는데, 식당이 이사(302)를 갔습니다. 당신은 **몸을 움직여서** 새 식당으로 가서 밥을 먹습니다. (위치 이동)
* **fetch 요청**: 당신은 집에 있고, **배달원(fetch)**을 식당에 보냈습니다. 식당이 이사를 갔으면 배달원은 새 식당에 가서 **음식(HTML)**을 받아옵니다. 배달원이 음식을 들고 당신 집에 돌아왔지만, **당신의 집 주소(브라우저 주소창)**가 바뀌지는 않습니다. 당신은 스테이크(JSON)를 기다렸는데 배달원이 비빔밥(HTML)을 가져오니 "이게 뭐야?" 하고 에러가 나는 상황입니다.

### 그래서 어떻게 해야 하나요?
지금처럼 Nuxt/React 같은 프론트엔드 프레임워크를 쓰실 때는 서버가 302를 뱉게 하면 안 됩니다.

1.  **권한이 없는 경우**: 서버가 302(이사 가!)가 아니라 **401 Unauthorized(너 누구야?)** 에러를 주도록 설정해야 합니다.
2.  **데이터가 필요한 경우**: 게시글 목록처럼 누구나 봐야 하는 건 **`permitAll()`**로 시큐리티 검문을 해제해야 합니다.
