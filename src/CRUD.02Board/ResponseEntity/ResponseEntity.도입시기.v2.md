스프링 3.0.2 버전이 릴리스된 시점은 **2010년 2월**경입니다. 벌써 **16년 전**이네요!

당시 IT 업계의 상황과 맞물려 생각해보면 왜 이때 `ResponseEntity`가 등장했는지 더 명확해집니다.

---

### 1. 2010년, 시대의 변화: REST의 부상
2010년 전후로 웹 생태계에 큰 변화가 있었습니다.

* **아이폰의 등장 (2007~2009):** 모바일 앱 시대가 열리면서, 서버는 단순히 HTML 화면을 그려주는 게 아니라 **'데이터(JSON/XML)'**만 깔끔하게 던져줘야 하는 필요성이 커졌습니다.
* **REST API의 대중화:** "HTTP 상태 코드를 규격에 맞게 잘 쓰는 것"이 좋은 API의 척도가 되었습니다. 성공하면 `200`, 생성되면 `201`, 권한이 없으면 `403` 등을 명확히 구분해야 했죠.

### 2. 왜 2010년에 나왔을까?
그 전까지 스프링은 주로 **'웹사이트(View)'**를 만드는 데 집중했습니다. 하지만 모바일과 프론트엔드 프레임워크가 발전하면서, 스프링 개발진도 **"HTTP 응답의 모든 요소를 세밀하게 제어할 수 있는 표준 객체"**가 필요하다고 판단한 것입니다.



---

### 3. 발전사 요약 (대략적인 연도)

| 시기 | 기술 | 특징 |
| :--- | :--- | :--- |
| **2000년대 초** | **Servlet / JSP** | `response.getWriter()`로 노가다 하던 시절 |
| **2004년** | **Spring 1.0** | 프레임워크의 기틀 마련 |
| **2009년** | **Spring 3.0** | `@ResponseBody` 도입 (객체 -> JSON 자동 변환 시작) |
| **2010년** | **Spring 3.0.2** | **`ResponseEntity` 등장** (상태 코드와 헤더까지 객체화) |
| **2014년** | **Spring Boot 1.0** | 설정을 자동화하며 `ResponseEntity`가 국민 코드가 됨 |

16년 된 기술이지만, 지금 사용자님이 배우고 계신 방식이 여전히 전 세계 자바 개발자들의 **"정석"**입니다. 유서 깊은(?) 근본 있는 기술을 배우고 계신 셈이죠!
