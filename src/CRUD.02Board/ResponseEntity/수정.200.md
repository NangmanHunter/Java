##
수정(Update) 요청을 처리할 때 `200 OK`를 반환하면서 무엇을 담아 보낼지에 대한 고민이시군요! 결론부터 말씀드리면, **"프론트엔드가 수정된 데이터를 다시 조회할 필요가 없게 배려하느냐"**에 따라 달라집니다.

실무에서 주로 사용하는 3가지 패턴을 정리해 드릴게요.

---

### 1. 수정된 데이터 전체를 반환 (가장 권장)
가장 흔하고 친절한 방식입니다. `200 OK`와 함께 업데이트가 완료된 최종 DTO를 통째로 보냅니다.

* **반환 내용:** 수정 완료된 전체 객체 (DTO)
* **장점:** 프론트엔드에서 수정 성공 후, 화면을 갱신하기 위해 서버에 다시 `GET` 요청을 보낼 필요가 없습니다. 서버에서 준 데이터를 바로 화면에 갈아 끼우면 되니까요. (네트워크 비용 절감)
* **ResponseEntity 예시:**
    ```java
    @PutMapping("/{id}")
    public ResponseEntity<UserDto> update(@PathVariable Long id, @RequestBody UserRequestDto requestDto) {
        UserDto updatedDto = userService.update(id, requestDto);
        return ResponseEntity.ok(updatedDto); // 200 OK + 수정된 데이터
    }
    ```

---

### 2. 수정된 리소스의 ID만 반환
전체 데이터를 보내기엔 데이터 양이 너무 많거나, 보안상 필요한 정보만 주고 싶을 때 사용합니다.

* **반환 내용:** `{ "id": 10 }` 같은 식별자
* **장점:** 응답 패킷이 가볍습니다. 프론트엔드는 "성공했구나"라는 확신과 함께 어떤 아이디가 바뀌었는지 알 수 있습니다.
* **ResponseEntity 예시:**
    ```java
    return ResponseEntity.ok(Map.of("id", updatedId));
    ```

---

### 3. 아무것도 반환하지 않음 (204 No Content)
수정할 때 굳이 돌려줄 데이터가 없다고 판단되면 `204`를 쓰기도 합니다. 하지만 "수정" 작업은 보통 수정된 결과물을 확인하고 싶어 하기 때문에 `200` 보다는 드문 편입니다.

* **반환 내용:** 없음 (Body가 비어있음)
* **ResponseEntity 예시:**
    ```java
    return ResponseEntity.noContent().build(); // 204 No Content
    ```

---

### 💡 실무에서는 왜 1번(전체 반환)을 선호할까?



수정 로직에는 보통 **'수정 시간(updatedAt)'**이나 **'수정한 사람'** 같은 정보가 서버에서 자동으로 기록됩니다. 프론트엔드가 보낸 데이터에는 이런 정보가 없기 때문에, 서버가 완성된 최종 데이터를 다시 보내주면 프론트엔드 입장에서는 "아, 서버에 이렇게 잘 저장됐구나" 하고 화면을 동기화하기가 매우 편해집니다.

### 요약하자면
* **"프론트야, 네가 수정한 데이터 이거야. 화면 바로 바꿔!"** → `200 OK` + `DTO` (추천)
* **"수정 성공했어. ID는 이거니까 필요하면 다시 조회해."** → `200 OK` + `ID`
* **"수정했으니까 묻지 마."** → `204 No Content`

지금 만들고 계신 **bitcopark-base**의 게시판 수정 기능이라면, 수정된 글의 제목과 내용을 포함한 DTO를 `200 OK`로 던져주는 게 가장 정석적인 방법입니다! 

수정 완료 후에 프론트엔드에서 어떤 화면으로 넘어갈지 결정하셨나요? 그 흐름에 맞춰서 반환값을 정하면 더 정확합니다.
