맞습니다! 실무에서 게시글 수정을 마친 뒤에 **해당 게시글의 상세 페이지로 바로 이동(Redirect)**시키는 기획이라면, 말씀하신 대로 **ID만 던지는 방식**을 정말 많이 사용합니다.

이 흐름이 왜 효율적인지 프론트엔드와 백엔드 사이의 티키타카로 설명해 드릴게요.

---

### 1. 왜 ID만 던지는 구조를 쓸까?

게시글 수정이 완료되면 보통 유저는 "자, 내가 고친 글이 잘 올라갔나?" 하고 확인하고 싶어 합니다. 이때 프론트엔드는 두 가지 선택을 할 수 있죠.

* **방식 A:** 서버가 준 DTO로 현재 화면만 살짝 바꾼다. (페이지 이동 없음)
* **방식 B:** 서버가 준 ID를 받아서 `.../view/{id}` 페이지로 **화면을 완전히 이동(Redirect)**시킨다.

**방식 B**를 선택할 경우, 어차피 상세 페이지로 이동하면서 새로운 `GET` 요청을 서버에 날려 최신 데이터를 다시 받아오게 됩니다. 그러니 수정 API(`PUT/PATCH`)에서 굳이 무거운 DTO를 다 보낼 필요 없이 **"성공했어! ID는 이거니까 이리로 가!"**라고 식별자만 주는 것이 네트워크 자원 면에서 이득입니다.

---

### 2. ResponseEntity로 ID만 던지는 코드 예시

단순히 숫자만 보내기보다는 JSON 형식으로 `{ "id": 10 }` 처럼 보내주는 것이 관례입니다.

```java
@PatchMapping("/{id}")
public ResponseEntity<Map<String, Long>> updatePost(@PathVariable Long id, @RequestBody PostRequestDto requestDto) {
    // 1. 비즈니스 로직 수행 (DB 수정)
    Long updatedId = postService.update(id, requestDto);
    
    // 2. ID만 담아서 200 OK 응답
    return ResponseEntity.ok(Map.of("id", updatedId));
}
```

### 3. 실무적인 한 단계 더 나아간 팁: `Location` 헤더

REST API의 정석을 따지는 곳에서는 `201 Created`나 `200 OK`를 주면서 응답 헤더의 `Location`이라는 곳에 **"새로 바뀐 리소스의 주소"**를 적어주기도 합니다.

* **Status:** `200 OK` (또는 상황에 따라 201)
* **Header:** `Location: /api/posts/15`
* **Body:** `{ "id": 15 }`

이렇게 주면 프론트엔드는 바디를 뜯지 않고도 헤더에 있는 주소로 바로 유저를 보내버릴 수 있어서 매우 편리해합니다.
