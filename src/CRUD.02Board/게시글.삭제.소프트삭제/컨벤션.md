아주 예리한 질문입니다! 전공이 물리와 수학이시라 그런지 이런 **네이밍 규칙과 매핑 로직**에 민감하신 게 느껴지네요.

결론부터 말씀드리면, **Spring Boot와 JPA(Hibernate)의 기본 설정** 덕분에 Java의 `isDeleted`와 DB의 `is_deleted`는 **자동으로 연결(매핑)**됩니다.

---

### 1. CamelCase vs SnakeCase
Java와 DB는 서로 선호하는 이름 표기법이 다릅니다.
* **Java:** `isDeleted` (CamelCase)
* **DB:** `is_deleted` (Snake_case)

Spring Boot는 이 차이를 해결하기 위해 `CamelCaseToUnderscoresNamingStrategy`라는 것을 기본으로 사용합니다. 즉, 자바 필드에 대문자가 있으면 DB로 보낼 때 **"언더바(`_`) + 소문자"**로 알아서 바꿔줍니다.



---

### 2. `@SQLDelete`와 `@Where` 사용 시 주의점
위에서 설명드린 자동 매핑은 보통의 `SELECT`, `INSERT` 시에 적용됩니다. 하지만 **`@SQLDelete`나 `@Where` 안에 직접 쿼리를 적을 때는 "DB의 실제 컬럼명"을 적어주어야 합니다.**

```java
@SQLDelete(sql = "UPDATE board SET is_deleted = true WHERE id = ?") // ✅ DB 컬럼명인 is_deleted 사용
@Where(clause = "is_deleted = false") // ✅ DB 컬럼명인 is_deleted 사용
public class Board {
    private boolean isDeleted; // ✅ Java 필드명은 CamelCase
}
```

### 3. Boolean 필드 네이밍의 관습 (중요!)
자바에서 `boolean` 타입의 변수명을 `isDeleted`라고 지으면, 나중에 롬복(`@Getter`)이나 자바 표준에 의해 생성되는 메서드 이름이 **`isDeleted()`**가 됩니다. 

가끔 어떤 라이브러리들은 `is`가 중복되는 것을 피하려고 `deleted`라는 필드명을 권장하기도 합니다.
* 필드명이 `deleted` (boolean) → Getter 이름: `isDeleted()`
* 필드명이 `isDeleted` (boolean) → Getter 이름: `isDeleted()` (똑같음)

그래서 DB 컬럼명을 명시적으로 맞추고 싶다면 아래처럼 적어주는 것이 가장 확실하고 안전합니다.

```java
@Column(name = "is_deleted") // DB 컬럼명을 명시적으로 지정
private boolean isDeleted = false;
```

---

### 💡 요약
1.  **필드명(`isDeleted`)**과 **DB 컬럼명(`is_deleted`)**은 스프링이 알아서 연결해준다.
2.  단, `@SQLDelete`나 `@Where` 처럼 **직접 SQL을 쓰는 곳에서는 반드시 DB 컬럼명(`is_deleted`)**을 써야 한다.
3.  불안하면 `@Column(name = "is_deleted")`를 붙여주는 게 가장 "Nangman" 있는 확실한 방법이다!

이제 DB 테이블을 확인해보시면 `is_deleted`라는 이름으로 예쁘게 생성되어 있을 거예요. :)
