과거의 기억이 아주 정확하십니다! XML 데이터를 그대로 가져와서 HTML에 '박아 넣는' 것은 불가능했습니다. HTML 입장에서는 XML 태그들(`<name>...</name>` 같은)이 정체불명의 외계어처럼 보였을 테니까요.

그 시절에 XML을 화면에 그리기 위해 겪었던 **'고난의 역사'**를 알면 왜 지금의 JSON이 천국인지 바로 이해되실 겁니다.

---

### 1. XML 시절: "그려지지 않는다, 파싱(Parsing)해야 한다"
XML은 그저 **데이터 뭉치**일 뿐입니다. 브라우저가 이걸 받으면 다음과 같은 복잡한 과정을 거쳐야 했습니다.

* **DOMParser 사용:** 자바스크립트로 XML 문자열을 받아서 "이건 XML이야"라고 선언하고 트리 구조로 변환합니다.
* **노드 찾기:** `getElementsByTagName("name")[0].childNodes[0].nodeValue` 같은 끔찍하게 긴 코드로 데이터를 하나하나 캐내야 했습니다.
* **XSLT 사용:** 아예 XML을 HTML로 변환해주는 별도의 언어(XSLT)를 공부해서 서버나 브라우저에서 변환 작업을 거치기도 했습니다.

즉, **그냥 가져온다고 그려지는 게 아니라, 개발자가 노가다로 데이터를 발라내서 HTML 태그 안에 일일이 넣어줘야 했습니다.**

---

### 2. 반면, 지금의 JSON은?
질문자님이 지금 하고 계신 방식이죠.

```javascript
// fetch로 받은 데이터가 이미 자바스크립트 객체!
const data = await response.json(); 
document.getElementById('name').innerText = data.name; // 끝!
```

자바스크립트 입장에서는 JSON이 **'자기 식구'**이기 때문에, 별도의 복잡한 파싱 과정 없이 점(`.`) 하나로 데이터에 접근할 수 있습니다. XML 시절에 비하면 거의 마법에 가깝죠.

---

### 3. "그려지나?"에 대한 예외 상황
혹시 예전에 XML이 브라우저에 **그대로 출력**되는 걸 보신 적이 있다면, 그건 HTML에 '그려진' 게 아니라 **브라우저가 XML 파일을 읽어서 보여준 것**일 가능성이 큽니다.

* 브라우저 주소창에 `.xml` 파일 주소를 치면, 브라우저는 친절하게도 트리 구조(접었다 펴기 가능)로 XML 내용을 보여줍니다.
* 하지만 이건 우리가 원하는 **웹 서비스의 디자인**이 입혀진 화면이 아니라, 그냥 데이터 구조를 시각화해서 보여준 것뿐입니다.

### 💡 정리하자면
XML을 가져와서 HTML에 그대로 박아 넣으면 브라우저는 그걸 화면에 그리지 못하고 무시하거나, 텍스트로만 띄워버립니다.
