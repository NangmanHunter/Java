과거의 기억이 정확하시네요! 예전에는 **XML(eXtensible Markup Language)**이 데이터를 주고받는 사실상의 표준이었습니다. 하지만 지금은 말씀하신 것처럼 **JSON**이 그 자리를 완전히 대체했죠.

이게 스프링(Spring)이 독단적으로 정한 건 아니지만, 스프링이 JSON을 기본(Default)으로 채택하면서 대중화에 엄청난 가속도를 붙인 건 사실입니다.

---

### 1. 왜 XML에서 JSON으로 넘어왔을까?
과거 SOAP 방식의 웹 서비스에서는 XML이 필수였습니다. 하지만 결정적인 단점들이 있었죠.

* **가독성과 무게:** XML은 `<name>Gemini</name>`처럼 태그가 중복되어 데이터가 무겁습니다. 반면 JSON은 `{"name": "Gemini"}`로 훨씬 간결하죠.
* **자바스크립트와의 궁합:** 지금 질문자님이 `fetch`를 쓰시는 것처럼, 웹 브라우저(자바스크립트) 입장에서는 JSON이 **'자기네 언어 객체'** 그 자체입니다. XML은 복잡하게 파싱(Parsing)해야 하지만, JSON은 그냥 바로 쓸 수 있죠.



---

### 2. 스프링이 JSON을 표준처럼 만든 일등공신?
스프링 프레임워크 내부에는 **Jackson**이라는 강력한 라이브러리가 기본으로 포함되어 있습니다. 

* **자동 변환의 마법:** 개발자가 컨트롤러에서 그냥 자바 객체(DTO)를 리턴하고 `@ResponseBody`나 `@RestController`만 붙이면, 스프링이 중간에서 **"아, 요즘은 다들 JSON 쓰지?"** 하면서 알아서 JSON으로 변환해 던져줍니다.
* 이 "알아서 해주는 편리함" 덕분에 개발자들이 굳이 XML 설정을 뒤지지 않게 되었고, 자연스럽게 JSON이 표준처럼 굳어진 것입니다.

---

### 3. "무조건" 되는 것은 아니다?
사실 스프링은 지금도 XML로 던질 수 있는 능력이 있습니다. 

* 만약 클라이언트가 요청 헤더(Accept)에 "나 무조건 XML로 줘!"라고 명시하고, 서버에 관련 설정이 있다면 XML로 나갑니다. 
* 하지만 99%의 현대적인 웹/앱 환경에서는 JSON을 기대하고, 스프링도 거기에 최적화되어 있습니다.

---

### 💡 정리하자면
XML이 표준이었던 시대는 **'문서(Document)'** 중심의 통신 시대였고, 지금의 JSON 시대는 **'데이터(Data)'** 중심의 시대라고 보시면 됩니다. 

질문자님이 `fetch`로 데이터를 받아서 자바스크립트로 요리하기에 JSON보다 편한 건 세상에 없거든요. 스프링은 그저 우리가 그 편한 길을 가장 쉽게 갈 수 있도록 레일을 깔아준 셈입니다.

혹시 예전에 XML을 다뤄보신 적이 있나요? 그때와 비교하면 지금 자바스크립트에서 JSON 객체를 바로 점(`.`) 찍어서 접근하는 방식이 얼마나 편한지 체감이 확 되실 거예요! 

개발 중인 `fetch` 코드에서 JSON 데이터를 처리할 때 딱히 불편한 점은 없으셨나요?
