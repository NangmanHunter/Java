##
- CSR.바로쏜다.클라이언트에서 백엔드로.`클라이언트→백엔드`
```
클라이언트 → 프론트엔드서버
          ↙
클라이언트 → 백엔드서버
```


##
CSR(Client Side Rendering)의 핵심은 **"화면을 그리는 주도권이 브라우저(클라이언트)에게 있다"**는 점입니다. 질문하신 것처럼 클라이언트가 백엔드로 직접 데이터를 요청하는 과정이 포함되는데, 그 전체적인 흐름을 단계별로 짚어볼게요.

---

### 1. CSR의 기본 흐름 (동작 방식)

CSR은 처음에 서버로부터 '텅 빈' HTML 파일과 아주 큰 자바스크립트(JS) 파일을 받으면서 시작됩니다.

1.  **초기 요청:** 사용자가 주소창에 URL을 입력하면 브라우저는 서버에 접속합니다.
2.  **빈 껍데기 다운로드:** 서버는 내용이 거의 없는 HTML(`div id="root"` 하나만 딸랑 있는 정도)과 JS 파일을 보냅니다. (이 시점에는 화면에 아무것도 안 보입니다.)
3.  **자바스크립트 실행:** 브라우저가 JS 파일을 한땀 한땀 읽으면서 React, Vue 같은 프레임워크를 구동합니다.
4.  **데이터 요청 (질문하신 부분!):** JS가 실행되면서 **"아, 화면에 뿌릴 데이터가 필요하네?"**라고 판단하고, 브라우저가 직접 **백엔드 API 서버**로 데이터를 요청합니다.
5.  **화면 렌더링:** 백엔드에서 받은 데이터(주로 JSON 형식)를 JS가 HTML 태그 속에 쏙쏙 집어넣어 화면을 완성합니다.



---

### 2. "클라이언트에서 바로 백엔드로 쏘는 건가요?"

**네, 맞습니다!** 정확히는 브라우저(User Agent) 안에서 실행되는 **자바스크립트(fetch나 axios 등)**가 백엔드 서버의 API 주소로 직접 HTTP 요청을 보냅니다.

여기서 중요한 구조적 차이가 발생합니다.

*   **웹 서버 (Static Server):** HTML, JS, CSS 같은 '파일'만 던져주는 곳 (예: S3, Vercel, Nginx).
*   **백엔드 서버 (API Server):** DB에 접속해서 진짜 '데이터'만 던져주는 곳 (예: Spring, Node.js, Django).

사용자의 브라우저는 처음에 웹 서버에서 소스 코드를 받아온 뒤, 그 소스 코드를 실행하면서 **백엔드 서버와는 직접 대화**하게 됩니다.

---

### 3. 왜 이렇게 하나요? (장단점)

이 방식은 마치 **"식당에 가서 요리법(JS)을 받아와서, 내 테이블에서 직접 재료(데이터)만 주문해 요리해 먹는 것"**과 비슷합니다.

| 장점 | 단점 |
| :--- | :--- |
| **빠른 인터랙션:** 첫 로딩 후에는 페이지 이동 시 화면 전체를 새로고침하지 않아 매우 부드럽습니다. | **느린 초기 로딩:** 처음에 커다란 JS 파일을 다 받아야 해서 첫 화면이 뜨기까지 시간이 걸립니다. |
| **서버 부하 감소:** 화면을 그리는 연산을 사용자의 컴퓨터가 대신해주므로 서버 비용이 절감됩니다. | **SEO(검색 엔진 최적화) 취약:** 검색 로봇이 빈 HTML만 보고 "내용이 없네?" 하고 돌아갈 수 있습니다. |

---

### 4. 정리
CSR에서 클라이언트는 단순히 화면을 보여주는 창구가 아니라, **스스로 데이터를 요청하고 화면을 조립하는 공장** 역할을 합니다. 그래서 "클라이언트에서 백엔드로 바로 쏜다"는 표현은 매우 정확한 이해입니다!

혹시 이 과정에서 보안이나 `CORS`(다른 주소로 요청을 보낼 때 발생하는 문제) 같은 이슈에 대해서도 들어보셨나요?
