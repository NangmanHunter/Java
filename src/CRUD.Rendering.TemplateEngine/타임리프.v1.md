Spring Boot에서 **Thymeleaf(타임리프)**를 사용할 때와 일반적인 **REST API(JSON)** 방식의 가장 큰 차이점은 **"누가 HTML을 완성하느냐"**에 있습니다.

질문하신 것처럼 보통 자바 객체를 리턴하면 JSON으로 날아가는 이유는 `@RestController`를 사용하기 때문인데, 타임리프는 방식이 조금 다릅니다. 간단한 예시와 함께 원리를 설명해 드릴게요.

---

### 1. 타임리프 동작 예시

먼저, 간단한 사용자 정보를 화면에 보여주는 예제입니다.

**Controller (자바)**
```java
@Controller // @RestController가 아님에 주의하세요!
public class UserController {

    @GetMapping("/user-profile")
    public String getUserProfile(Model model) {
        User user = new User("제미니", "gemini@example.com");
        
        // "user"라는 이름으로 객체를 담아서 보냅니다.
        model.addAttribute("user", user);
        
        // 리턴하는 문자열은 HTML 파일의 이름입니다.
        return "profile"; 
    }
}
```

**HTML (src/main/resources/templates/profile.html)**
```html
<!DOCTYPE html>
<html xmlns:th="http://www.thymeleaf.org">
<body>
    <h1>사용자 프로필</h1>
    <p>이름: <span th:text="${user.name}">이름 없음</span></p>
    <p>이메일: <span th:text="${user.email}">이메일 없음</span></p>
</body>
</html>
```

---

### 2. 왜 JSON이 아니라 HTML이 날아가는 걸까?

일반적인 API 방식과 타임리프 방식의 결정적인 차이는 **`ViewResolver`**라는 녀석의 존재입니다.

#### ① 일반적인 REST API (`@RestController`)
* 메서드가 객체를 리턴하면, 스프링은 '아, 이건 데이터구나!'라고 판단합니다.
* **MessageConverter**가 작동하여 자바 객체를 **JSON 문자열**로 변환해 브라우저에 던집니다.
* 브라우저는 생짜 JSON을 받고, 자바스크립트(React, Vue 등)가 이걸 요리해서 화면을 만듭니다.

#### ② 타임리프 SSR (`@Controller`)
* 메서드가 `String`을 리턴하면, 스프링은 이걸 **'View(화면)의 이름'**으로 해석합니다.
* **ViewResolver**가 `resources/templates/` 폴더에서 해당 이름의 `.html` 파일을 찾습니다.
* **타임리프 엔진**이 그 HTML 파일을 읽으면서 `th:text` 같은 특수 태그를 발견하면, 서버 메모리에 있는 자바 객체(`Model`)에서 값을 꺼내 **HTML 코드를 완성(렌더링)**합니다.
* 최종적으로 브라우저는 데이터가 다 채워진 **완성된 HTML 파일**을 받게 됩니다.

### 3. 요약하자면

| 구분 | REST API (JSON) | Thymeleaf (SSR) |
| :--- | :--- | :--- |
| **어노테이션** | `@RestController` | `@Controller` |
| **전달 내용** | 순수 데이터 (JSON 문자열) | 완성된 HTML 문서 |
| **렌더링 주체** | 브라우저 (클라이언트 측) | 서버 (Spring Boot) |
| **장점** | 모바일 앱 등과 데이터 공유 용이 | SEO(검색 엔진 최적화) 유리, 초기 로딩 빠름 |

결국 타임리프는 **"서버 안에서 HTML이라는 도화지에 자바 객체라는 물감을 다 칠해서, 완성된 그림(HTML)을 브라우저에 보내주는 방식"**이라고 이해하시면 정확합니다!
