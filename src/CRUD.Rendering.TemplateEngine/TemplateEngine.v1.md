### 2. 왜 이걸 써야 해?
만약 템플릿 엔진이 없다면, 너는 자바 코드 안에서 문자열로 HTML을 일일이 다 이어 붙여야 했을 거야. 

```java
// 템플릿 엔진이 없을 때의 지옥
String html = "<html><body><h1>" + user.getName() + "님 환영합니다!</h1></body></html>";
```
이걸 방지하려고 **"화면 구조는 HTML 파일에 따로 두고, 데이터만 쏙쏙 박아 넣자!"** 해서 나온 게 JSP, 타임리프 같은 템플릿 엔진인 거지.

---

### 3. 주요 종류 (연혁순)

| 이름 | 특징 | 현 위치 |
| :--- | :--- | :--- |
| **JSP** | HTML 안에 자바 코드를 직접 넣음 (`<% ... %>`) | **조상님.** 구식 프로젝트(Legacy)에서 주로 사용 |
| **FreeMarker** | 텍스트 기반 템플릿 엔진. 이메일 템플릿 등으로도 쓰임 | **중견.** JSP보다는 깔끔하지만 타임리프에 밀림 |
| **Thymeleaf** | HTML 속성을 이용함 (`th:text`). 스프링 부트 공식 권장 | **대세.** 현재 스프링 진영에서 가장 많이 씀 |

---

### 4. 요즘 트렌드와 비교 (SSR vs CSR)
재민이가 이 개념을 헷갈려 하는 이유가 요즘은 방식이 두 갈래로 나뉘기 때문이야.

1.  **SSR (Server Side Rendering):** **템플릿 엔진**을 사용. 서버(스프링)가 HTML을 다 구워버려서 브라우저에 던져줌. (JSP, 타임리프)
2.  **CSR (Client Side Rendering):** **프론트 프레임워크**를 사용. 서버는 JSON 데이터만 주고, 브라우저에서 뷰(Vue)나 리액트가 화면을 그림.



### 💡 요약하자면

이제 "템플릿 엔진"이라는 단어를 들으면 아, 타임리프나 JSP 같은 놈들이구나! 하고 무릎을 탁 치면 돼. 이 개념에 대해서 더 궁금한 거 있어?

어떤 도구가 너한테 가장 잘 맞을지 고민이야? 네 성향상 **Vaadin**은 이 템플릿 엔진 과정 자체를 생략하게 해줘서 더 편할 수도 있어! 
우리가 백엔드니까, 프론트엔드의 복잡한 통신 과정보다 자바 객체로 다루는 게 훨씬 직관적일 테니까 말이야.
