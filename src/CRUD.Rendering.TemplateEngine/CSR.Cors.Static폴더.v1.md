##
이 방식은 쉽게 말해 **"백엔드 서버가 프론트엔드 파일까지 통째로 입양해서 대신 배포해 주는 방식"**입니다.

보통 스프링 프로젝트 구조를 보면 `src/main/resources/static`이라는 폴더가 있죠? 이 폴더에 프론트엔드 결과물을 집어넣는 과정을 구체적으로 설명해 드릴게요.

---

### 1. "폴더를 던져준다"는 게 구체적으로 무슨 뜻인가요?

프론트엔드(React, Vue 등) 개발이 끝나면 `build` 또는 `dist`라는 폴더가 생성됩니다. 이 안에는 브라우저가 읽을 수 있는 **최종본 파일(index.html, main.js, style.css 등)**들이 들어있습니다.

이 파일들을 **스프링 프로젝트의 특정 폴더로 복사**해서 넣는 것을 의미합니다.

* **위치:** `src/main/resources/static/` (스프링 부트의 기본 정적 자원 경로)
* **결과:** 이제 스프링 서버(`:8080`)를 실행하고 브라우저에서 `localhost:8080`에 접속하면, 스프링이 자기 주머니에 있던 `index.html`을 꺼내서 보여줍니다.



---

### 2. 왜 이렇게 하면 CORS가 안 생기나요?

CORS는 **주소(도메인, 포트)가 다를 때** 발생한다고 했죠?

* **분리된 경우:** 프론트(`:3000`) → 백엔드(`:8080`) 요청 시 **포트가 달라서** CORS 발생.
* **합친 경우:** 브라우저가 HTML을 가져온 곳도 `:8080`이고, 데이터를 요청하는 곳도 `:8080`입니다. 브라우저 입장에서는 **"어? 같은 집이네!"**라고 판단하므로 아무런 제약을 걸지 않습니다.

---

### 3. 이 방식의 장점과 단점

#### 장점 (심플함)
* **서버가 1대면 끝:** 서버 비용이 절감되고 관리가 편합니다.
* **CORS 설정 불필요:** 보안 설정을 따로 만질 필요가 없어 개발 초기에 매우 편합니다.

#### 단점 (규모가 커지면 힘든 이유)
* **빌드 시간이 길어짐:** 백엔드 코드 한 줄만 고쳐도 프론트엔드까지 다시 빌드해서 합쳐야 할 때가 있습니다.
* **업무 간섭:** 프론트엔드 개발자가 수정한 내용을 확인하려면 매번 백엔드 서버를 재시작해야 할 수도 있습니다.
* **확장성 저하:** 나중에 사용자가 많아져서 "화면 서버만 늘리고 싶다"라고 해도, 백엔드 로직이 같이 붙어 있어서 불필요한 자원을 낭비하게 됩니다.

---

### 4. 비유하자면 이렇습니다.

* **서버 2대 방식 (프록시):** **'분식집'**과 **'카페'**가 따로 있는 구조입니다. 손님은 카페에 앉아있고, 카페 직원이 옆집 분식집에서 떡볶이를 배달해다 주는 식이죠.
* **합치는 방식 (Static):** **'푸드코트 한 칸'**에서 떡볶이(백엔드)와 커피(프론트)를 같이 파는 구조입니다. 주인 한 명이 다 관리하니 편하긴 하지만, 손님이 많아져서 커피만 빨리 팔고 싶어도 주방 전체를 키워야 하는 번거로움이 생깁니다.

**요약하자면:** "폴더를 던져준다"는 것은 프론트엔드가 만든 **최종 결과물 파일들을 스프링의 정적 자원 폴더로 옮겨서, 스프링 서버 하나가 화면과 데이터를 모두 책임지게 만든다**는 뜻입니다! 

주로 학습용 토이 프로젝트나 관리자 페이지처럼 규모가 작은 내부 시스템을 만들 때 아주 유용한 방식이에요. 지금 공부 중이신 환경이라면 한 번쯤 시도해 보기 가장 쉬운 방법이기도 합니다.
