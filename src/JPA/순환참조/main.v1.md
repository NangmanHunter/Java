JPA에서 **순환 참조(Circular Reference)**는 서로 다른 두 엔티티가 서로를 참조하고 있을 때, 이를 무한히 파고들다가 결국 서버가 뻗어버리는 현상을 말합니다.

가장 흔한 예시는 **게시글(Post)**과 **댓글(Comment)**의 관계입니다.

---

## 1. 순환 참조가 발생하는 메커니즘

1.  **엔티티 구조:**
    * `Post` 엔티티는 여러 개의 `Comment`를 리스트로 가지고 있습니다.
    * `Comment` 엔티티는 자신이 속한 `Post` 하나를 참조합니다.
2.  **문제 발생 (직렬화 단계):**
    * 컨트롤러에서 `Post` 엔티티를 JSON으로 변환하여 응답하려고 합니다.
    * JSON 변환기(Jackson)가 `Post`를 그리다가 `Comment` 리스트를 발견합니다.
    * `Comment`를 그리러 들어갔더니, 그 안에 다시 `Post`가 있습니다.
    * 다시 그 `Post` 안의 `Comment`로... 이 과정이 **무한 반복**됩니다.



---

## 2. 왜 서버가 터지나요?

이 무한 루프가 반복되면 결국 메모리가 꽉 차면서 **`StackOverflowError`**가 발생하며 서버가 강제 종료됩니다. 

수학적으로 보면 **재귀 함수(Recursive Function)**에 탈출 조건이 없는 것과 같습니다.
$$f(Post) \rightarrow f(Comment) \rightarrow f(Post) \rightarrow \dots$$

---

## 3. 해결 방법 (어떻게 끊어내나?)

이 고리를 끊는 방법은 크게 세 가지가 있습니다.

### ① 가장 권장되는 방법: DTO 사용
앞서 강조드린 것처럼, 엔티티를 직접 응답으로 보내지 않고 필요한 데이터만 담은 **DTO**를 만들면 참조 고리가 원천 봉쇄됩니다.
* `PostResponseDto`에는 댓글 리스트를 담되, 댓글 DTO 안에는 다시 게시글 정보를 담지 않는 식입니다.

### ② `@JsonIgnore` 사용
JSON으로 변환할 때 특정 필드를 무시하라고 명시하는 방법입니다.
```java
@Entity
public class Comment {
    @ManyToOne
    @JsonIgnore // JSON 변환 시 이 필드는 쳐다보지도 마!
    private Post post;
}
```
* **단점:** 어떤 API에서는 이 정보가 필요할 수도 있는데, 일률적으로 막아버려 유연성이 떨어집니다.

### ③ `@JsonManagedReference` & `@JsonBackReference`
부모(Post)와 자식(Comment) 관계를 명시해서, 자식 쪽에서 부모로 돌아가는 참조를 막아주는 Jackson 전용 어노테이션입니다.
* **부모 쪽:** `@JsonManagedReference` (정상적으로 직렬화)
* **자식 쪽:** `@JsonBackReference` (직렬화하지 않음)

---

## 4. 양방향 매핑의 딜레마

사실 순환 참조는 **'양방향 매핑'** 때문에 발생합니다. 

수학이나 물리 문제를 풀 때도 불필요한 변수가 많으면 식이 복잡해지듯, JPA에서도 객체 그래프를 양방향으로 다 연결해 놓으면 관리 포인트만 늘어납니다. 

> **팁:** 실무에서는 일단 **단방향**으로 설계를 시작하고, 정말로 반대 방향의 조회가 절실할 때만 양방향을 추가하는 것이 순환 참조를 예방하는 가장 좋은 습관입니다.

혹시 지금 작성하신 엔티티 구조에서 서로를 참조하고 있는 코드가 있나요? 그 구조를 보여주시면 어떤 방식으로 끊는 게 가장 깔끔할지 봐 드릴게요.
