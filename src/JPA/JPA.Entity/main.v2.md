맞습니다. 실무에서 **"엔티티를 외부에 노출하지 마라"** 혹은 **"API 응답으로 엔티티를 직접 던지지 마라"**는 말은 거의 금기사항처럼 여겨집니다.

단순히 귀찮아서 DTO를 만드는 게 아니라, 엔티티를 그대로 쓰면 시스템이 **'스파게티'**처럼 꼬여버릴 위험이 크기 때문인데요. 크게 3가지 이유로 정리해 드릴게요.

---

## 1. 엔티티는 "DB의 거울", DTO는 "화면의 요구사항"
엔티티는 데이터베이스 구조와 밀접하게 결합되어 있습니다. 반면 화면(UI)이나 API 요구사항은 수시로 변하죠.

* **꼬이는 상황:** 화면에서 "사용자 이름"만 보여주면 되는데, 엔티티를 그대로 보내면 "비밀번호", "주민번호" 같은 민감한 정보까지 딸려 나갑니다.
* **해결:** DTO를 쓰면 딱 필요한 데이터만 골라서(Filtering) 깔끔하게 보낼 수 있습니다.

## 2. 순환 참조와 무한 루프의 늪
엔티티는 보통 서로 연결되어 있습니다 (`Member`가 `Order`를 가지고, `Order`가 다시 `Member`를 가지는 식).

* **꼬이는 상황:** JSON으로 변환할 때, JPA 엔티티를 그대로 직렬화하면 `Member`를 그리다 `Order`를 부르고, `Order`를 그리다 다시 `Member`를 부르는 **무한 루프**에 빠져 서버가 뻗어버릴 수 있습니다.
* **해결:** DTO는 단순한 데이터 덩어리라 이런 관계의 사슬을 끊어줍니다.

## 3. 의도치 않은 DB 수정 (사이드 이펙트)
앞서 말씀드린 **'변경 감지(Dirty Checking)'** 기능 때문입니다.

* **꼬이는 상황:** 누군가 서비스 레이어나 컨트롤러에서 "그냥 화면에 예쁘게 보이려고" 엔티티의 값을 살짝 바꿨다고 칩시다. 그런데 이게 트랜잭션 안이라면? **JPA는 "어? 값이 바뀌었네?" 하고 DB에 `UPDATE` 쿼리를 날려버립니다.** 나는 그냥 조회만 하려 했을 뿐인데 DB 데이터가 변질되는 무서운 일이 벌어지죠.
* **해결:** DTO는 JPA의 관리 대상(영속성 컨텍스트)이 아니기 때문에, 안에서 값을 아무리 지지고 볶아도 DB에 영향을 주지 않아 안전합니다.

## 요약하자면

| 구분 | 엔티티 (Entity) | DTO |
| :--- | :--- | :--- |
| **목적** | DB 테이블 설계 및 비즈니스 로직 | 데이터 전달 (화면/API 맞춤형) |
| **영향도** | 변경 시 DB 스키마 등에 큰 영향 | 변경해도 화면만 바뀜 (안전) |
| **결합도** | 높음 (DB 의존적) | 낮음 (독립적) |

> **한 줄 정리:**
> 엔티티는 우리 집 안방(DB)의 설계도이고, DTO는 손님에게 보여주는 모델하우스 팜플렛입니다. 손님에게 설계도를 통째로 넘겨줬다가 손님이 설계도에 낙서(데이터 수정)를 하면 실제 집이 망가질 수 있기 때문에 분리하는 것이죠.
