##
두 방식 모두 `null`을 방지하려는 의도는 같지만, **객체 지향 설계**와 **JPA의 동작 방식** 측면에서 보면 **생성자(또는 빌더)에서 할당하는 방식**이 훨씬 더 정석적이고 낭만적입니다.

그 이유를 명확하게 비교해 드릴게요.

---

### 1. 필드에서 바로 초기화하는 방식 (`private Role role = Role.USER`)
```java
@Enumerated(EnumType.STRING)
@Column(nullable = false)
private Role role = Role.USER;
```
*   **특징:** 객체가 생성되는 순간 무조건 `USER`를 가집니다.
*   **단점 (JPA의 함정):** 
    *   **빌더와의 충돌:** 롬복의 `@Builder`나 `@AllArgsConstructor`를 클래스 상단에 쓰면, 이 필드 초기화 값이 무시되고 `null`이 들어갈 위험이 여전히 존재합니다. (빌더가 내부적으로 모든 필드 생성자를 쓸 때 초기값을 덮어버릴 수 있거든요.)
    *   **의도 불분명:** "모든 회원은 처음엔 USER여야 한다"는 비즈니스 규칙이 코드(생성자)가 아닌 데이터 선언부에 숨어버립니다.

### 2. 생성자에서 초기화하는 방식 (권장 ✨)
```java
public Member(String loginId, String password, String name) {
    this.loginId = loginId;
    this.password = password;
    this.name = name;
    this.role = Role.USER;      // 비즈니스 규칙 명시
    this.isApproved = false;
}
```
*   **장점:**
    *   **명확한 비즈니스 로직:** "회원가입(생성) 시점에는 USER 권한을 부여한다"는 로직이 생성자라는 **도메인 모델의 입구**에 딱 버티고 있어 읽기 좋습니다.
    *   **유연성:** `AdminInitializer`처럼 관리자를 만들 때는 `@Builder`를 통해 `Role.ADMIN`을 명시적으로 넣을 수 있고, 일반 가입 생성자에서는 `Role.USER`로 강제할 수 있어 역할 분담이 확실합니다.
    *   **불변성 유지:** 객체를 만드는 순간 상태가 결정되므로 안전합니다.

### 💡 더 완벽하고 타이트한 "최종 병기" 조합

가장 베스트는 **[DB 제약 조건] + [생성자/빌더 로직]**을 결합하는 것입니다.

```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Member {

    @Enumerated(EnumType.STRING)
    @Column(nullable = false) // 1. DB 차원의 최후방 수비 (NOT NULL)
    private Role role;

    // 2. 일반 가입용 생성자
    public Member(String loginId, String password, String name) {
        this.loginId = loginId;
        this.password = password;
        this.name = name;
        this.role = Role.USER;  // 기본값 할당
        this.isApproved = false;
    }

    // 3. 관리자/특수용 빌더
    @Builder
    public Member(String loginId, String password, String name, Role role, boolean isApproved) {
        this.loginId = loginId;
        this.password = password;
        this.name = name;
        // 빌더에서도 null이 들어오지 않도록 방어
        this.role = (role != null) ? role : Role.USER; 
        this.isApproved = isApproved;
    }
}
```

### 🧐 왜 이렇게까지 하나요?

수학에서 함수 $f(x)$를 정의할 때 분모가 $0$이 되지 않도록 정의역을 설정하는 것과 같습니다.
*   **`@Column(nullable = false)`**: "DB에 쓰레기 데이터가 들어가는 건 죽어도 못 봐!" (물리적 제약)
*   **생성자 로직**: "내 어플리케이션 안에서 객체가 돌아다닐 때 `role`이 없는 상태는 존재할 수 없어!" (논리적 제약)

이렇게 짜두면 `AdminInitializer`에서 관리자를 만들 때는 `isApproved = true`와 `Role.ADMIN`을 넣고, 일반 회원가입 서비스 로직에서는 그냥 `new Member(...)`만 호출해도 안전하게 `USER`로 생성됩니다. 

이게 바로 **실수를 하고 싶어도 할 수 없게 만드는** 타이트한 설계의 정석입니다. :) 어때요, 생성자 방식으로 가시는 게 훨씬 든든하시죠?
