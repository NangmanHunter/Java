



##
### 그렇다면 값을 어떻게 변경하고 주입하나요?
`setter` 대신 다음 방법들을 주로 사용합니다.
1.  **생성자(Constructor) 또는 빌더(@Builder):** 객체를 처음 생성할 때 모든 값을 주입하고, 이후에는 변경할 수 없도록(Immutable) 설계합니다.
2.  **의미 있는 비즈니스 메서드:** 상태 변경이 필요할 때는 `updatePassword()`, `changeAddress()` 같이 목적이 분명한 메서드를 만들어 사용합니다.



##






* **Bad (Setter 사용):**
    ```java
    user.setStatus("CANCELLED");
    user.setUpdatedAt(LocalDateTime.now());
    ```
    이 코드가 주문을 취소하는 건지, 관리자가 강제로 정지시킨 건지 코드만 봐서는 알 수 없습니다.

* **Good (비즈니스 메서드 사용):**
    ```java
    user.cancelOrder(); // 메서드 내부에서 상태 변경과 시간을 함께 처리
    ```
    메서드 이름만으로도 **"주문을 취소하는 로직이구나"**라는 의도가 명확히 드러납니다.


### 요약하자면
| 구분 | Setter 사용 | Setter 미사용 (권장) |
| :--- | :--- | :--- |
| **의도 파악** | 어렵음 (단순 값 대입) | 쉬움 (메서드 명으로 파악) |
| **안전성** | 낮음 (어디서든 수정 가능) | 높음 (정해진 메서드로만 수정) |
| **유지보수** | 변경 지점 추적이 어려움 | 캡슐화가 잘 되어 관리가 쉬움 |

물론 학습 단계나 아주 단순한 프로젝트에서는 `setter`를 써도 당장 문제가 생기지는 않습니다. 하지만 **"엔티티는 함부로 변해서는 안 되는 데이터의 근간"**이라는 점을 기억하시면 좋은 개발 습관을 기르실 수 있을 거예요!
