Spring Entity에서 초기값을 설정할 때, 사실 정답이 정해져 있다기보다는 **객체의 생명주기와 의무적인 값인지 여부**에 따라 선택이 달라지는 경우가 많습니다.

보통 실무에서는 아래의 두 가지 방식 중 상황에 맞춰 선택하곤 합니다.

---

### 1. 필드 선언 시 초기화 (Inline Initialization)
가장 간단하고 직관적인 방식입니다. 주로 **'모든 객체에 공통적으로 적용되는 고정 디폴트값'**이 있을 때 유용합니다.

```java
@Entity
public class Member {
    @Column(nullable = false)
    private String role = "USER"; // 기본값 선언

    @OneToMany(mappedBy = "member", cascade = CascadeType.ALL)
    private List<Order> orders = new ArrayList<>(); // 컬렉션은 이 방식이 권장됨
}
```

*   **장점:** 코드가 간결하고 읽기 편합니다. `NullPointerException` 방지를 위해 리스트 같은 컬렉션을 초기화할 때 가장 많이 쓰입니다.
*   **단점:** 비즈니스 로직에 따라 기본값이 유동적으로 변해야 하는 경우에는 대응하기 어렵습니다.

---

### 2. 생성자에서 초기화 (Constructor Initialization)
객체를 생성하는 시점에 **'필수적으로 받아야 하는 값'**과 **'그에 따른 기본값'**을 명확히 구분하고 싶을 때 좋습니다.

```java
@Entity
public class Post {
    private LocalDateTime createdAt;
    private boolean isPublished;

    protected Post() {} // JPA를 위한 기본 생성자

    public Post(String title) {
        this.title = title;
        this.createdAt = LocalDateTime.now(); // 생성 시점의 값 주입
        this.isPublished = false;
    }
}
```

*   **장점:** 객체 생성 시점에 복잡한 로직(예: 현재 시간 계산 등)을 포함할 수 있고, 필수 값을 강제할 수 있어 객체의 일관성을 유지하기 좋습니다.
*   **단점:** 코드가 조금 더 길어지고, JPA 전용 기본 생성자(`protected`)를 잊지 말고 챙겨줘야 합니다.

---

### 💡 결정에 도움을 줄 가이드라인

어떤 방식이 더 나을지 고민될 때는 이 기준을 참고해 보세요.

| 구분 | 추천 방식 | 이유 |
| :--- | :--- | :--- |
| **컬렉션 (List, Set 등)** | **필드 선언 시 초기화** | Hibernate가 엔티티를 추적할 때 컬렉션을 래핑하므로, 선언 시점에 `new ArrayList<>()`로 비워두는 것이 가장 안전합니다. |
| **단순 상수값 (0, false 등)** | **필드 선언 시 초기화** | 코드가 깔끔해지고 의도가 명확합니다. |
| **도메인 로직이 필요한 값** | **생성자 초기화** | 생성 시점에 따라 값이 변하거나(날짜 등), 특정 조건에 따라 기본값이 달라져야 할 때 적합합니다. |
| **Lombok `@Builder` 사용 시** | **주의 필요** | 빌더 패턴을 쓰면 필드 초기화 값이 무시될 수 있습니다. 이 경우 `@Builder.Default`를 붙이거나 생성자 방식을 써야 합니다. |

**결론적으로:** 
일반적인 기본값(상태값, 리스트 등)은 **필드 선언 시**에 바로 넣으시고, 객체가 만들어지는 순간에 계산이 필요하거나 필수 데이터와 함께 셋팅되어야 하는 값은 **생성자**를 활용하시는 것을 추천합니다. 

사소해 보이지만 엔티티의 의도를 명확히 하는 중요한 고민이네요! 보통은 **필드 초기화**를 기본으로 가되, 특이 케이스만 생성자로 분리하는 방식이 코드를 관리하기 가장 편하실 겁니다.
