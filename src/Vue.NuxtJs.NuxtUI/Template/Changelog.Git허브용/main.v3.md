##
백엔드 개발자 입장에서 보면 "어? CORS 문제인가? 아니면 내 서버가 GitHub이랑 통신이 안 되나?"라고 의심하실 수 있는 상황입니다. 🏹

결론부터 말씀드리면, **서버(Nuxt Nitro)는 긁어올 수 있는 게 맞습니다.** 하지만 유저님이 보시는 그 404 에러는 **"주소는 맞는데, 그 주소 안에 내용물(Release)이 하나도 없다"**는 GitHub API의 전형적인 응답입니다.

그 이유를 낱낱이 파헤쳐 드릴게요.

---

### 1. 왜 나만 못 가져오고, 서버는 가져올 수 있나? (Proxy의 원리)

유저님이 브라우저 주소창에 직접 쳤을 때와 코드가 실행될 때의 차이입니다.

* **코드의 작동 방식:** `useFetch`를 쓰면, Nuxt 서버(Nitro)가 백엔드 서버처럼 동작해서 GitHub API에 요청을 보냅니다.
* **404의 진의:** API 서버가 살아있고 응답도 해주지만, **"네가 요청한 그 저장소(`repository`) 경로에 'Release' 데이터가 단 한 개도 없어!"**라는 뜻입니다. 
    * GitHub은 데이터가 비어있을 때 빈 배열을 주기도 하지만, 경로 자체가 잘못되었거나 데이터가 아예 생성된 적이 없으면 404를 뱉기도 합니다.

### 2. 범인은 `appConfig.repository` 값입니다

유저님의 `app.config.ts` 파일을 다시 한 번 봐주세요. 아마 초기 설정값이 **`nuxt-ui-templates/changelog`** 같은 기본값으로 되어 있을 겁니다.

* 이 저장소는 템플릿 코드만 있지, **'Release'** (버전 배포 기록)를 등록해두지 않았을 확률이 높습니다.
* 그러니 API(`ungh.cc`)가 GitHub에 물어봐도 "거기 릴리즈 내역 없는데?" 하고 404를 돌려주는 거죠.

### 3. "말이 되나?"에 대한 답변 (ungh.cc의 정체)

코드에 있는 `https://ungh.cc/...` 이 주소는 **GitHub API를 쓰기 편하게 가공해주는 프록시(Proxy) 서비스**입니다. 

1.  **유저의 Nuxt 앱**이 `ungh.cc`에 요청을 보냄.
2.  **`ungh.cc` 서버**가 GitHub API에 접속해서 데이터를 긁어옴.
3.  데이터를 예쁘게 정리해서 **유저의 Nuxt 앱**에 전달.

이 과정에서 GitHub 저장소 이름이 유효하지 않거나, 릴리즈 내역이 없으면 중간에서 저런 JSON 에러 메시지를 던지게 됩니다.

---

### 🛠️ 당장 해결해서 화면을 띄우는 방법

테스트를 위해 **진짜 데이터가 있는 저장소**를 연결해 보세요. 

**`app.config.ts` 수정:**
```typescript
export default defineAppConfig({
  // 예시: Nuxt 공식 저장소는 릴리즈가 아주 많습니다.
  repository: 'nuxt/nuxt', 
  // ... 나머지 설정
})
```

이렇게 바꾸고 저장하면, `ungh.cc`가 Nuxt 공식 저장소의 릴리즈 내역을 긁어와서 유저님 화면에 **수십 개의 업데이트 내역**을 쫙 뿌려줄 겁니다. 



### 🏹 낭만 사냥꾼님을 위한 조언
만약 유저님의 프로젝트 내용을 띄우고 싶다면:
1.  유저님의 **GitHub 저장소**를 만든다.
2.  저장소 오른쪽 메뉴의 **[Releases]** -> **[Create a new release]**를 눌러서 글을 하나 등록한다.
3.  `app.config.ts`의 `repository`를 유저님의 `아이디/저장소명`으로 바꾼다.

이렇게 하면 유저님이 GitHub에 글을 쓸 때마다 웹사이트가 자동으로 업데이트되는 **진정한 개발자의 낭만**이 완성됩니다! 🏹

지금 `app.config.ts`에 뭐라고 적혀 있나요? 주소만 잘 적어주면 404는 바로 사라집니다!
