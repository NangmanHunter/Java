하... 이 에러, 백엔드로 치면 **라이브러리 A는 Log4j 2.x를 요구하는데, 라이브러리 B는 Log4j 1.x만 지원해서 빌드가 터지는 전형적인 의존성 지옥(Dependency Hell)** 상황입니다. 🏹

범인은 **`shiki`**라는 라이브러리입니다.
* **루트 프로젝트:** "난 최신 기능이 필요하니 `shiki` **4.x**를 쓸 거야!"
* **Tiptap 확장 프로그램:** "난 옛날 사람이라 `shiki` **3.x**가 아니면 대화 안 해!"

서로 버전이 안 맞아서 `npm`이 "나 이거 어떻게 합쳐야 할지 모르겠어!"라며 손을 놓아버린 상태입니다. 이럴 때는 **'강제로 쑤셔 넣는'** 방법이 필요합니다.

---

### 🛠️ 해결 방법: --legacy-peer-deps

터미널에 아래 명령어를 입력하세요. `npm`의 엄격한 검사를 한시적으로 끄고 설치를 강행하는 치트키입니다.

```powershell
npm install --legacy-peer-deps
```

**이 명령어의 의미:**
"버전 좀 안 맞는 거 나도 알아. 일단 그냥 깔아줘. 내가 책임지고 돌아가게 할게!"라고 명령하는 것입니다. 대부분의 경우 `shiki`의 3.x와 4.x는 호환성이 크게 깨지지 않아서 이 정도로 해결됩니다.

---

### 🧨 만약 위 방법도 안 된다면? (더 확실한 처방)

가끔 캐시나 기존 `node_modules`가 꼬여있으면 `legacy-peer-deps`도 무시될 때가 있습니다. 그때는 **'클린 설치'**로 가야 합니다.

1.  **찌꺼기 제거:**
    ```powershell
    rm -rf node_modules package-lock.json
    ```
2.  **강제 설치:**
    ```powershell
    npm install --force
    ```

---

### 🧐 백엔드 개발자를 위한 팁: `overrides`
만약 앞으로도 계속 이 버전 에러가 유저님을 괴롭힌다면, `package.json`에 "무조건 이 버전을 써!"라고 못 박아버릴 수 있습니다. (마치 Gradle의 `force`나 `exclude` 같은 설정이죠.)

```json
// package.json 에 추가
"overrides": {
  "shiki": "^4.0.2"
}
```
이렇게 적어두면 `tiptap`이 3.x를 달라고 떼를 써도 `npm`이 무시하고 4.x를 먹입니다.
