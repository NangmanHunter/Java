##
- catch에도 refresh() ㄱㄱ.


##
네, 아주 날카로운 질문입니다! 결론부터 말씀드리면 **`catch` 블록에서도 `refresh()`를 호출해 주는 것이 훨씬 안전하고 권장되는 방식**입니다.

단순히 `member.isApproved = !member.isApproved`라고 수동으로 반전시키는 것보다 `refresh()`가 더 나은 이유 3가지를 정리해 드릴게요.

---

### 1. "데이터의 무결성" 보장
수동으로 값을 반전시키는 것은 **"에러가 났으니 서버 값은 당연히 이전 값일 것이다"**라는 추측에 기반합니다. 하지만 실제로는 다음과 같은 복잡한 상황이 생길 수 있습니다.
*   네트워크 타임아웃으로 에러가 났지만, 실제 서버 DB에는 반영이 되었을 경우.
*   서버에서 값을 변경하다가 다른 로직(트리거 등)에 의해 제3의 값으로 변했을 경우.
이때 `refresh()`를 하면 내 추측이 아니라 **서버에 저장된 실제 데이터**를 가져오므로 화면과 DB가 완벽하게 일치하게 됩니다.

### 2. "복합적인 상태 변경" 대응
체크박스 하나만 관리하는 게 아니라, 승인 여부에 따라 **'승인 날짜'**나 **'승인자 이름'** 같은 다른 필드들도 같이 표시되고 있다면? 수동으로 코드를 짜면 이 모든 변수를 다 일일이 원복해야 합니다. `refresh()`는 이 모든 고민을 한 방에 해결해 줍니다.

### 3. 코드의 단순함 (Maintainability)
`isApproved`가 아닌 다른 필드가 추가되더라도 `catch` 로직을 수정할 필요가 없습니다. "실패하면 무조건 다시 긁어온다"는 규칙 하나로 코드가 견고해지죠.

---

### 추천하는 패턴

가장 깔끔한 구현 모습은 이렇습니다:

```javascript
async function toggleApprove(member) {
  try {
    await $fetch(`/api/admin/members/${member.id}/approve`, {
      method: 'PATCH',
      body: { isApproved: member.isApproved },
      // ... 생략
    })
    // 성공 시: 최신 상태 동기화 (업데이트 시간 등 반영)
    await refresh()
  } catch (e) {
    alert('수정에 실패했습니다.')
    // 실패 시: 사용자가 바꾼 UI를 서버의 원래 값으로 강제 롤백
    await refresh()
  }
}
```

### 💡 한 가지만 더 고려한다면? (사용자 경험)
만약 네트워크가 아주 느린 환경이라면 `catch`에서 `refresh()`가 완료될 때까지 체크박스가 어정쩡한 상태로 남아있을 수 있습니다. 이때는 다음과 같은 순서가 가장 "친절"합니다.

1.  `alert('실패했습니다')` 띄우기
2.  `member.isApproved = !member.isApproved` (즉시 UI 원복해서 사용자 안심시키기)
3.  `await refresh()` (백그라운드에서 혹시 모를 데이터 불일치 최종 확인)

이렇게 하면 사용자는 **"아, 안 됐구나"**를 바로 알 수 있고, 개발자는 **"데이터가 정확하구나"**라는 확신을 가질 수 있습니다.

**결론:** `catch`에서도 `refresh()`를 호출하는 습관은 데이터 사고를 막아주는 아주 좋은 습관입니다!
