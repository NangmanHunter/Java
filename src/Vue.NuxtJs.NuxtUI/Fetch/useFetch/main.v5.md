네, 아주 날카로운 질문입니다! 결론부터 말씀드리면 **`useFetch`는 초기 로딩 이후 데이터가 변경될 때 똑똑하게 CSR(Client Side Rendering) 방식으로 동작합니다.**

단순히 페이지를 처음 불러올 때만 쓰는 게 아니라, 페이지 내에서 파라미터가 바뀌거나 수동으로 갱신해야 할 때도 아주 유용하죠. 구체적으로 어떻게 작동하는지 정리해 드릴게요.

---

## 1. 반응형 파라미터 감지 (Auto-refresh)

`useFetch`의 가장 큰 장점 중 하나는 URL이나 옵션에 `ref` 또는 `computed` 변수가 들어가면, **그 값이 바뀔 때마다 자동으로 데이터를 다시 불러온다**는 점입니다. 이때는 당연히 서버를 거치지 않고 브라우저에서 바로 API를 호출(CSR)합니다.

```javascript
const page = ref(1)

// page 값이 바뀔 때마다 useFetch가 이를 감지하고 자동으로 CSR 요청을 보냄
const { data } = await useFetch(() => `/api/posts?page=${page.value}`)

const nextPage = () => {
  page.value++ // 이렇게만 하면 데이터가 새로 고쳐집니다!
}

```

---

## 2. 수동 갱신 (refresh 함수)

데이터를 강제로 다시 가져와야 할 때(예: '새로고침' 버튼 클릭)는 `useFetch`가 반환하는 `refresh` 함수를 사용합니다. 이 역시 클라이언트 사이드에서 실행됩니다.

```javascript
const { data, refresh, pending } = await useFetch('/api/user-profile')

const updateData = async () => {
  await refresh() // 서버에 다시 요청하여 data를 업데이트함
}

```

---

## 3. Hydration 이후의 동작 원리

`useFetch`의 일생을 그려보면 다음과 같습니다:

1. **초기 접속 (SSR):** 서버가 API를 호출해서 HTML에 데이터를 담아 보냅니다. 브라우저는 이 데이터를 그대로 써서 화면을 그립니다. (네트워크 요청 0번)
2. **페이지 체류 중 (CSR):**
* 사용자가 위 예시처럼 `page` 번호를 바꿉니다.
* `useFetch`는 "어? 주소가 바뀌었네?" 하고 브라우저에서 직접 서버로 Fetch 요청을 보냅니다.
* 이때는 일반적인 SPA(Single Page Application)처럼 동작하며 화면의 일부분만 업데이트됩니다.



---

## 요약하자면

`useFetch`는 "처음에는 서버에서 가져오고(SSR), 그 이후 데이터 변경은 브라우저에서 처리(CSR)"하는 하이브리드 방식입니다.

* **상태 유지:** `data` 객체가 반응형(`ref`)이기 때문에, 새로운 데이터를 가져오면 이 데이터를 참조하고 있는 UI 컴포넌트들이 자동으로 리렌더링됩니다.
* **사용자 경험:** `pending` 값을 활용해 데이터를 가져오는 동안 로딩 스피너를 보여주는 등의 처리가 매우 쉽습니다.

**팁:** 만약 URL이 바뀌어도 자동으로 데이터를 새로 가져오지 않게 하고 싶다면, `watch: false` 옵션을 주거나 함수 형태가 아닌 고정된 문자열로 URL을 전달하면 됩니다.

혹시 특정 변수가 바뀌어도 데이터를 안 불러오게 막고 싶거나, 반대로 강제로 추적하게 하고 싶은 상황이 있으신가요?
