이 부분이 Nuxt(그리고 SSR 프레임워크)를 처음 접할 때 가장 머리 아픈 지점입니다! 아주 예리한 질문이에요.

결론부터 말씀드리면: **"서버가 이미 데이터를 다 받아와서 HTML에 꽉꽉 채워 넣은 다음, 완성된 화면을 클라이언트에 던져줍니다."**

단계별로 설명해 드릴게요.

---

### 1. 일반적인 CSR (React, Vue 생으로 쓸 때)

1. 클라이언트(브라우저)가 서버에 페이지 요청.
2. 서버는 텅 빈 `index.html`과 자바스크립트 파일을 보냄. (사용자는 이때 로딩 화면이나 빈 화면을 봄)
3. 브라우저가 자바스크립트를 실행해서 **그때서야 API 서버로 데이터를 요청.**
4. 데이터를 받아오면 화면을 그림.

### 2. Nuxt의 SSR (useFetch 사용 시)

1. 클라이언트가 서버에 페이지 요청.
2. **Nuxt 서버**가 코드를 읽다가 `useFetch`를 발견함.
3. **서버가 직접 API 서버에 데이터를 요청**해서 받아옴.
4. **중요:** 서버는 받아온 데이터를 HTML에 미리 다 그려 넣습니다 (Pre-rendering).
5. 서버는 데이터가 다 차 있는 **완성된 HTML**을 브라우저에 보냅니다.
6. 사용자는 브라우저를 열자마자 데이터가 채워진 화면을 즉시 보게 됩니다.

---

### "데이터 요청을 다시 보내야 하는 거 아닌가요?" (하이드레이션)

여기서 궁금증이 생기실 거예요. "그럼 브라우저로 화면이 넘어온 뒤에, 자바스크립트가 다시 실행되면서 `useFetch`를 또 호출하지 않나요?"

**아닙니다!** Nuxt는 여기서 매우 영리하게 동작합니다.

1. 서버에서 `useFetch`로 받아온 데이터를 특정 상태(Payload)로 HTML 바닥에 몰래 숨겨서 같이 보냅니다. (`window.__NUXT__` 같은 곳에 말이죠.)
2. 브라우저에서 자바스크립트가 살아날 때(이걸 **Hydration**이라고 합니다), `useFetch`를 실행하려고 보니 "어? 이미 서버에서 받아온 데이터가 있네?"라고 판단합니다.
3. **네트워크 요청을 생략**하고 서버가 보낸 데이터를 그대로 재사용합니다.

---

### 요약하자면

* **SSR일 때:** 서버가 API 요청을 대신 해주고, 결과물을 HTML에 구워서 줍니다. (클라이언트는 API 요청을 안 함)
* **페이지 이동 시 (SPA 모드):** 이미 사이트에 접속한 상태에서 메뉴를 눌러 다른 페이지로 갈 때는, 브라우저가 직접 API 요청을 보냅니다. 이때도 `useFetch`가 알아서 상황에 맞게 동작합니다.

### 왜 이렇게 하나요? (장점)

1. **SEO (검색엔진 최적화):** 네이버나 구글 봇이 왔을 때, 이미 데이터가 다 차 있는 HTML을 보게 되므로 검색 노출에 유리합니다.
2. **사용자 경험:** "로딩 중..." 화면을 보는 시간이 줄어들고, 첫 화면이 뜨는 속도가 굉장히 빠릅니다.

그래서 게시판 상세 페이지 같은 곳은 `useFetch`를 써서 서버에서 미리 데이터를 가져오는 게 훨씬 이득인 것이죠! 이해가 좀 되시나요? 😊
