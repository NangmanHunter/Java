##
오, 날카로운 지적이세요! 맞습니다. 결과적으로 **"No data"가 떴다는 건 `null`이 아니라는 뜻**이니, 지금 상황만 놓고 보면 굳이 필요 없어 보일 수 있습니다.

하지만 **`?? []`**를 써주는 습관은 일종의 **'방어 운전'** 같은 거예요. 왜 개발자들이 귀찮아도 이 코드를 습관적으로 넣는지 그 이유를 간단히 짚어 드릴게요.

---

### 1. 에러로 인한 '화면 뻗음' 방지
지금은 `useFetch`가 에러 없이 작동해서 `[]`라도 받아온 상태지만, 만약 다음과 같은 상황이 벌어지면 이야기가 달라집니다.
* **백엔드 서버가 꺼졌을 때**
* **네트워크가 끊겼을 때**
* **API 주소가 오타 났을 때**

이때 `boards`는 `null`이 됩니다. 만약 `?? []` 처리가 없으면 `UTable`은 `null`을 읽으려다 **자바스크립트 에러(Runtime Error)**를 내뱉고, 화면 전체가 하얗게 멈춰버릴 수 있습니다. `?? []`가 있으면 최소한 화면은 유지된 채로 "No data"만 띄워주죠.

### 2. 초기 렌더링 찰나의 순간
`useFetch`는 비동기입니다. 아주 짧은 순간이지만, 데이터를 가져오기 전에는 `boards`의 값이 기본적으로 `null`입니다. 
* **`?? []`가 없을 때:** 0.1초 동안 에러가 났다가 데이터가 들어오면서 화면이 그려짐 (깜빡임이나 콘솔 에러 발생 가능).
* **`?? []`가 있을 때:** 처음부터 빈 배열로 인식해서 조용히 기다리다가 데이터가 오면 자연스럽게 업데이트됨.

---

### 3. 결론: "안 써도 되긴 하지만, 쓰는 게 정신 건강에 좋다"

지금 당장 "No data"가 나오는 건 **API 응답이 빈 배열(`[]`)**이라서 그런 것이니, 코드상으로는 문제가 없습니다. 

다만, 나중에 **웹툰 플랫폼**이나 **통계 서비스**처럼 실제 사용자가 접속하는 환경을 만드실 때는, 서버가 불안정할 때를 대비해 이렇게 작성하시는 걸 추천드려요.

```typescript
// 가장 깔끔한 방법은 script단에서 끝내는 것!
const { data: boards } = await useFetch('/api/boards', {
  default: () => [] // 이렇게 하면 템플릿에서 ?? [] 안 써도 됩니다.
});
```

**한 줄 요약:** 지금 "No data"가 뜨는 건 **`null` 처리가 이미 잘 됐거나**, **API가 빈 값(`[]`)을 줬기 때문**입니다. 이제 백엔드 DB에 데이터 하나만 넣고 새로고침 해보세요! 바로 리스트가 뜰 겁니다. :)
