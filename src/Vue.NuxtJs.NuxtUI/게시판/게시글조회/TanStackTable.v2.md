##
개발자 입장에서는 충분히 헷갈릴 수 있는 변화예요! 기존에는 그냥 "이 열에 이거 보여줘" 수준이었다면, **TanStack Table(v3의 기반)**로 넘어오면서 테이블을 단순한 '그림'이 아니라 아주 정교한 **'데이터 처리 엔진'**으로 취급하기 시작했거든요.

왜 이렇게 복잡한 용어(`accessorKey`, `header` 등)를 쓰기로 했는지, 그 이유를 알면 조금 덜 억울(?)하실 거예요.




### 1. "데이터 꺼내기"와 "보여주기"의 분리
예전에는 `key` 하나로 데이터도 찾고 내부 식별자로도 썼지만, 실제 실무에서는 이런 경우가 많습니다.
* **데이터는 `writerName`인데, 화면에는 `작성자`라고 보여주고 싶고, 정렬할 때는 `writerId`를 기준으로 하고 싶을 때.**
이걸 해결하려고 역할을 쪼갠 거예요.

* **`accessorKey`**: "원천 데이터(JSON)의 어디서 값을 가져올 것인가?" (Data Access)
* **`id`**: "이 컬럼을 시스템 내부적으로 뭐라고 부를 것인가?" (Unique ID)
* **`header`**: "사용자 눈에는 뭐라고 보여줄 것인가?" (Display)


### 2. 점 표기법(Dot Notation) 지원
`accessorKey`라는 이름을 쓰는 큰 이유 중 하나는 **중첩된 데이터**를 쉽게 가져오기 위해서입니다.

```json
// 데이터 구조가 이럴 때
{
  "title": "공지사항",
  "author": { "name": "김철수", "level": "Admin" }
}
```
기존 방식으로는 힘들었지만, 이제는 `accessorKey: 'author.name'`이라고만 적으면 TanStack이 알아서 깊숙이 들어가 값을 꺼내옵니다. 이걸 가능하게 하려고 단순한 `key` 이상의 기능을 담은 `accessor`라는 개념을 가져온 거죠.

### 3. "Headless" 철학 (가장 핵심)
TanStack Table은 **"기능만 담당하고 디자인은 네가 알아서 해"**라는 철학을 가지고 있습니다. 

* **Headless**: 정렬, 필터링, 페이징 로직은 완벽하게 짜놓을 테니, 너는 그 결과값(Cell)을 어떻게 그릴지만 결정해라.
* 이 규격을 맞추다 보니 라이브러리 제작자들이 공통적으로 사용하는 표준 용어(`header`, `cell`, `accessor`)를 그대로 수용하게 된 것입니다.

---

### 💡 헷갈림을 줄여주는 매칭 표
이것만 기억하시면 예전 지식과 연결하기 쉬울 거예요.

| 옛날 방식 (Nuxt UI v2) | 최신 방식 (Nuxt UI v3 / TanStack) | 의미 |
| :--- | :--- | :--- |
| `label` | **`header`** | 머리글에 나올 글자 |
| `key` | **`accessorKey`** | JSON 데이터의 필드명 |
| `slot` (자동 생성) | **`cell`** | 칸의 내용을 커스텀할 때 |
