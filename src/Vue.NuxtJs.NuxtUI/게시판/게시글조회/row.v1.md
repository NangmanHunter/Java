`row`라는 객체가 낯설게 느껴지는 건 당연합니다! 기존 게시판 만들기 방식에서는 데이터 배열을 `v-for="item in items"`로 돌려서 `item`을 직접 썼지만, **TanStack Table(Nuxt UI v3)**은 데이터를 한 번 더 가공해서 **"똑똑한 행(Row)"** 객체를 새로 만들기 때문입니다.

`row`가 어떻게 구성되어 있는지, 그 핵심 명세를 핵심 위주로 짚어드릴게요.

---

### 1. `row` 객체의 정체 (구조)
우리가 넘겨준 데이터가 `{ id: 1, title: '안녕' }`이라면, TanStack은 이를 받아서 아래와 같은 **메타 데이터가 포함된 객체**로 감쌉니다.

```typescript
// row 객체의 대략적인 내부 모습 (실제 명세 기반)
{
  id: "0",             // 테이블 내에서의 인덱스 기반 ID
  index: 0,            // 행의 순서
  original: {          // ⭐ 우리가 서버에서 가져온 "진짜 데이터" (가장 중요!)
    id: 1,
    title: "안녕",
    writerName: "작성자",
    ...
  },
  getIsSelected: () => boolean,   // 이 줄이 체크박스로 선택되었는가?
  getCanSelect: () => boolean,    // 이 줄을 선택할 수 있는가?
  getValue: (columnId) => any,    // 특정 컬럼의 값만 쏙 뽑아오기
  // ... 기타 정렬, 확장, 필터링 관련 함수들
}
```



---

### 2. 왜 `original` 안에 숨겨놨을까?
데이터를 바로 보여주지 않고 `original`이라는 속성 안에 넣어둔 이유는 **"테이블 관리용 데이터"**와 **"사용자의 실제 데이터"**가 섞이지 않게 하기 위해서입니다.

* **테이블 관리 데이터:** "이 줄이 지금 펼쳐져 있나?", "이 줄이 체크되어 있나?" 같은 상태값.
* **사용자 데이터 (`original`):** "DB에서 가져온 제목, 내용, 날짜" 등.

이걸 구분해 놓았기 때문에, 나중에 테이블에 **체크박스 다중 선택**이나 **행 확장(Accordion)** 기능을 추가해도 서버 데이터가 오염되지 않고 깔끔하게 관리됩니다.

---

### 3. 가장 많이 쓰는 3대 속성/함수

테이블 만드실 때 이 세 가지만 기억하시면 90%는 해결됩니다.

1.  **`row.original`**: 
    * **용도:** 슬롯에서 `id`를 꺼내 링크를 걸거나, 제목을 보여줄 때 사용합니다.
    * **예시:** `:to="'/board/' + row.original.id"`
2.  **`row.getValue('columnId')`**:
    * **용도:** 특정 컬럼의 값만 가져오고 싶을 때 사용합니다. (가공이 필요할 때 유용)
    * **예시:** `row.getValue('viewCount')`
3.  **`row.index`**:
    * **용도:** 게시판에서 DB ID 말고, 그냥 화면에 보이는 **순번(1, 2, 3...)**을 매기고 싶을 때 사용합니다.

---

### 4. TypeScript에서의 정의 (Interface)
만약 코드에서 타입을 정확히 알고 싶다면, `@tanstack/vue-table` 패키지에서 정의된 `Row` 인터페이스를 참고하게 됩니다.

```typescript
import type { Row } from '@tanstack/vue-table'

// 우리가 만든 데이터 타입이 Board라면
// row: Row<Board> 형태가 됩니다.
```

### 요약하자면
`row`는 **"내가 준 데이터를 TanStack이 테이블용으로 한 번 포장한 상자"**입니다. 
우리는 그 상자 안에 있는 **내용물**이 필요하니까 항상 **`row.original`**이라는 딱지를 열어보는 것이라고 이해하시면 완벽합니다!
