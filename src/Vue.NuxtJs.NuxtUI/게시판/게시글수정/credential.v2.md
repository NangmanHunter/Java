이 부분은 네트워크 인프라를 처음 접하면 충분히 헷갈릴 수 있는 지점입니다! 결론부터 말씀드리면, **포트 번호까지 아예 없애고 하나의 주소처럼 보이게 만드는 마법**을 부리는 거예요.

`nangman.com:8080` 처럼 포트를 붙이는 게 아니라, 사용자 눈에는 **둘 다 그냥 `80`번(HTTP) 혹은 `443`번(HTTPS) 포트 하나**로 보이게 됩니다.

---

### 1. Nginx가 "교통 정리"를 해줍니다

Nginx라는 녀석을 앞단에 세우면, 이 친구가 **'리버스 프록시(Reverse Proxy)'** 역할을 수행합니다. 사용자는 오직 Nginx의 주소(`nangman.com`)만 보고 접속하고, Nginx가 뒤쪽에서 요청을 배달해주는 구조입니다.

| 사용자 요청 주소 | Nginx의 판단 (설정) | 실제 배달되는 곳 (내부망) |
| :--- | :--- | :--- |
| `nangman.com/` | "이건 프론트 요청이네?" | `localhost:3000` (Nuxt) |
| `nangman.com/api/` | "이건 백엔드 요청이네?" | `localhost:8080` (Spring) |



---

### 2. 왜 이게 "같은 도메인"인가요?

브라우저가 판단하는 **Same-Origin(같은 출처)**의 기준은 **[프로토콜 + 도메인 + 포트]**입니다.

* **사용자 브라우저가 보는 주소:**
    * 메인 페이지: `https://nangman.com`
    * API 요청: `https://nangman.com/api/boards`
* **브라우저의 판단:** "어? 둘 다 똑같이 `https://nangman.com`에서 왔네? **포트도 똑같이 443(HTTPS 생략)이네?** 이건 완벽하게 같은 집(Origin)이다!"

브라우저 입장에서는 뒤에서 Nginx가 3000번으로 보내는지 8080번으로 보내는지 전혀 모릅니다. 그냥 앞문(`nangman.com`)만 보고 판단하기 때문에 **CORS 문제가 원천 봉쇄**되고, 쿠키도 "내 집 주소니까 당연히 가져가야지" 하면서 알아서 챙기게 되는 거죠.

---

### 3. 구조적 차이 (Before & After)

#### 지금 (개발 환경)
* 프론트: `localhost:3000`
* 백엔드: `localhost:8080`
* **결과:** 도메인은 같지만 **포트가 다름** → 브라우저는 "다른 집"으로 인식 → `credentials: 'include'` 필수.

#### 배포 시 (Nginx 활용)
* 프론트 & 백엔드 통합 주소: `nangman.com` (포트 80/443)
* **결과:** 모든 주소가 **동일** → 브라우저는 "같은 집"으로 인식 → 아무 설정 없어도 쿠키 자동 전송.

---

### 💡 요약하자면
"묶는다"는 말은 포트를 여러 개 노출하는 게 아니라, **모든 포트를 Nginx 뒤로 숨기고 밖으로는 `80`번이나 `443`번이라는 하나의 통로만 열어두는 것**을 의미합니다.

이렇게 구성하면 보안상으로도 훨씬 유리하고(실제 서버 포트가 밖으로 안 보이니까요), 쿠키 설정 때문에 머리 싸맬 일도 없어져서 훨씬 "Nangman" 있는 배포 방식이 됩니다! :)
