##
스레드 풀의 개수를 정하는 것은 백엔드 성능 튜닝의 **'꽃'**이자, 정답이 없는 **'예술'**에 가깝습니다. 유저님이 짐작하신 대로 "코어 개수대로"는 아주 기초적인 가이드일 뿐, 실제 서비스에서는 작업의 성격에 따라 완전히 달라집니다.

가장 유명한 공식과 실무적인 기준 3가지를 정리해 드릴게요.

---

### 1. 작업의 성격에 따른 황금 공식

컴퓨터 공학에서 가장 널리 쓰이는 공식은 **Goetz(게츠)의 공식**입니다.


#### ① CPU 집약적 작업 (CPU Bound)
* **어떤 일:** 복잡한 수학 연산, 암호화, 이미지 변환 등 CPU가 계속 돌아가야 하는 일.
* **기준:** **`Core 개수 + 1`**
* **이유:** 일꾼이 코어 개수보다 많아봤자, CPU는 한정되어 있어 '누구 일을 먼저 할지' 고민하는 **컨텍스트 스위칭(Context Switching)** 비용만 커집니다. `+1`은 혹시나 일꾼 한 명이 잠시 쉴 때를 대비한 예비군입니다.

#### ② I/O 집약적 작업 (I/O Bound) - 유저님의 URL Unfurling 케이스
* **어떤 일:** DB 조회, 외부 API 호출, 파일 읽기/쓰기 등 **'기다림'**이 많은 일.
* **공식:** **`Core 개수 * (1 + 대기 시간 / 작업 시간)`**
* **이유:** 일꾼이 밖으로 나가서 데이터를 기다리는 시간이 길수록, 그동안 놀고 있는 CPU 코어를 활용하기 위해 더 많은 일꾼을 투입해도 됩니다. 
    * 예: 기다리는 시간(2초)이 실제 연산 시간(0.1초)보다 월등히 길면, 코어 수의 20배 이상 스레드를 늘려도 서버는 잘 돌아갑니다.



---

### 2. 실무적인 스레드 풀 설정 3단계 (Step-by-Step)

이론은 이론일 뿐, 실제로는 다음 단계를 거쳐 설정합니다.

#### 1단계: Tomcat(Spring Boot) 기본값 이해
* 스프링 부트의 내장 톰캣은 보통 **`max-threads: 200`**이 기본입니다.
* 대부분의 일반적인 서비스는 이 200개 안에서 다 해결됩니다. "굳이?"라는 생각이 든다면 일단 기본값으로 시작하세요.

#### 2단계: 핵심 부대의 분리 (Bulkhead)
아까 말씀드린 **분업화**가 여기서 나옵니다.
* **메인 부대:** 일반적인 로그인, 조회 등 (200개)
* **URL 분석 부대 (따로 설정):** 이 작업은 외부 사이트 상태에 따라 수 초씩 걸릴 수 있으니, 메인 부대를 다 잡아먹지 않게 **`Core 개수 * 2`** 정도로 작게 시작해서 모니터링하며 늘려갑니다.

#### 3단계: 부하 테스트 (Stress Test)
* 설정값을 바꾼 뒤에는 **nGrinder**나 **JMeter** 같은 도구로 가짜 손님을 만 명쯤 보내봅니다.
* 스레드 수를 늘렸는데 응답 속도가 오히려 느려진다면? 컨텍스트 스위칭 부하가 발생한 것이니 다시 줄여야 합니다.

---

### 3. 설정 시 주의할 점: "메모리는 공짜가 아니다"

스레드 개수를 무작정 늘리지 못하는 결정적인 이유는 **메모리(RAM)** 때문입니다.

* **스레드 1개당 약 1MB**의 스택 메모리를 점유합니다.
* 스레드를 1,000개 만들면, 아무 일도 안 해도 벌써 1GB의 메모리가 사라집니다. 


* **결론:** 스레드 개수 설정은 우리 서버의 **RAM 용량** 안에서 노는 일꾼들의 잔치여야 합니다.
