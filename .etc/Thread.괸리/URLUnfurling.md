매우 날카로운 질문입니다! "프런트엔드(브라우저)에서 바로 URL 정보를 긁어오면(Scraping) 백엔드 스레드 고민도 안 해도 되고 훨씬 편할 텐데, 왜 굳이 백엔드를 거칠까?"라는 의문이 드는 게 당연합니다.

결론부터 말씀드리면, **'보안(CORS)'**과 **'성능/데이터 관리'**라는 거대한 벽 때문에 백엔드가 그 짐을 대신 짊어지는 것입니다.


---

### 1. 가장 큰 장벽: CORS (보안 정책)
브라우저에는 **'동일 출처 정책(Same-Origin Policy)'**이라는 강력한 보안 규칙이 있습니다.

* **상황:** 유저님이 만든 사이트(`my-app.com`)에서 자바스크립트로 네이버(`naver.com`)나 구글의 HTML을 직접 가져오려고 시도합니다.
* **결과:** 브라우저가 차단합니다. "너 네이버 아니지? 남의 집 대문(데이터)을 함부로 열려고 해? 안돼!"라며 에러를 뿜습니다.
* **백엔드의 역할:** 서버(Java)는 브라우저가 아니기 때문에 이런 CORS 제한을 받지 않습니다. 그래서 백엔드가 대신 가서 데이터를 긁어온 뒤, 유저님의 프런트엔드에 "자, 여기 데이터 있어"라고 안전하게 전달해 주는 **'대리인'** 역할을 하는 것입니다.

### 2. 성능과 리소스: "한 번만 긁자" (Caching)
프런트엔드에서 처리하면 모든 유저가 각자 그 사이트를 긁어와야 합니다.

* **프런트엔드 방식:** 유저 1,000명이 똑같은 `https://google.com` 링크를 올리면, 구글 서버에 1,000번의 요청이 가고 유저의 데이터(통신비)도 1,000번 소모됩니다.
* **백엔드 방식:** 백엔드에서 한 번 긁어서 **DB나 캐시(Redis 등)**에 저장해 둡니다. 다음 유저가 똑같은 링크를 올리면, 실제 사이트에 갈 필요 없이 백엔드 메모리에서 0.001초 만에 꺼내 줄 수 있습니다. 이게 훨씬 효율적이죠.


### 3. 데이터 가공의 복잡성
URL 미리보기를 하려면 HTML을 다 긁어와서 `<meta property="og:image">` 같은 태그를 찾아서 분석해야 합니다.

* 브라우저에서 이 무거운 작업을 하면 유저의 폰이나 컴퓨터가 느려질 수 있습니다.
* 자바에는 `Jsoup` 같은 아주 강력하고 안정적인 라이브러리들이 많아서, 이런 **'무거운 텍스트 분석'** 작업을 서버의 강력한 CPU로 처리해서 깔끔한 JSON 데이터만 프런트엔드로 쏴주는 게 표준입니다.

---

### 4. 그럼에도 프런트엔드에서 'Async'를 쓰는 이유
백엔드에서 이 작업을 하기로 결정했다면, 이제 프런트엔드와 백엔드 사이의 통신에서 **비동기(Async)**가 빛을 발합니다.

1.  유저가 URL 입력 (Input 변함)
2.  프런트: "자바 서버야, 이 URL 정보 좀 가져와줘!" (비동기 Ajax 요청)
3.  **프런트: (기다리지 않고 유저가 다른 걸 타이핑할 수 있게 해줌 - Non-blocking)**
4.  백엔드: (열심히 긁어옴 - 여기서 스레드 관리 필요!)
5.  백엔드: "다 됐어!" 응답 보냄.
6.  프런트: 화면에 예쁘게 미리보기 카드 출력.

### 요약하자면
* **프런트엔드에서 직접 하기 힘든 이유:** 보안(CORS) 때문에 남의 사이트 데이터를 맘대로 못 가져옵니다.
* **백엔드로 가져가는 이유:** 보안 문제를 해결하고, 결과를 저장(캐싱)해서 성능을 높이며, 복잡한 분석 작업을 서버에서 처리하기 위해서입니다.
