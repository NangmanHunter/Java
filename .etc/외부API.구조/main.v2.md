외부 API 연동이 포함될 때 데이터가 흐르는 **'길'**을 생각하면 위치를 잡기가 훨씬 수월합니다.

사용자가 요청을 보냈을 때, 데이터가 어디를 거쳐서 외부 API까지 갔다가 다시 돌아오는지 그 순서대로 위치를 잡아드릴게요.

### 1. 전체적인 데이터 흐름 (Flow)


1.  **Controller:** 사용자의 요청(예: 버튼 클릭)을 가장 먼저 받습니다.
2.  **Service:** "외부 데이터를 가져와서 가공해라"라는 **명령**을 내립니다.
3.  **Client (Infra):** 실제로 외부 API 주소로 요청을 날려 데이터를 **받아옵니다.**
4.  **Service:** 받아온 데이터를 우리 서비스에 맞게 정리합니다.
5.  **Controller:** 최종 결과물을 사용자에게 보여줍니다.

---

### 2. 구체적인 패키지 위치와 역할

이해를 돕기 위해 **`WeatherService`**(날씨 정보를 가져오는 기능)를 예로 들어 구조를 짜보겠습니다.

```text
src/main/java/com/example/project
├── controller
│   └── WeatherController.java      <-- (1) "날씨 알려줘" 요청을 받음
├── service
│   └── WeatherService.java         <-- (2) 비즈니스 로직 (Client 호출 + 데이터 가공)
└── infra
    └── weather
        ├── WeatherApiClient.java   <-- (3) 실제 외부 API(OpenWeather 등)와 통신하는 코드
        └── WeatherApiResponse.java <-- (4) 외부 API가 주는 날씨 데이터 규격 (DTO)
```

### 3. 왜 `infra`나 `external` 패키지에 따로 두나요?

입력받은 데이터를 외부 API에 단순히 전달만 하는 것 같아도, 별도의 위치에 두는 이유는 **"외부의 변화로부터 내 코드를 보호하기 위해서"**입니다.

* **외부 API 주소가 바뀌거나:** `infra` 패키지 안에 있는 코드만 수정하면 됩니다. `Service`나 `Controller`는 건드릴 필요가 없죠.
* **외부 API 업체가 바뀌면 (A사 -> B사):** `Service`는 그대로 두고, `infra` 쪽 구현체만 싹 갈아끼우면 끝납니다.

---

### 요약하자면 이렇습니다!

* **받는 곳:** `controller`
* **시키는 곳:** `service`
* **진짜로 외부랑 통신하는 곳:** `infra` (또는 `external`)

처음에는 `service` 패키지 안에 다 넣고 싶을 수도 있지만, **"외부 API와 연결되는 통신 전문 부서(`infra`)"**를 따로 하나 차린다고 생각하시면 편할 거예요.
