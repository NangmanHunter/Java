프로세스(Process)는 각각 독립된 메모리 공간을 가지기 때문에, 남의 집 안방에 마음대로 들어갈 수 없는 것처럼 서로의 데이터에 직접 접근할 수 없습니다. 그래서 등장한 개념이 바로 **IPC(Inter-Process Communication, 프로세스 간 통신)**입니다.

운영체제가 제공하는 주요 IPC 매커니즘들을 알기 쉽게 정리해 드릴게요.

---

## 1. 대표적인 IPC 모델

### 🧊 공유 메모리 (Shared Memory)
프로세스들이 공유해서 사용할 수 있는 **공통의 메모리 영역**을 할당받는 방식입니다.
* **특징:** 데이터 복사 오버헤드가 없어 **가장 빠릅니다.**
* **단점:** 여러 프로세스가 동시에 한 곳을 건드리면 데이터가 꼬일 수 있어, 별도의 동기화(예: 세마포어) 처리가 필수입니다.

### ✉️ 메시지 전달 (Message Passing)
커널(Kernel)을 거쳐서 메시지를 주고받는 방식입니다.
* **파이프 (Pipe):** 한쪽에서는 쓰고 한쪽에서는 읽는 단방향 통신입니다. (부모-자식 프로세스 간 주로 사용)
* **메시지 큐 (Message Queue):** 메시지들을 큐(Queue) 구조로 쌓아두고 주고받는 방식입니다.
* **소켓 (Socket):** 본래 네트워크 통신용이지만, 로컬 컴퓨터 내의 프로세스끼리 통신할 때도 사용합니다.

---

## 2. 한눈에 비교하는 IPC

| 종류 | 통신 방식 | 주요 특징 |
| :--- | :--- | :--- |
| **공유 메모리** | 직접 참조 | 속도가 매우 빠르나 동기화가 복잡함 |
| **파이프** | 흐름(Stream) | 단순하지만 단방향 통신 위주 |
| **메시지 큐** | 메시지 단위 | 비동기적이며 구조화된 데이터 전송 가능 |
| **소켓** | 양방향 | 원격(다른 컴퓨터) 통신까지 확장 가능 |

---

## 3. 핵심은 "동기화(Synchronization)"
프로세스들이 대화를 나눌 때 가장 중요한 건 **"순서"**입니다. A가 다 쓰지도 않았는데 B가 읽어버리면 엉뚱한 데이터를 얻게 되죠. 이를 방지하기 위해 다음과 같은 장치들을 함께 사용합니다.

* **뮤텍스(Mutex):** "나 지금 이거 쓰고 있으니까 건드리지 마!" (상호 배제)
* **세마포어(Semaphore):** "지금 빈자리가 3개 있으니 3명까지 들어와!" (개수 제한)

> **요약하자면:** 프로세스들은 직접 대화할 수 없어서, 운영체제가 마련해준 **공용 게시판(공유 메모리)**이나 **우체통(메시지 전달)**을 이용해 소통한다고 이해하시면 됩니다.
