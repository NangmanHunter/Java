





## 2. 자바의 통신 계층 (JDBC)
자바에서 DB와 대화하기 위해 사용하는 표준 인터페이스가 바로 **JDBC(Java Database Connectivity)**입니다.

* **JDBC API:** 자바 코드가 DB에 쿼리를 던지고 결과를 받는 표준화된 방법입니다.
* **JDBC 드라이버:** 각 DB 제조사(Oracle, MySQL 등)가 제공하는 일종의 '통역사'입니다. 자바의 요청을 해당 DB가 알아들을 수 있는 고유 프로토콜로 변환해 소켓에 실어 보냅니다.

---

## 3. 통신 최적화: 커넥션 풀 (Connection Pool)
매번 DB에 연결할 때마다 소켓을 열고 닫는 것은 "프로세스 간 통신" 관점에서 매우 무거운 작업입니다. (핸드쉐이킹 등 비용 발생)

* 그래서 미리 여러 개의 연결(Connection)을 맺어놓고 **'풀(Pool)'**에 담아둡니다.
* 애플리케이션은 필요할 때 풀에서 연결을 빌려 쓰고 반납하는 방식으로 통신 효율을 극대화합니다. (Spring Boot에서 흔히 사용하는 `HikariCP`가 이 역할을 합니다.)

---

## 4. 왜 공유 메모리가 아닐까?
앞서 설명한 IPC 방식 중 가장 빠른 건 '공유 메모리'였지만, 자바와 DB 사이에서는 소켓을 주로 씁니다. 그 이유는 **확장성** 때문입니다.
* **공유 메모리:** 두 프로세스가 같은 물리 장비에 있어야만 가능합니다.
* **소켓:** DB 서버가 지구 반대편에 있어도 통신이 가능하므로, 현대적인 분산 시스템 환경에 가장 적합합니다.

## 요약
> 자바와 DB의 통신은 **네트워크 소켓(TCP/IP)** 방식이며, 이를 자바에서 편하게 쓰도록 추상화한 것이 **JDBC**와 **커넥션 풀**입니다.

최근에 세팅 중인 서버리스 환경(Cloudflare Workers 등)이나 Java 백엔드 구성에서 DB 연결 시 발생하는 타임아웃이나 CPU 오버헤드 문제가 고민이신가요? 구체적인 상황을 알려주시면 더 자세히 짚어드릴 수 있습니다.
