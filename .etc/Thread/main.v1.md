1스레드/1리퀘스트
1Thread/1Request

자바의 표준 웹 서버 방식(Servlet Container)을 그대로 따르고 계신 것입니다.

우리가 흔히 쓰는 **Spring Boot**나 **Tomcat** 같은 서버들은 기본적으로 **'Thread-per-Request' (요청당 스레드 하나)** 모델을 사용하도록 미리 설계되어 있거든요.

그 구조가 어떻게 돌아가는지 쉽게 풀어드릴게요.

---

### 1. "스레드 풀(Thread Pool)"이라는 대기조가 있습니다
서버를 켜면 자바는 미리 일꾼(스레드)들을 수십~수백 명 뽑아놓습니다. 이걸 **스레드 풀**이라고 합니다.

1.  **최초 요청 발생:** 유저가 브라우저에서 '전송' 버튼을 누릅니다.
2.  **일꾼 배정:** 대기실에서 놀고 있던 스레드 하나가 튀어나와서 그 요청을 전담합니다.
3.  **작업 수행:** 유저님이 짠 Java 코드(Controller -> Service -> DB)를 이 스레드가 위에서 아래로 한 줄씩 실행합니다.
4.  **응답 및 복귀:** 결과를 유저에게 돌려주면, 이 스레드는 죽는 게 아니라 다시 대기실(Pool)로 돌아가 다음 요청을 기다립니다.



### 2. 왜 이게 기본 설계가 되었을까?
자바가 이 방식을 선택한 이유는 **'개발자가 코딩하기 가장 편하기 때문'**입니다.

* **순차적 사고:** 코드가 위에서 아래로 순서대로 실행되니 이해하기 쉽습니다. (A 실행 -> B 실행 -> C 응답)
* **독립성:** 1번 스레드에서 에러가 나서 뻗어도, 2번 스레드에서 일하는 다른 유저의 요청에는 영향을 주지 않습니다.

### 3. 유저님이 느끼신 "부담"의 포인트
질문에서 "input 변할 때마다 Ajax 보내면 부담될 것 같다"고 하신 부분은 바로 이 **'대기조의 한계'**를 본능적으로 직감하신 거예요.

* 만약 스레드 풀에 일꾼이 200명 있는데, 사람들이 input을 미친 듯이 입력해서 동시에 300개의 요청이 들어오면 어떻게 될까요?
* 남은 100명은 일꾼이 날 때까지 줄을 서야 합니다. 이게 누적되면 서버가 느려지거나 응답 불가 상태가 되죠.

---

### 4. 그래서 유저님이 하실 선택은?

지금 공부하고 계신 단계에서는 **"기본 설계(Thread-per-Request)"**를 그대로 쓰시는 게 맞습니다. 대신 서버의 부담을 줄이기 위해 다음 전략을 쓰시면 됩니다.

* **프론트엔드 최적화 (Debounce):** 아까 말씀드린 대로, 타이핑할 때마다 보내지 말고 입력을 멈췄을 때만 보내기. (서버로 오는 요청 횟수 자체를 줄임)
* **가벼운 로직:** Ajax 요청을 받는 컨트롤러에서는 아주 최소한의 작업만 하기. (일꾼이 빨리 대기실로 복귀할 수 있게 함)

### 결론
"내가 스레드 하나를 잡도록 설계한 건가?"라는 질문에 대한 답은 **"YES"**입니다. 자바의 전통적인 방식(Spring MVC)을 쓰신다면 아주 자연스러운 현상입니다. 

자바 개발자로서 **"내 코드가 지금 일꾼 한 명을 꽉 붙잡고 있구나"**라는 사실을 인지하고 코딩하는 것만으로도 이미 중급 개발자로 가는 길에 들어서신 겁니다! 

이제 이 일꾼을 어떻게 하면 빨리 쉬게 해줄지(효율적인 코드 작성) 고민해 보시면 더 재미있어질 거예요. 다음엔 어떤 부분을 더 개선해 보고 싶으신가요?
