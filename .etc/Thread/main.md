##
자바가 **'스레드(Thread)'**를 기본 작업 단위로 채택한 이유는 자바의 탄생 배경과 설계 철학에 깊게 뿌리 박혀 있습니다. 
자바가 처음 나올 당시(90년대 중반)는 현대적인 멀티태스킹 운영체제들이 대중화되던 시기였거든요.
자바.왜 하필 스레드인지


### 1. 프로세스(Process)보다 가볍기 때문입니다
컴퓨터에서 실행 중인 프로그램 하나를 **'프로세스'**라고 합니다. 만약 자바가 요청 하나당 프로세스를 하나씩 만들었다면 서버는 금방 터졌을 거예요.

* **프로세스:** 독립된 메모리 공간을 통째로 가집니다. 집을 새로 짓는 것과 같아서 만드는 비용도 비싸고 메모리도 많이 먹습니다.
* **스레드:** 프로세스(집) 안에서 방을 나누어 쓰는 **'가족'**과 같습니다. 집의 거실(공유 메모리)을 같이 쓰기 때문에 만드는 속도가 훨씬 빠르고 메모리도 적게 듭니다.

### 2. "공유 메모리"를 통한 효율적인 협업
자바의 큰 강점 중 하나는 여러 작업이 **데이터를 쉽게 주고받는 것**입니다.

* 스레드들은 같은 프로세스 내의 메모리(Heap 영역)를 공유합니다. 
* 덕분에 한 스레드가 계산한 결과를 다른 스레드가 즉시 보고 활용할 수 있습니다. 만약 프로세스 단위였다면, 서로 남남이라 데이터를 주고받기 위해 복잡한 통신(IPC) 과정을 거쳐야 했을 겁니다.



### 3. CPU의 성능을 끝까지 뽑아쓰기 위해
요즘 CPU는 코어(Core)가 여러 개입니다. 자바가 스레드 단위로 일을 시키는 이유는 **여러 개의 코어에 일을 골고루 나눠주기 위해서**입니다.

* 만약 자바가 단일 작업(Single Thread)만 가능했다면, 아무리 좋은 16코어 CPU를 사도 일꾼은 1명만 일하고 나머지 15명은 노는 상황이 벌어집니다.
* 스레드 단위를 사용함으로써, **"A 스레드는 1번 코어에서 계산하고, B 스레드는 2번 코어에서 DB를 조회해라"** 같은 병렬 처리가 가능해진 것이죠.

---

### 자바의 특징: "언어 차원에서 스레드를 지원한다"
과거 C나 C++ 같은 언어들은 운영체제(OS)의 기능을 빌려서 어렵게 스레드를 관리해야 했습니다. 하지만 자바는 처음부터 언어 스펙 자체에 `Thread` 클래스와 `Runnable` 인터페이스를 넣어두었습니다. 

**"어떤 OS에서든 똑같이 스레드를 쉽게 쓸 수 있게 하겠다"**는 자바의 'Write Once, Run Anywhere' 철학이 반영된 결과입니다.

### 하지만 최근의 변화 (Java 21 가상 스레드)
질문하신 것처럼 기존 자바 스레드는 **'OS 스레드'**와 1:1로 연결되어 있어 여전히 조금 무거웠습니다(1MB 정도 점유). 

그래서 최근 자바 21에서는 이 단점을 극복하기 위해, OS의 도움 없이 자바 스스로 수백만 개를 만들 수 있는 **'가상 스레드(Virtual Thread)'**를 도입했습니다. 이제는 "스레드가 무거워서 부담된다"는 말도 옛말이 되어가고 있답니다.

자바의 이 '일꾼 관리' 방식이 이제 좀 이해가 가시나요? 결국 **"가장 적은 비용으로 가장 많은 일을 동시에 하기 위한 선택"**이었다고 보시면 됩니다.
