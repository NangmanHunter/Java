자바 개발을 하다 보면 꼭 마주하게 되는 **Mock(모크)**! 쉽게 말해 **"진짜 객체처럼 행동하지만, 실제로는 가짜인 시험용 객체"**라고 생각하면 돼요.

우리가 요리 수업을 듣는데, 진짜 불과 칼을 쓰기엔 위험하거나 재료비가 너무 많이 들 때 '모형 가스레인지'나 '플라스틱 칼'로 연습하는 것과 비슷합니다.

---

## 1. Mock이 왜 필요한가요?
테스트 코드를 짤 때, 내가 검증하려는 기능이 **다른 클래스나 외부 시스템(DB, API 등)과 얽혀 있으면** 테스트가 복잡해집니다.

* **의존성 분리:** DB가 연결되지 않은 상태에서도 비즈니스 로직만 테스트하고 싶을 때.
* **속도 향상:** 실제 네트워크 통신을 하면 테스트가 느려지지만, Mock을 쓰면 순식간에 끝납니다.
* **특정 상황 재현:** "네트워크 에러가 났을 때"나 "DB 서버가 터졌을 때" 같은 특수한 상황을 강제로 만들어서 테스트할 수 있습니다.



---

## 2. Mock의 핵심 특징 (Stubbing)
Mock 객체는 비어있는 껍데기 같아서, 우리가 직접 **"이런 요청이 들어오면 이런 값을 돌려줘"**라고 프로그래밍할 수 있습니다. 이걸 **Stubbing(스터빙)**이라고 해요.

> **예시:**
> "가짜 데이터베이스(Mock)야, 내가 ID 1번 유저를 찾아달라고 하면 '홍길동' 객체를 리턴해줘!" 라고 미리 설정해두는 것이죠.

---

## 3. 자바에서 주로 사용하는 도구: Mockito
자바 진영에서 Mock을 가장 편하게 만들게 해주는 라이브러리가 바로 **Mockito**입니다. 코드로 보면 대략 이런 느낌이에요.

```java
// 1. 가짜 객체 생성
UserRepository mockRepo = mock(UserRepository.class);

// 2. 행동 정의 (Stubbing)
// "1번 유저를 찾으면 이름이 'Gemini'인 유저를 반환해라"
when(mockRepo.findById(1)).thenReturn(new User("Gemini"));

// 3. 테스트 진행
User result = userService.getUserDetails(1);

// 4. 검증 (Verification)
// "실제로 가짜 객체의 findById 메서드가 실행됐는지 확인"
verify(mockRepo).findById(1);
```

---

## 4. 요약하자면
* **진짜 대신 가짜:** 실제 객체를 만들기 어렵거나 무거울 때 사용합니다.
* **통제 가능:** 내가 원하는 결과를 반환하도록 조종할 수 있습니다.
* **독립적 테스트:** 외부 요인에 방해받지 않고 "내가 짠 코드"가 잘 돌아가는지만 확인할 수 있게 해줍니다.

혹시 지금 진행 중인 프로젝트에서 테스트 코드를 짜다가 궁금해진 건가요? 어떤 부분에서 Mock을 써야 할지 고민이라면 상황을 더 알려주세요!
