실무에서 커넥션 개수를 정하는 것은 "무조건 많다고 좋은 게 아니다"라는 점이 핵심입니다. 커넥션이 너무 많으면 오히려 서버가 각 커넥션을 관리(Context Switching)하느라 실제 일은 못 하고 힘만 쓰게 되거든요.

보통은 다음의 기준과 공식들을 참고해서 결정합니다.

---

## 1. DB 커넥션 풀(DBCP)의 공식
가장 유명한 기준은 **HikariCP**(Java 진영의 표준적인 커넥션 풀 라이브러리)에서 권장하는 공식입니다.

$$pool\_size = (core\_count \times 2) + effective\_spindle\_count$$

* **core_count:** 현재 서버의 CPU 코어 수입니다.
* **effective_spindle_count:** 하드디스크(HDD)의 개수인데, 요즘은 대부분 SSD를 쓰므로 보통 **0** 또는 **1**로 계산합니다.

> **왜 코어 수의 2배인가요?**
> CPU 코어 하나는 한 번에 하나의 작업만 할 수 있지만, DB 작업 중에는 디스크에서 데이터를 읽어오는 등 '대기 시간(I/O Wait)'이 발생합니다. 그 비어 있는 시간에 다른 커넥션의 작업을 처리할 수 있도록 2배 정도를 잡는 것이 효율적이라는 통계적 수치입니다.

---

## 2. '병목 현상'을 고려한 설정
무작정 DB 커넥션을 늘린다고 성능이 좋아지지 않는 이유는 **DB 서버의 자원(CPU, Disk I/O)**이 한정되어 있기 때문입니다.

1.  **서버 사양 확인:** DB 서버의 CPU와 RAM이 버틸 수 있는 최대 커넥션 수를 먼저 확인합니다 (MySQL의 경우 `max_connections` 설정).
2.  **애플리케이션 개수:** 만약 서버(WAS)를 3대 띄운다면, 각 서버가 가져가는 커넥션의 합이 DB의 한계를 넘지 않아야 합니다.
    * 예: DB 최대 허용이 100개인데, WAS 3대가 각각 50개씩 가져가려 하면 에러가 나겠죠?

---

## 3. 실제 서비스 운영 시의 흐름
보통 처음부터 완벽한 숫자를 맞추기는 어렵습니다. 그래서 다음과 같은 단계를 거칩니다.

* **기본값 시작:** 보통 10~20개 정도로 작게 시작합니다.
* **부하 테스트 (Load Test):** nGrinder나 JMeter 같은 툴로 가상의 사용자들을 몰아넣어 봅니다.
* **모니터링:** * 커넥션 풀이 꽉 차서 사용자들이 대기하는지? (Wait Time 확인)
    * DB 서버의 CPU가 100%를 찍는지?
* **튜닝:** 대기가 길면 조금씩 늘리고, DB 사양이 부족하면 서버 사양을 올리거나 쿼리를 최적화합니다.



---

## 4. 커넥션은 '예약석'이다
커넥션을 들고 있는 상태를 식당의 **'예약석'**이라고 생각하면 편합니다.

* 예약석을 너무 많이 만들면(Connection Pool 과다), 정작 일반 손님이 앉을 자리가 없는데 예약석은 비어 있는 상황이 생깁니다. (메모리 낭비)
* 예약석을 너무 적게 만들면, 손님들이 줄을 서다가 그냥 돌아갑니다. (Connection Timeout 에러)

지금 개발 중인 모듈형 게시판 프로젝트라면, 로컬 환경에서는 **10개 내외**로도 충분히 쾌적하게 테스트하실 수 있을 거예요. 나중에 실제 배포를 고민하실 때 위 공식을 적용해 보시면 좋습니다.
